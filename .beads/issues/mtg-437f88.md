# Description

## Description

The RNG state serialization in `advance_step` accounts for 12.3% of allocations (150KB in profiling).

**Current implementation:**
- Uses `ChaCha12Rng` from rand_chacha
- Serializes with serde_json::to_vec() - creates Vec<u8> allocation
- Happens once per turn in advance_step (src/game/state.rs:456)
- Stored in GameAction::ChangeTurn { rng_state: Option<Vec<u8>> }

**Problem:**
- JSON serialization is verbose (~375 bytes per turn)
- Requires heap allocation for Vec<u8>
- 12.3% of gameplay allocations in DHAT profiling

**Goal:**
Find a more efficiently serializable RNG that either:
1. Has a simple struct-of-scalars that is Copy + fixed-size
2. Works efficiently with rkyv (zero-copy serialization)
3. Can be serialized to a small fixed-size buffer (e.g., [u8; 64])

**Requirements:**
- Must maintain cryptographic quality for gameplay determinism
- Must be seedable for reproducible games
- Must support serde for compatibility with existing undo system
- Should be widely used/well-tested

**Candidate RNGs to investigate:**
- PCG family (pcg_rand crate) - very compact state
- Xoshiro/Xoroshiro (rand_xoshiro) - simple state, fast
- SmallRng from rand - already designed for compact state
- Custom lightweight RNG with explicit [u64; N] state

**Expected impact:**
- If using fixed-size state: 12% allocation reduction
- If using more compact serialization: smaller undo log, better cache locality
- Performance impact likely minimal (serialization work still required)

**Related:**
- See mtg-2 (optimization tracking) for context
- Part of medium-priority allocation optimizations
- DHAT profiling results: 2025-11-04_#706
