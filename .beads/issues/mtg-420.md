# Description

## Summary

The client's local shadow game did not enumerate Cycle (typecycling) abilities that the server enumerates, leading to:

1. `ABILITY SYNC BUG - server has 3 abilities, local has 2` warnings every priority pass
2. eventually a `FATAL DESYNC: Choice mismatch - index 1 selected Cycle { card_id: 31 ... search_type: Some(Subtype("Mountain")) }, but client expected CastSpell { card_id: 36 }` because the indexes diverged.

The client never offered the Cycle option to its controller (its action list was `[CastSpell { card_id: 36 }, CastSpell { card_id: 39 }]`), so when the server received "index 1" it believed CastSpell {36} was a Cycle command.

## Reproducer

```bash
cd mtg-forge-rs
./tests/network_vs_local_equivalence_e2e.sh 315 random random
## Or the dedicated regression wrapper:
./tests/cycle_ability_network_sync_e2e.sh
```

## Status: FIXED 2026-05-15

Root cause: NOT in the cycling enumeration path itself. `scry_cards` (and `surveil_cards`) used a heuristic that called `is_land()` / `is_creature()` on the top card of library. On the SERVER, with full card data, the heuristic put a Swamp on the bottom (lands_in_hand >= 3 -> want_lands=false). On the CLIENT shadow game, the top card had not yet been revealed, so `try_get(card_id).is_some_and(|c| c.is_land())` returned false and the card was kept on top. The two libraries diverged by one slot, the next mandatory draw produced different CardIds (server drew Mongoose Lizard; client drew the next Swamp), and Mongoose Lizard never made it to the client's hand — so the Mountaincycling ability was missing from the client's action list. The choice index for Cycle on the server then mapped to CastSpell on the client -> FATAL DESYNC.

In the seed=315 trace the trigger source was Rumble Arena's "When this land enters, scry 1" ETB.

### Fix

`mtg-engine/src/game/state.rs`:
- `GameState::pending_library_reorders: RefCell<Vec<PlayerId>>` — a transient queue of players whose library order on the server changed via a hidden-info-dependent operation and therefore needs to be broadcast to clients. Not serialised, not cloned.
- `scry_cards()`: short-circuits with no mutation when `is_shadow_game && any top card is unrevealed`. Otherwise runs the existing heuristic; when it actually moves a card AND we are the server (`!skip_reveals && !is_shadow_game`), pushes the player onto `pending_library_reorders`.
- `surveil_cards()`: identical guard for the same family of bug (creature/land heuristic on hidden cards).

`mtg-engine/src/game/controller.rs`:
- `GameStateView::take_pending_library_reorders()` drains the queue, snapshotting each player's current library order in protocol direction (top-to-bottom).

`mtg-engine/src/network/controller.rs`:
- `ChoiceRequest.library_reorders: Vec<(PlayerId, Vec<CardId>)>`. `NetworkController::request_choice` drains the queue from the view and includes it in every outgoing `ChoiceRequest`.

`mtg-engine/src/network/server.rs`:
- New `GameToHandler::LibraryReordered { player, new_order }` variant; the per-connection handler forwards it as `ServerMessage::LibraryReordered`.
- The coordinator drains `choice_request.library_reorders` and broadcasts a `LibraryReordered` to BOTH player handlers BEFORE forwarding the originating `ChoiceRequest` (in both the P1 and P2 branches). Each client's existing `sync_callback` drains its `pending_library_reorders` queue at the next priority-sync point — which already happens before ability enumeration in `priority.rs:374` (`self.sync_to_action()` after `prepare_for_priority_choice()`), so the shadow library is correct by the time `get_available_spell_abilities` runs.

### Regression coverage

- `tests/cycle_ability_network_sync_e2e.sh` — runs the seed=315 random/random reproducer and asserts (a) byte-identical local-vs-server gamelogs, (b) no `ABILITY SYNC BUG` warnings, (c) no `FATAL DESYNC` errors, (d) sanity-checks that the Mountaincycling code path was actually exercised.
- `mtg-engine/src/game/state.rs::tests::test_scry_shadow_game_skips_when_top_unrevealed` — locks in the client-side guard.
- `mtg-engine/src/game/state.rs::tests::test_scry_server_enqueues_library_reorder_when_changed` — locks in the server-side queueing.
- `mtg-engine/src/game/state.rs::tests::test_scry_server_no_enqueue_when_unchanged` — guards against spamming empty `LibraryReordered` messages when the heuristic produces a no-op.

## Discovered by

`bug_finding/network_fuzz_test.py --configs 30 --controller random` 2/30 occurrences with seed=315 random+random.

# Notes

Fixed by deferring scry/surveil reorder on shadow games and broadcasting LibraryReordered after server-side scry/surveil. See mtg-engine/src/game/state.rs (scry_cards / surveil_cards / pending_library_reorders), mtg-engine/src/network/controller.rs (ChoiceRequest.library_reorders + drain in request_choice), mtg-engine/src/network/server.rs (GameToHandler::LibraryReordered + coordinator broadcast in both P1 and P2 ChoiceRequest branches). Regression coverage: tests/cycle_ability_network_sync_e2e.sh + 3 unit tests in game::state::tests::test_scry_*. Local + network gamelogs are now byte-identical for seed=315 random/random.
