## feat(deploy): TLS termination on VM with Cloudflare Origin Certificate

The code infrastructure is in place (web_server/mod.rs supports TLS via tls_cert/tls_key args; deploy-cloud.sh has --tls-cert/--tls-key flags). Whether the cert was actually provisioned on deepscry.net and CF is in Full(strict) mode requires a live smoke test. Cannot verify from code alone.

## Context

Part of the web-server-unification design. Depends on **mtg-468** (the unified axum process) — TLS is bolted on once axum exists.

## Port + TLS plan

- Port 80 is taken by another web server on the VM. DeepScry already runs on **8080**.
- Cloudflare has an Origin Rule mapping `deepscry.net` → VM port 8080.
- Switch CF SSL/TLS mode to **Full (strict)**. CF will speak HTTPS to origin on 8080.
- axum binds 8080 with rustls.

## Code status (2026-06-01)

- mtg-engine/src/web_server/mod.rs: TLS supported via tls_cert/tls_key fields
- scripts/deploy-cloud.sh: --tls-cert/--tls-key flags + TLS_CERT_PATH/TLS_KEY_PATH env vars

BLOCKING: The actual CF Origin Certificate provisioning and VM file write has not been confirmed from code inspection. Needs VM access or a deploy log showing TLS enabled.

## Acceptance criteria

- `curl https://deepscry.net/` returns 200 with valid CF-edge cert chain
- DevTools shows zero mixed-content warnings
- `wss://deepscry.net/lobby` connects without port number
- `curl http://localhost:8080/` still works (plain HTTP fallback)
