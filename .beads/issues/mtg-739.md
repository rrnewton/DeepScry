# Description

GOAL: A high-concurrency game-hosting mode to support the MAXIMUM number of simultaneous users/games (e.g. if we go viral on Hacker News). In this mode the server does ONLY the initial handshake (WebRTC signaling + matchmaking), then DROPS OUT of the data path -- the entire game runs peer-to-peer between the two browser clients. The server crunches ZERO game-engine state here; it is a pure coordinator. Even static serving is minimal: a TINY index.html (HTTP-cacheable ~10 min) plus all other assets hash-coded and served from the CDN. Should scale far on modest server resources.

TWO MODES, NO MIDDLE GROUND:
1. Scalable P2P mode: server = signaling-only coordinator; game runs P2P; server runs NO engine. Secret/hidden-information protection is EXPLICITLY TRADED AWAY -- in P2P both clients are deterministic replicas holding full game info in memory, so a modified client could read the opponent's hidden zones. This is ACCEPTED for the scale/casual mode; it is a deliberate non-goal.
2. Fallback = the EXISTING server-hosted model: the server runs the authoritative "golden" game with hidden-info protection (opponent cards as reserved CardIds), exactly as today. This is also the fallback when WebRTC cannot establish a P2P connection (hard NAT / no TURN). If a player wants serious secret protection, they use this mode.

DECISION (user, 2026-06-03): do NOT build a "thin secret-oracle" middle mode (server holds hidden zones + emits reveals while clients simulate). REJECTED -- any per-game secret-arbitration server work undercuts the whole point (max concurrency). Two clean modes only.

TRANSPORT: WebRTC DataChannel (browsers cannot open WAN listen ports; WebRTC ICE/STUN/TURN is the browser P2P mechanism). Server relays SDP offer/answer + ICE candidates over the existing lobby WebSocket (a few KB, once), then leaves the data path. STUN hole-punching covers ~80-90% of NAT pairs; a TURN relay (coturn or managed) handles the rest, relaying CHOICES ONLY.

WHY P2P IS CHEAP HERE: the engine is a deterministic sequential simulation; both clients are already full replicas (shared ChaCha seed -> identical state). The wire carries only player CHOICES, never game state, so per-game bandwidth is trivial and the work is pure client-side simulation. The existing server<->client state-hash comparison still works peer-to-peer (each peer verifies the other's choices reproduce the same state hash), so cheating-by-divergence is still DETECTED -- only the hidden-info read is unprotected in mode 1.

HARD PREREQUISITE: netarch determinism (mtg-728 / mtg-725) must be truly green first. P2P lockstep is only trustworthy if both clients stay bit-identical from the shared seed with zero desync. The 2026-06-03 controller-seed-pinning breakthrough (pin BOTH AI controller seeds, not just the engine seed -- the e2e harness was leaking entropy via unseeded controllers) is essential and part of that prerequisite.

OPEN DESIGN POINTS (later): reconnection/disconnect (a dropped peer needs state to resume -- surviving peer or snapshot); matchmaking -> signaling handoff; TURN hosting choice; graceful degrade to mode 2 when WebRTC fails.

NEXT: design-only. Later: benchmark + estimate achievable concurrency per server. Depends on netarch being solid. Related: deck-storage CAS pointer-store (same thin-self-hosted-server philosophy); mtg-726 (network-test perf).
