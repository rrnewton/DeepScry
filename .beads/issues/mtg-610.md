# Description

ARCHITECTURE: WASM step/spell-resolution re-entry should RESUME (rewind/replay), not RE-RUN — unify the idempotency-guard family onto the shared undo-log time-travel mechanism.

=== LINCHPIN QUESTION ANSWERED 2026-05-28 (branch wasm-rewind-replay, see mtg-j4128) ===

"Is undo.rs::rewind_to_turn_start COMPLETE for arbitrary state (mid-resolution / network)?"

PARTIAL ANSWER, with evidence:
- For the LOCAL path: YES. make validate-snapshot-resume-step passes 7/7 with the guards removed; native snapshot/resume rewinds to turn start and replays correctly. The undo log is a faithful inverse for local single-process play.
- For the NETWORK harness path: NOT YET. The WASM AI harness was rewritten to rewind+replay every step (deleting all 9 TurnStructure guards). It builds green, native tests green (841), but the network GUI e2e mirror gate fails with a P2 shadow/server state-hash mismatch on the FIRST turn>=2 undo-rewind (same action_count, different content hash => a card in a different zone post-replay). The debug turn-start round-trip verifier passes, so the turn-start reconstruction is internally stable; the divergence is in destructively-consumed NETWORK inputs that the undo log does NOT capture:
  * Server library reorders (client.drain_library_reorders) — applied to zones.library directly, not undo-logged; gone from the queue on replay, so a replayed draw can pop a different card.
  * Opponent-choice side-channels (WasmRemoteController.last_spell_ability/last_library_search_result) — not repopulated when ReplayController short-circuits the inner remote.

So: rewind_to_turn_start correctly inverts the GameState's undo-logged actions, but the NETWORK harness has out-of-band consumed inputs (reorders/reveals/opponent side-channels) that must ALSO be made replayable for a complete round-trip. This is the precise gap to close.

PROPER FIX OPTIONS (for the follow-up):
A. Make consumed network inputs replayable: record library reorders / reveals / opponent choices keyed by action_count; on replay re-feed from the record instead of re-draining the live queue.
B. Snapshot full GameState at each turn boundary in the harness (as already done for turn-1 via the new GameLoop::with_post_setup_hook) and restore-from-clone instead of undo-rewinding — sidesteps undo-completeness for the harness entirely, at the cost of per-turn clones.

MCTS DEPENDENCY: option A (replayable inputs) is the network-robust path; option B (turn-boundary GameState clones) is likely what MCTS rollouts want anyway. Either unblocks both. The guard-family deletion + harness rewind/replay scaffolding is DONE and green except for this input-replay completeness gap.
