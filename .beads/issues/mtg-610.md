# Description

ARCHITECTURE: WASM step-harness / spell-resolution re-entry should RESUME, not RE-RUN — unify the growing idempotency-guard family.

Root cause (shared by mtg-609 FIXED + mtg-559 DEFERRED): the WASM AI harness recreates the GameLoop wrapper each step_harness() call; when a step or a spell resolution blocks with NeedInput (awaiting a server ChoiceRequest) and is re-entered, the code RE-RUNS from the top of the step prologue / resolution instead of RESUMING at the blocking point. This re-executes already-applied logic on the WASM shadow, diverging it from the server (desync).

Current state = point-patches (band-aids), NOT the proper architecture:
- Step prologues: per-turn guards keyed by turn_number — draw_step_executed_turn, attackers_declared_turn, and now upkeep_triggers_checked_turn / end_step_triggers_checked_turn (mtg-609 fix in game_loop/steps.rs + phase.rs). Each new once-per-turn begin-of-phase event that precedes a blocking priority_round needs another guard.
- Spell resolution: NO guard yet — mtg-559 (robots42 Copy Artifact, also Balance, and any in-resolution choice: DiscardCards/Loot/Scry/Surveil/Clone/Dig/SearchLibrary). resolve_top_spell_with_discard_hook unwinds with the spell still on the stack; re-entry re-runs already-applied effects.

PROPER FIX (toward the deterministic replicated state machine + shared-code principle):
- A general RESUME model: step execution and spell resolution carry an explicit resume/progress index (analogous to the existing pending_activation_effect_idx) so a re-entered step/resolution continues from where it blocked rather than re-running. Then the per-turn guard family (draw/attackers/upkeep/endstep) collapses into the general mechanism and can be deleted.
- Must be validated across ALL choice-bearing in-resolution effects + all begin-of-phase blocking events, in BOTH execution paths (native single-pass and WASM step-harness), with native behavior byte-for-byte unchanged.
- This is shared engine code (no per-mode divergence) — the WASM harness should drive the SAME resumable simulation the server/native uses, never a parallel re-run.

Why not done now: too broad to land alongside the isolated mtg-609 fix; mtg-559 is the immediate concrete instance (robots42 still excluded from the network gate). Sequence: do this architectural resume model, then mtg-559 falls out + robots42 re-joins the gate.

Refs: mtg-609 (closed, the narrow upkeep/endstep guard), mtg-559 (open, the spell-resolution instance), mtg-589 (original network desync family). See docs/NETWORK_ARCHITECTURE.md.

--- SHARPENED 2026-05-28 (rewind/replay vs re-run; MCTS dependency) ---
CONFIRMED the compromise concretely:
- INTENDED model (ai_docs/reference/snapshot_architecture.md): snapshot at turn-start; on resume undo.rs::rewind_to_turn_start() + REPLAY intra-turn choices re-executing them, suppressing ONLY logging (external). Internal game-state evolves back-then-forward. The snapshot/resume + network-snapshot paths use this and pass.
- COMPROMISE: mtg-engine/src/wasm/network/ai_harness.rs::step_harness keeps harness.game persistent, recreates GameLoop each call, run_until_input RE-ENTERS the current step WITHOUT rewinding (no rewind_to_turn_start call in the harness). The step prologue re-runs on re-entry.
- SYMPTOM: TurnStructure (phase.rs) accumulated 9 once-per-turn guard fields (draw_step_executed_turn, attackers_declared_turn, blockers_declared_turn, combat_first_strike_damage/priority, combat_damage_dealt, turn_state_reset, upkeep_triggers_checked, end_step_triggers_checked) to suppress double-application. These exist ONLY because of re-run-without-rewind; a true rewind+replay would not need them (resume path proves it — suppresses logging only).
PROPER FIX: unify the harness blocking simulation onto the undo-log rewind/replay (rewind to checkpoint, replay forward re-evolving state, suppress external effects only) -> deletes the 9-guard family AND fixes mtg-559 (in-stack-resolution re-entry re-runs already-applied effects).
MCTS DEPENDENCY (critical): MCTS rollouts MUST use rewind_to_turn_start, NOT the harness guard-based re-run (guards would skip already-done once-per-turn actions -> corrupt rollout states). PREREQUISITE TO VERIFY: is undo.rs::rewind_to_turn_start COMPLETE for arbitrary MID-RESOLUTION state? mtg-559 suggests NOT (spell-resolution re-entry re-applies effects). Verify undo-log coverage of mid-resolution/in-stack state before building MCTS on it. This is the linchpin: same root fix unblocks network-resume robustness AND MCTS rollouts.
