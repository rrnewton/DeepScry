# Description


Test all behavioral aspects of Ironclaw Orcs in MTG Forge-rs (1994 Old School, Troll Disk deck mtg-562).

Card file: cardsfolder/i/ironclaw_orcs.txt
Card text: {1}{R} 2/2 Creature — Orc. Ironclaw Orcs can't block creatures with power 2 or greater.
Script: S:Mode$ CantBlockBy | ValidAttacker$ Creature.powerGE2 | ValidBlocker$ Creature.Self

Findings (2026-05-29_#2456(e30f4ce1), compat-oldschool-wave5):

1. [x] Parses as {1}{R} 2/2 Creature — Orc. Castable, attacks as a 2/2.
2. [BROKEN] Blocking restriction silently dropped. ROOT CAUSE: the static
   parser (loader/card.rs parse_static_abilities) only recognizes
   Mode$ Continuous / ReduceCost / RaiseCost. The Mode$ CantBlockBy line
   produces NO StaticAbility (there is no CantBlockBy variant in the
   StaticAbility enum, mtg-engine/src/core/effects.rs ~1901), and the combat
   block-legality predicate (game/combat_rules.rs can_block_impl) has no
   branch for a per-creature 'can't block attackers matching <filter>'
   restriction. Net effect: Ironclaw Orcs can illegally block power>=2
   attackers. Affects every card with this CantBlockBy-on-blocker pattern
   (e.g. Foggy Swamp Vinebender). Implementing it requires: (a) a structured
   StaticAbility::CantBlock { attacker_filter } variant, (b) parser support,
   (c) a can_block_impl branch evaluating the attacker filter against the
   blocker.
3. [N/A] No triggers/activated/replacement abilities.

Reproducer (parses but restriction unenforced):

```sh
./target/release/mtg tui decks/old_school/06_troll_disk_daniellebrunazzo.dck \
  decks/old_school/06_troll_disk_daniellebrunazzo.dck \
  --p1=heuristic --p2=heuristic --seed 7 --verbosity 3
```

CARD STATUS: PARTIAL — castable 2/2, but the printed 'can't block power>=2'
restriction is silently dropped (BROKEN aspect). Engine feature gap; left
open pending CantBlockBy static-restriction support.


Findings (2026-05-31_#2549(221c7867), compat-finish-trolldisk) — FIXED:

[WORKING] The block restriction is now ENFORCED (CR 509.1b / 509.4).
FIX (general, no card hack):
- Added StaticAbility::CantBlockMatching { attacker_filter, description }
  (core/effects.rs). Lowered from the `Mode$ CantBlockBy | ValidAttacker$
  <filter> | ValidBlocker$ Creature.Self` shape in loader/card.rs
  parse_static_abilities (gated to the ValidBlocker$ Creature.Self self-blocker
  form; the distinct evasion form `ValidAttacker$ Creature.Self` is left
  untouched). A faithfulness guard (cant_block_filter_is_faithful) only emits
  the static when every ValidAttacker$ modifier is one TargetRestriction
  evaluates faithfully (type/color/power/token); keyword filters like
  withoutFlying remain unmodelled (no worse than prior silent-drop) pending a
  TargetRestriction extension.
- Enforced in combat_rules::can_block (the shared predicate used by both the
  engine's validate_blocking_restrictions and the UI menu filter) AND mirrored
  in heuristic_controller::can_block_impl so the AI never proposes an illegal
  block. The attacker filter (Creature.powerGE2) is evaluated via the existing
  TargetRestriction::matches.

Generalizes to all ~46 cards using ValidBlocker$ Creature.Self with a
power/color/type/token ValidAttacker$ filter (4 other powerGE2/GE3 cards, the
color/type hosers). withoutFlying-class filters (31 cards) deferred.

Evidence:
- Unit (parser shape): game::actions::tests::effects::test_card_compat_ironclaw_orcs
  (CantBlockMatching present; powerGE2 matches power-2, not power-1).
- Unit (predicate): tests/blocker_legality_test.rs
  ::ironclaw_orcs_cant_block_power_2_or_greater (can't block power 3 or power 2;
  CAN block power 1).
- E2E: tests/puzzle_e2e.rs::test_ironclaw_orcs_cant_block_power2
  (test_puzzles/ironclaw_orcs_cant_block_power2.pzl): Hill Giant (3/3) attacks
  each turn; Orcs survive (never trade) and P1 loses life every combat.
- CLI gamelog (seed 42): repeated "Player 2 declares Hill Giant (7) (3/3) as
  attacker" / "Hill Giant (7) deals 3 damage to Player 1" with an untapped
  Ironclaw Orcs (2/2) available but never blocking.

MTG Rules Review — Verdict: PASS (CR 509.1b restriction on block declaration;
public-state-only check; no decision moved server-side; deterministic
native/WASM; general mechanic not a card hack).
