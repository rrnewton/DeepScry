# Description

CLUSTER: UI/infra. RESOLVED 2026-05-28_#2366.

native_game.html battlefield wasted space: the container .card-grid used
`flex-direction: column`, forcing every view-model section
(Creatures | Artifacts | Lands) onto its OWN full-width row — one section
per row, 1-2 cards each — while the Web TUI rendered the same view-model
sections densely.

Investigation outcome: the shared layout IS already wired — native_game.html
consumes the Rust GUI view model (gui_view_model.rs via
tui_get_gui_view_model_json) for section grouping/order, and the shared Rust
battlefield_layout engine for per-card sizing. The defect was purely the JS
container's CSS flow, NOT a duplicated layout reimplementation.

FIX (web/native_game.html CSS only): .card-grid is now a wrapping flex ROW
(flex-wrap: wrap) and each .bf-section is a content-sized flex item, so small
sections pack side by side and only wrap when they don't fit. Section
grouping/order and card sizing still come from the single Rust source of
truth, so no drift is introduced.

EVIDENCE (heuristic-vs-heuristic, deck 06_jeskai_aggro, seed 7, ~turn 14):
- BEFORE: YOUR battlefield "Creatures (1)" and "Lands (3)" on 2 separate
  rows (distinct offsetTop), ~half the pane wasted; opp same.
- AFTER: "Creatures (1)" + "Lands (3)" share ONE row; cards larger; pane
  densely packed. Measured sections=2/rows=1 on both battlefields.

REGRESSION TEST: web/test_battlefield_layout.js (wired into
validate-wasm-e2e-step). Asserts .card-grid flows as a wrapping row and that
>=2 small sections share a row (distinctRows < sectionCount). Verified to
FAIL on the old column CSS and PASS on the fix.
