# Description

## Summary

Network gate desync work. The gate test (web/test_network_gui_e2e.js) =
native server + native client P1 + **WASM browser client P2**. Desync is always
"P2" = the WASM shadow.

## RESOLVED in this pass

### 1. Harness fix: single `--deck` => BOTH seats (deterministic mirror match)
Previously `--deck X` only set the NATIVE P1 deck; the browser P2 fell back to
the dropdown DEFAULT (an old-school Moxen deck), and several top-level decks
(monored, counterspells) are not even in the WASM export globs so they could
not be selected. The matchup was monored-vs-default and non-deterministic.

Fix (commit on fix-desync-vk4b7):
- web/test_network_gui_e2e.js reads the `.dck` file content and the `[metadata]
  Name=`, then injects it into the browser as a custom deck and selects it.
- web/tui_game.html exposes `window.__mtg_test_import_and_select_deck(player,
  content, expectedName)` which reuses the page's own `parseDckFormat` +
  `getCustomDecks`/`saveCustomDecks` + collection-switch (DRY; works for ANY
  .dck, including decks outside the WASM export globs).
- web/test_network_multideck.js scenario list updated to the proven-stable
  mirror set; known-broken mirror scenarios excluded WITH tracking refs.

Effect: `--deck decks/monored.dck --seed 13` is now a deterministic monored
mirror and runs CLEAN to completion (16 turns, NativeAI wins).

### 2. The prior pass's "monored choice_seq=40 / Turn 4" divergence is GONE
That was an artifact of the NON-mirror matchup (monored native vs Moxen-default
browser). Once both seats use monored, the already-landed native fixes
(a78c7b9e library-search tutors + forced discards; 3acceac2 WASM LibraryReordered
apply) cover it. No code change needed for monored beyond the harness mirror fix.
