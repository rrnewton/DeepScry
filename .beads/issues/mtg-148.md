# Description

Tournament testing (2025-11-23_#894) revealed "Entity not found: 0" errors occurring when creatures with no ETB triggers resolve and enter the battlefield. The error rate was ~15% (688 failures out of 5252 games across two tournaments).

## Reproduction

All errors are fully reproducible with provided seeds. Example:

```bash
cargo run --release --bin mtg -- tui --p1 random --p2 heuristic --seed 17546037831651397168 \
  "forge-java/forge-gui/res/quest/world/2014-05 Theros/duels/THS 3 GP Manchester Jund Midrange by Juan Gonzalez.dck" \
  "forge-java/forge-gui/res/quest/world/1996-05 Ice Age/duels/ICE 1 Lore Soldev The Machinists.dck"
```

Error occurs on Turn 7 when "Voyaging Satyr" (1/2 creature, no ETB triggers) resolves:
```
Player1 casts Voyaging Satyr (57) (putting on stack)
Tap Forest for {G}
Tap Mana Confluence for {G}
Voyaging Satyr (57) resolves
Error: EntityNotFound(0)
```

## Root Cause Analysis

Location: `mtg-engine/src/game/actions.rs:312`

When a creature spell resolves:
1. Line 302: Moves card from Stack to Battlefield
2. Line 308: Records turn_entered_battlefield  
3. **Line 312: Calls `check_triggers(TriggerEvent::EntersBattlefield, card_id)`**

The `check_triggers` function (`actions.rs:1291`) iterates over ALL battlefield permanents looking for triggers. For each matching trigger effect, it tries to fill in placeholder targets (ID 0) with actual targets.

**Hypothesis**: A trigger effect from an existing permanent has a target with CardId(0), and when `check_triggers` tries to use `source_card_id` to get the controller (lines 1326, 1339, 1364, 1372), it's passing CardId(0) to `self.cards.get()`, which fails with "Entity not found: 0".

**Alternative hypothesis**: The `source_card_id` parameter itself is CardId(0) being passed from somewhere.

## Impact

- 13-15% of games fail in tournaments
- Affects any creature entering the battlefield when certain permanents are present
- Fully reproducible with seeds (688 reproduction commands in tournament_errors.txt)

## Investigation Needed

1. Add logging to `check_triggers` to see what `source_card_id` value is passed
2. Check if any existing permanents have triggers with CardId(0) targets
3. Verify the card entering (Voyaging Satyr) has correct ID
4. Check if static abilities on existing permanents are creating bogus trigger effects

## Related Code

- `mtg-engine/src/game/actions.rs:312` - Call site when creature enters
- `mtg-engine/src/game/actions.rs:1291-1426` - check_triggers implementation
- Locations where EntityNotFound(0) can be thrown (from grep):
  - `game_loop.rs:2098`
  - `entity.rs:240, 245`
  - `state.rs:188, 196, 469, 496, 612`

## Resolution

**FIXED in commit 2eabce7**

The bug was NOT in check_triggers as hypothesized. The root cause was in card loading:

In `mtg-engine/src/loader/card.rs`, the `parse_effects()` method was incorrectly accepting both:
- `A:SP$` (spell effects that execute when spell resolves)
- `A:AB$` (activated abilities that require explicit activation)

This caused Voyaging Satyr's activated ability `A:AB$ Untap | Cost$ T | ValidTgts$ Land` to be added to the creature's `effects` list instead of `activated_abilities`. When the creature spell resolved, it tried to execute UntapPermanent{target: CardId(0)}, causing the EntityNotFound error.

**The fix**: Modified parse_effects() to only process A:SP$ lines, ensuring activated abilities are handled separately by parse_activated_abilities().

All 491 tests pass, and the reproducer now completes successfully:
```
=== Game Over ===
Winner: Player1
Turns played: 43
```

No EntityNotFound errors.
