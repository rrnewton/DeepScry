# Description

EXPERIMENT (user-requested 2026-05-28; PR-reviewed before merge, do NOT auto-merge): replace the WASM "re-run-and-skip" blocking model with a PROPER undo-log rewind/replay, per the original intent in ai_docs/archived/wasm_human_controller_design_20251206.md. Implements the proper fix tracked in mtg-610.

GOAL: ONE shared time-travel mechanism (the undo log) for all needs — WASM blocking, network-message waits, snapshot/resume, and (future) MCTS rollouts. Remove WASM-specific hacks from the core engine.

WHAT TO DO:
- When the WASM client must BLOCK (waiting for local input OR a remote network message / ChoiceRequest), REWIND via the undo log to the appropriate checkpoint (turn start, or the nearest clean choice point) instead of returning NeedInput and later re-entering the step from the top.
- On resume, REPLAY forward from the checkpoint, re-evolving the game state, suppressing ONLY external effects (duplicate game-log prints / network sends) — the INTERNAL game state must evolve backward then forward to EXACTLY where it was.
- REMOVE the per-turn guard complexity from TurnStructure: draw_step_executed_turn, turn_state_reset_turn, attackers_declared_turn, blockers_declared_turn, combat_first_strike_damage_dealt_turn, combat_first_strike_priority_done_turn, combat_damage_dealt_turn, upkeep_triggers_checked_turn, end_step_triggers_checked_turn (9 fields). These were band-aids for re-run-without-rewind AND violate the principle of no-WASM-specific-hacks-in-the-core-engine. With true rewind+replay they should be unnecessary.
- DEBUG-MODE INVARIANT: add #[cfg(debug_assertions)] hash checks that verify, after a rewind+replay round-trip, the game state hash returned EXACTLY to where it started (and matches the server at the resume point). This is the proof the undo log is complete.

WHY: the current ai_harness.rs::step_harness keeps the game persistent, recreates GameLoop, and re-enters the current step WITHOUT rewinding — re-running the prologue (triggers/draw/combat) and suppressing re-application via the 9 guards. snapshot_architecture.md + the snapshot/resume path already do the proper rewind (undo.rs::rewind_to_turn_start + replay-with-suppressed-logging); unify the harness onto that.

ACCEPTANCE / what success looks like:
- The 9 TurnStructure guard fields deleted (or drastically reduced), engine has no WASM-specific re-entry hacks.
- The network gate (test_network_gui_e2e / multideck) is GREEN and MORE ROBUST across decks/mechanics — including the currently-excluded robots42 (mtg-559, the in-stack-resolution re-entry case) and ideally re-enabling other excluded scenarios.
- Debug hash-check invariant passes (rewind+replay round-trips to the exact starting state).
- Full make validate green; net SIMPLER code (removed complexity > added).
- This also de-risks MCTS rollouts (they can reuse the same rewind), and answers mtg-610's "is rewind_to_turn_start complete for mid-resolution state?" — extend it if not (the robots/Copy-Artifact case needs mid-resolution rewind, not just turn-start).

CONSTRAINTS: experimental branch off the POST-crate-rename integration (uses mtg_engine crate name). Solo validate (one at a time). It's a big refactor; expect iteration. Report what got simpler (line counts, fields removed) + which previously-excluded decks now pass. Relates: mtg-610 (arch), mtg-559 (robots in-stack-resolution), mtg-273/mtg-589 (network desync family), snapshot_architecture.md, undo.rs.
