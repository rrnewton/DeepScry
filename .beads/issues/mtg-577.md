# Description

CLUSTER: UI/infra.

`make validate` intermittently fails with:
  === Running Network E2E tests ===
  Error: invalid type: map, expected a string at line 7 column 16
  make[3]: *** [Makefile:689: wasm-dev] Error 1
  make[2]: *** [Makefile:266: validate-wasm-step] Error 2

Root cause: validate runs wasm-dev and wasm-network (and possibly other wasm targets) in parallel, and EACH has a `if ! command -v wasm-pack; then cargo install wasm-pack`. When wasm-pack is missing/outdated, multiple `cargo install wasm-pack` run concurrently and corrupt each other / the cargo metadata, surfacing as the cryptic serde "invalid type: map" error. Has bitten ~4 separate agents' local validates (deploy-healthcheck, fix-gameplay-bugs-4pack, lobby-ui, others). Workaround that unblocks: manually `cargo install wasm-pack` once (also upgrades 0.13.1 -> 0.15.0) so the concurrent installs become no-ops.

FIX OPTIONS:
1. Make wasm-pack a pinned dev-dependency / install it once in a serialized `setup` step that the parallel wasm targets depend on (so the install can't race).
2. Guard the install with a flock, or move it out of the per-target recipes into a single prerequisite target.
3. Pin wasm-pack version in rust-toolchain / a tool-versions file and install in `make setup` + CI, never lazily inside wasm-dev/wasm-network.

This is a real CI/validate reliability issue (it produces false validate failures that mask the real result). Note: CI itself installs wasm-pack in a dedicated step so it doesn't hit the race; this is primarily a local-validate problem.
