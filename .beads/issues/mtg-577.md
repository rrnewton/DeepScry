# Description

CLUSTER: UI/infra. STATUS: FIXED on branch fix-wasmpack-race-mtg577.

`make validate` intermittently failed with:
  === Running Network E2E tests ===
  Error: invalid type: map, expected a string at line 7 column 16
  make[3]: *** [Makefile:689: wasm-dev] Error 1
  make[2]: *** [Makefile:266: validate-wasm-step] Error 2

Root cause: three wasm recipes (wasm, wasm-dev, wasm-network) each carried
their own inline `if ! command -v wasm-pack; then cargo install wasm-pack`
block. validate runs the wasm builds from SEPARATE recursive sub-make
processes (validate-wasm-step -> $(MAKE) wasm-dev; validate-network-e2e-step
-> $(MAKE) wasm-network), so when wasm-pack was missing, concurrent
`cargo install wasm-pack` invocations corrupted each other's cargo metadata,
surfacing as the cryptic serde "invalid type: map" error. Bit ~4 agents'
local validates. CI was unaffected (installs wasm-pack in a dedicated step).

FIX (DRY, single serialized install site):
- Added ONE `ensure-wasm-pack` target holding the single install block.
- `wasm`, `wasm-dev`, `wasm-network`, and `setup` now depend on
  `ensure-wasm-pack`; the three inline install blocks were removed.
- Because the wasm builds run from SEPARATE sub-make processes (not a single
  `-j` graph), make's "build a shared prerequisite only once" guarantee does
  NOT span them, so a plain shared prerequisite is insufficient. The install
  is therefore guarded with `flock /tmp/mtg-wasm-pack.lock`, so concurrent
  sub-makes serialize on a lock file; after the first install completes, every
  other process sees wasm-pack on PATH and the inner block is a no-op.
- Added `.PHONY: ensure-wasm-pack`. CI unchanged.

Build-infra only; no engine/gameplay change; MTG-rules-review N/A.
