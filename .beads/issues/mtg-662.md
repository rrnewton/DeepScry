# Description

scripts/check_clean_environment.py (line ~74) treats any process whose cmdline contains '-p mtg' (or '-p mtg-engine') as a conflicting build, REGARDLESS of which worktree it runs in. So when agent A runs 'make validate' in worktree A (its shell_script_tests fan out into many 'cargo test -p mtg-engine' invocations), agent B's validate precheck in worktree B aborts with 'Environment not clean - conflicting processes' even though they are fully isolated worktrees with separate target/ dirs.

IMPACT (observed 2026-05-31, multi-agent compat-finish session): concurrent deck/fix agents cannot validate in parallel — their validates serialize and AGENTS REPEATEDLY PREMATURE-YIELD mid-validate or spin orphaned nohup retry-loops polling the precheck (the robots agent spawned 5 such orphans; coordinator had to kill them by PID). Major drag on multi-agent throughput, which is the whole point of the worktree model.

FIX: scope the -p mtg conflict check to the CURRENT worktree only — e.g. match the process's CWD / the worktree path in the cmdline (--manifest-path or remap-path-prefix to this worktree), or check the target/ dir lock, not a bare global '-p mtg' substring. A cargo build in a DIFFERENT worktree's target/ is not a conflict for this worktree.

Repro: run 'make validate' in two worktrees concurrently → 2nd aborts on precheck while 1st's cargo runs. Found by the robots/trolldisk/databin-fix agents. Coordinator workstream: multi-agent compat-finish + databin-cache-skew fix.
