# Description

## Description

When multiple copies of the same card (e.g., 3 Swamps) are in play, display them as a visually stacked group with diagonal offsets and partial card rendering.

This is the full visual stacking feature that comes AFTER simple stacking (mtg-148).

## Visual Design

- Draw cards stacked from upper-left to lower-right with diagonal offsets
- Only the topmost card shows full content  
- Other copies show as border outlines behind
- When N cards are tapped, the TOP N appear tapped with untapped ones visible behind
- Diagonal offset: 1-2 characters per card in the stack

## Example Visual

```
  ┌────────┐
 ┌┤3x Swamp│
┌┼┤   {B}  │
│││        │
└┴┴────────┘
```

## Prerequisites

**Depends on: mtg-148 (Simple stacking)**

The simple stacking feature provides:
- `BattlefieldEntity` abstraction with `SingleCard` and `SimpleStack` variants
- `EntityPosition` for mouse hit testing
- `group_cards_into_entities()` grouping logic
- Entity-based rendering pipeline
- Entity-based dimension calculations

All of these abstractions are needed for visual stacking but are introduced in the simpler upstream task.

## Implementation Steps

### Add VisualStack Variant

1. **Extend BattlefieldEntity enum**
   ```rust
   enum BattlefieldEntity {
       SingleCard { card_id: CardId },
       SimpleStack { /* existing */ },
       VisualStack {
           card_ids: Vec<CardId>,
           card_name: String,
           tapped_count: usize,  // How many are tapped (tapped ones on top)
           representative_card: CardId,
       },
   }
   ```

2. **Update calculate_entity_dimensions() for VisualStack**
   ```rust
   BattlefieldEntity::VisualStack { card_ids, .. } => {
       let stack_depth = card_ids.len();
       let offset = 1; // characters per stacked card
       (
           base_width + (stack_depth as u16 - 1) * offset,
           base_height + (stack_depth as u16 - 1) * offset,
       )
   }
   ```

3. **Implement render_visual_stack()**
   - Calculate diagonal offsets for each card in stack
   - Render bottom cards as border outlines only
   - Render top card with full content
   - Show tapped cards on top if any are tapped

4. **Update group_cards_into_entities()**
   - Add `enable_visual_stacking: bool` parameter
   - When enabled, create `VisualStack` instead of `SimpleStack`
   - Group by card name only (not tapped state)
   - Track `tapped_count` within the stack

5. **Handle mixed tapped/untapped in visual stack**
   - Render untapped cards at bottom
   - Render tapped cards on top (rotated)
   - Visual indication: top N cards appear tapped

6. **Test visual stacking thoroughly**
   - Various stack depths (2-10 cards)
   - Mixed tapped/untapped states
   - Mouse interaction still works
   - Choice highlighting still works

## Implementation Challenges

- Complex geometry calculations for diagonal offsets
- Rendering partial cards (borders only) behind the top card
- Handling mixed tapped/untapped states requires careful ordering
- Hit testing still works (bounding box from simple stacking handles this)
- Layout algorithm already handles variable dimensions (from simple stacking)

## Benefits

- More compact battlefield representation (better than simple stacking)
- Visually similar to physical MTG gameplay
- Easier to see total count of each card type at a glance
- Reduces horizontal space usage significantly

## Relationship to Simple Stacking (mtg-148)

Simple stacking provides:
- "3x Island" text-based representation
- All necessary abstractions (BattlefieldEntity, EntityPosition, etc.)
- Proven grouping and rendering logic

Visual stacking builds on this by:
- Adding diagonal offsets and partial rendering
- Mixing tapped/untapped cards in same stack
- More visually appealing but same functionality
