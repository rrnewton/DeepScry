# Description

## Summary

A real determinism bug (NOT a logging artifact) surfaced during the debug-log
noise-reduction audit of a 56K-line WASM console capture. The network game DIED
at turn 13 because the WASM rewind/replay verifier detected a genuine divergence
between the forward simulation pass and the replay pass — a bottoming/scry-style
effect produces different results under rewind+replay.

## CONFIRMED STILL BROKEN + ROOT CAUSE NARROWED (2026-06-10_#3177(9e100acf), agent compat-2025-deepverify slot07)

Re-reproduced on current integration (9e100acf) NATIVELY via the
network-vs-local equivalence harness (`tests/network_vs_local_equivalence.py` /
`bug_finding/network_test_lib.py`) pointed at the exact user matchup
(decks/championship/2025/04_henry_temur_otters vs 02_shibata_izzet_lessons,
random/random). **ALL seeds tested FAIL** (1,3,5,7,9,13,42 — 7/7), so this is
NOT seed-specific or rare; the 04-vs-02 pairing desyncs essentially every game.
The matchup is UNPLAYABLE in network/multiplayer mode (it works fine in native
LOCAL single-process mode — local 04v02 random games complete cleanly across
seeds 7/42/13/100, ending in Decking/PlayerDeath with no FATAL).

### ROOT CAUSE (narrowed): multi-card Dig selection + bottom-ORDER is not replicated to the client

The divergence is a `DB$ Dig | ... | ChangeNum$ N` effect (look at top K, keep N,
put the rest on the BOTTOM in an order the controller chooses). The SMOKING-GUN
diff (seed=1):

```
SERVER:  [GAMELOG Turn5 M2] Ryan puts Roaring Furnace into Hand
CLIENT1: [GAMELOG Turn5 M2] Ryan puts 4 cards on the bottom of their library
SERVER:  [GAMELOG Turn5 M2] Ryan puts 3 cards on the bottom of their library
```

The SERVER's Dig keeps `Roaring Furnace` and bottoms 3; the CLIENT's shadow Dig
keeps a DIFFERENT card and bottoms 4 → instant state-hash divergence → FATAL
(`P2 state hash mismatch! server=... client=...`).

Mechanism: `ChoiceAccepted` carries only a SINGLE `library_search_result:
Option<CardId>` (network/choice_entry.rs:77) — enough to disambiguate a
one-card library search (typecycling, mtg-252 fix). But a Dig effect selects
MULTIPLE cards to keep AND a multi-card bottom ORDER. That richer selection is
NOT serialized into `ChoiceAccepted`, so the client's shadow game re-runs the
Dig against its own (hidden-information-shadowed) library view and picks
differently. This is the SAME class as the original turn-13 report — just
caught EARLIER and on MORE cards now.

### Affected cards in the 2025 decks (all `DB$ Dig | ChangeNum`)

- Stock Up (cardsfolder/s/stock_up.txt): `DB$ Dig | DigNum$ 5 | ChangeNum$ 2`
  (look at 5, keep 2, bottom 3). Decks 03, 04. The original turn-13 culprit.
- Accumulate Wisdom (cardsfolder/a/accumulate_wisdom.txt): a Dig that "digs N
  from library to hand, puts rest on bottom" — observed diverging at turn 9.
  Decks 01-04.
- (Any other look-at-top-N-keep-some card shares the gap.)

## Suspected fix area (DO NOT fix here — netarch scope)

Extend the server→client choice replication so a Dig / look-at-top-N effect
sends the FULL server-authoritative outcome: which CardIds were kept (to hand /
to top) and the exact ORDER of the cards put on the bottom — analogous to the
existing single `library_search_result`, but a list. The client's shadow game
must CONSUME that server-authoritative selection instead of re-deciding from its
own library view (controllers must be information-independent; the Dig "keep
which" choice depends on hidden library contents the client cannot legally see).
See CLAUDE.md "Controllers must be information-independent" + the mtg-252
typecycling precedent (`library_search_result` in ChoiceAccepted).
