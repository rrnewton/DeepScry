### ROOT CAUSE (narrowed): multi-card Dig selection + bottom-ORDER is not replicated to the client

The divergence is a `DB$ Dig | ... | ChangeNum$ N` effect (look at top K, keep N,
put the rest on the BOTTOM in an order the controller chooses). The SMOKING-GUN
diff (seed=1):

```
SERVER:  [GAMELOG Turn5 M2] Ryan puts Roaring Furnace into Hand
CLIENT1: [GAMELOG Turn5 M2] Ryan puts 4 cards on the bottom of their library
SERVER:  [GAMELOG Turn5 M2] Ryan puts 3 cards on the bottom of their library
```

The SERVER's Dig keeps `Roaring Furnace` and bottoms 3; the CLIENT's shadow Dig
keeps a DIFFERENT card and bottoms 4 â†’ instant state-hash divergence â†’ FATAL
(`P2 state hash mismatch! server=... client=...`).

Mechanism: `ChoiceAccepted` carries only a SINGLE `library_search_result:
Option<CardId>` (network/choice_entry.rs:77) â€” enough to disambiguate a
one-card library search (typecycling, mtg-252 fix). But a Dig effect selects
MULTIPLE cards to keep AND a multi-card bottom ORDER. That richer selection is
NOT serialized into `ChoiceAccepted`, so the client's shadow game re-runs the
Dig against its own (hidden-information-shadowed) library view and picks
differently. This is the SAME class as the original turn-13 report â€” just
caught EARLIER and on MORE cards now.

## FIX PLAN (design/scoping only â€” 2026-06-10_#3180(d7cc1319), agent mtg908-fixplan slot07)

NO source edited in this pass. This section maps the exact fix, classifies it
A (network-only, parallel-safe) vs B (effects-overlapping, blocks on the
execute_effect refactor mtg-245), and gives a regression-test outline.

### Q1 â€” is this "wired to the wrong protocol message", or "protocol can't express it"?

NEITHER. It is a THIRD thing: **the Dig effect submits NO controller choice at
all** â€” there is no protocol message in play, wrong or right. The "look at top N,
keep some, bottom the rest" decision is made by an INLINE LOCAL AI HEURISTIC that
peeks at the actual (hidden) library contents:

- `GameState::execute_effect` â†’ `Effect::Dig { .. }` arm at
  `mtg-engine/src/game/actions/mod.rs:5591`.
- The "which cards to keep" decision is `valid_ids.sort_by(dig_card_score)` at
  `mtg-engine/src/game/actions/mod.rs:5678`, scoring fn `dig_card_score` at
  `mtg-engine/src/game/actions/mod.rs:6547`.
- The "bottom the rest" application is the `add_to_bottom` loop at
  `mtg-engine/src/game/actions/mod.rs:~5765`.

There is NO `controller.choose_*` call anywhere in the Dig arm. So on a network
game the SERVER runs `dig_card_score` against the real top-N, while the CLIENT
SHADOW runs the SAME `dig_card_score` against its shadow library â€” whose top-N
card identities are HIDDEN and therefore differ â€” and the two pick different
cards / bottom counts. Smoking-gun (mtg-908 repro): server keeps "Roaring
Furnace" + bottoms 3, client bottoms 4. This also violates CLAUDE.md
"controllers must be information-independent" â€” except the hidden-info peek is in
the EFFECT, not a controller, which is why it was never caught by controller
audits.

The `protocol.rs` top/bottom partition variant the scoping note referenced â€”
`ChoiceType::Scry { count, revealed_card_ids }` at
`mtg-engine/src/network/protocol.rs:1313` â€” DOES exist and DOES correctly express
a multi-card top/bottom partition with bottom-ORDER (client returns `indices` of
cards to bottom; first index â†’ deepest). It is the right SHAPE to copy. But Dig
is NOT routed through it (nor through any controller method). The
single-card `library_search_result: Option<CardId>` in
`ChoiceAccepted` (`mtg-engine/src/network/choice_entry.rs`) is a red herring for
Dig â€” Dig never reaches the library-search path either.

### Q2 â€” how Scry does it right (the pattern Dig must mirror)

Scry is INTERCEPTED in the effect-resolution loop BEFORE `execute_effect`, where
the `controller` handle is in scope:

- `mtg-engine/src/game/game_loop/priority.rs:2180` (and a second site at 3731):
  `let choice = controller.choose_scry_order(&view, &revealed);` â€” after
  `push_reveals` + `sync_to_action()`.
- Controller trait method: `choose_scry_order` â€” default in
  `mtg-engine/src/game/controller.rs:1874`; network impl in
  `mtg-engine/src/network/controller.rs:1144` (sends `ChoiceType::Scry` with
  server-authoritative `revealed_card_ids`, validates returned `indices`, builds
  `ScryDecision { top, bottom }`); random impl `random_controller.rs:284`; fixed
  `fixed_script_controller.rs:327`.
- `execute_effect`'s Scry arm is only a NO-OP fallback (keep all on top) â€”
  `mtg-engine/src/game/actions/effects/cards.rs:execute_scry` (the refactor
  already moved Scry into effects/cards.rs; the controller-dispatched real work
  stays in priority.rs).

So the SERVER decides the partition, sends the revealed CardIds + receives the
client's chosen bottom indices; the CLIENT never re-decides from hidden info.
That is exactly what Dig needs.

### Q3 â€” minimal fix sketch

Introduce a Dig partition choice mirroring Scry, but with a THREE-WAY partition
(to destination [hand/exile/etc.] vs kept-on-top vs bottomed) instead of Scry's
two-way:

1. PROTOCOL (network layer, A): add `ChoiceType::Dig { dig_count, change_count,
   change_all, destination, rest_destination, revealed_card_ids: Vec<CardId> }`
   in `mtg-engine/src/network/protocol.rs` (next to `Scry`/`Surveil`). Response
   reuses the existing `indices` field of the choice reply: the client returns
   the positions (into `revealed_card_ids`) of the cards to MOVE to `destination`
   (the "keep" set); the rest go to `rest_destination` in revealed order
   (matching today's valid-then-invalid behavior, OR an explicit bottom-order
   list if we want player-chosen bottom ordering â€” Stock Up/Accumulate Wisdom are
   "any order" so an explicit order is technically correct but a fixed order is
   determinism-safe and rules-legal since bottom order of unrevealed-to-opponent
   cards is unobservable).
2. CONTROLLER TRAIT (A): add `fn choose_dig_partition(&mut self, view,
   revealed: &[CardId], change_count, change_all, destination, rest_destination)
   -> ChoiceResult<DigDecision>` to the `PlayerController` trait
   (`mtg-engine/src/game/controller.rs`), with a `DigDecision { to_destination:
   Vec<CardId>, rest: Vec<CardId> }`. Impls:
   - network (`network/controller.rs`): send `ChoiceType::Dig`, validate indices,
     build `DigDecision` from the server-authoritative `revealed` â€” COPY
     `choose_scry_order` almost verbatim.
   - default/heuristic (`controller.rs`): MOVE the existing `dig_card_score`
     ranking here (it becomes a legitimate controller decision over the
     controller's OWN view â€” which on the server is the real library, and the
     server's decision is then shipped to the client; the client's impl is never
     consulted because the network controller takes the server's answer).
   - random (`random_controller.rs`), fixed (`fixed_script_controller.rs`):
     simple deterministic pick (first `change_count`), mirroring their
     `choose_scry_order`.
3. INTERCEPTION SITE (game_loop, NOT effects): add an `Effect::Dig { .. }` arm in
   the `priority.rs` effect-resolution loop (next to the `Effect::Scry` arm at
   `priority.rs:2180`/`3731`) that: snapshots top-N, `push_reveals` to BOTH
   players, `sync_to_action()`, calls `controller.choose_dig_partition`, then
   applies the returned `DigDecision` via a new pure
   `dig_apply_decision(player, &revealed, &decision)` helper (mirroring
   `scry_apply_decision`).
4. EFFECT ARM (effects layer, B â€” THE OVERLAP): the existing
   `actions/mod.rs:5591` Dig arm must be REDUCED from "decide-with-hidden-info +
   apply" to a controller-less NO-OP/fallback (apply a default keep-first-N,
   exactly like `execute_scry`'s fallback) â€” because the real decision now
   happens in priority.rs. The hidden-info `dig_card_score` sort (5678) and the
   bottoming loop move out / become the fallback.

Server/client cross-check to compare: after the choice, both sides hold the same
post-Dig library + destination zone, so the existing turn-start / per-action
state-hash (`SRV_P1_RECV ... client_hash vs server_hash`) is the verification â€”
no NEW cross-check needed; the fix simply makes the two sides AGREE by having the
server dictate the partition. The opponent's `push_reveals` must NOT leak the
dug card identities to the opponent (CR 701 / hidden info): send `ChoiceType::Dig`
ONLY to the digging player's controller, exactly as `ChoiceType::Scry` is gated
(see protocol.rs:1297 visibility note).

### Q4 â€” A-vs-B VERDICT: **(B) EFFECTS-OVERLAPPING â€” blocks on mtg-245**

- Parts 1â€“3 (protocol variant, controller trait+impls, priority.rs interception)
  are network/game_loop layer and DISJOINT from the refactor â€” those could be
  staged in parallel.
- BUT Part 4 is unavoidable: the current Dig arm at
  `mtg-engine/src/game/actions/mod.rs:5591` MAKES the hidden-info decision, so it
  MUST be edited (gutted to a fallback) for the fix to actually remove the
  desync. That arm is squarely inside mtg-245's decomposition map (the `zones:
  ...Dig...` family). On the refactor branch `claude/refactor-execute-effect`,
  Dig is STILL at `actions/mod.rs:5591` (zones.rs not yet extracted; only
  cards/counters/damage/life are done) and WILL move to
  `mtg-engine/src/game/actions/effects/zones.rs`. Editing the Dig arm now would
  create exactly the conflicting parallel edit we must avoid.

RECOMMENDATION: SEQUENCE it. Either (a) wait for mtg-245 to extract the `zones`
family (Dig â†’ effects/zones.rs), THEN do all 4 parts with Part 4 landing in the
new zones.rs; or (b) coordinate a single combined change with the refactor owner
so the Dig extraction AND the controller-dispatch rewrite land together (the
extraction is the natural moment to also split Dig into "priority.rs dispatch +
zones.rs fallback", exactly as Scry was split into priority.rs + effects/cards.rs).
Parts 1â€“2 (protocol `ChoiceType::Dig` + the controller trait method/impls) are
network-only and MAY be pre-staged now without touching effects, leaving only the
priority.rs interception + the mod.rs/zones.rs fallback gutting for the
