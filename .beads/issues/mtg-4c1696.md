# Description

FIXED 2026-05-12 (compat1).

Root cause: Chaos Orb's FlipOntoBattlefield was mapped to Effect::DestroyPermanent with restriction.requires_nontoken=true and ControllerRestriction::Any. The targeting code at mtg-engine/src/game/actions/targeting.rs:798 used restriction.matches() (no controller check) and the auto-target picker selected the first valid permanent — often Chaos Orb itself.

Fix:
1. mtg-engine/src/loader/effect_converter.rs: FlipOntoBattlefield converter now sets restriction.controller = ControllerRestriction::OppCtrl. This restricts valid_targets to opponent permanents only.
2. mtg-engine/src/game/actions/targeting.rs: DestroyPermanent placeholder branch in get_valid_targets_for_ability now uses restriction.matches_with_controller(card, ability_controller, card.controller) instead of restriction.matches(card). This makes the controller restriction actually take effect for activated abilities (the spell-side targeting was already correct).

Tests added:
- mtg-engine/src/loader/effect_converter.rs: test_convert_flip_onto_battlefield_targets_opponent (unit test pinning the converter contract)
- tests/chaos_orb_targets_opponent_e2e.sh: end-to-end shell test that activates Chaos Orb on a puzzle board and verifies (a) targeting Mountain (opponent permanent), (b) Mountain goes to graveyard, (c) Chaos Orb self-destroys, (d) Chaos Orb does NOT self-target.

Gameplay evidence (test_puzzles/chaos_orb_destroys_target.pzl):
  Chaos Orb activates ability: ...
    -> targeting Mountain (16)
  Mountain (16) goes to graveyard
  Chaos Orb (3) goes to graveyard

Test result: 710 lib tests pass; 50 puzzle integration tests pass; new shell test passes.
