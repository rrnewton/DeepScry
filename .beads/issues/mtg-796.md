# Description

PRIZE BLOCKER: robots seed-19 Fireball shadow divergence -> client sends illegal choice index (turn 24)

## RESOLVED 2026-06-05 (slot03-seed19) — ROOT CAUSE WAS NOT ANIMATION; it was the WASM "0 targets" index encoding

The earlier "animated man-land 51 stale-creature" hypothesis (slot03-lockstep checkpoint) was DISPROVEN by direct WASM-side instrumentation. Empirical facts (mtime-fresh native+wasm, DEEPAC_-prefixed log::warn surfaced via the e2e deepac_wasm.log):
- At the rewind apply site (unwind_state_sync_to, R=1675) EVERY Mishra's Factory on the WASM shadow is correctly is_creature=FALSE (plain Land), card 51 included. The shadow does NOT keep a stale animation. So no animation reconstruction is needed.
- At the actual Fireball target computation (get_valid_targets_for_spell, traced on the shadow) the browser computes EXACTLY the same 2 valid creature targets as the server: valid=[119,110] (WebRandom's own two factories animated THIS turn). Card 51 is correctly excluded. The target SET matches; the predecessor's "browser sees 3 targets" was an unverified inference, not measured.

THE REAL BUG (protocol parity): Fireball is a DivideEvenly X-spell with target bounds (0, num_valid). WebRandom's RNG legally rolled count=0 ("choose zero of the 2 targets"). The WASM local controller encoded "no targets" as a SENTINEL index `vec![valid_targets.len()]` = `vec![2]` (wasm/network/local_controller.rs choose_targets). But the server's NetworkController::choose_targets builds exactly `valid_targets.len()` options (NO trailing "none" entry) and validates `idx < options.len()`, so index 2 is out of range -> FATAL desync ("invalid choice index 2, only 2 options"). The animation was only a PRECONDITION: animating 110/119 created the 2 creature targets that made a 0-of-2 choice possible. Creature-state is not in compute_view_hash, so hashes matched up to the fatal — consistent with, but not caused by, an animation divergence.

THE FIX (DRY, WASM<->native parity): the NATIVE NetworkLocalController::choose_targets already encodes the chosen targets as `filter_map(position).collect()` — an EMPTY index list for 0 targets — and the server decodes an empty list as "0 targets chosen". The WASM client was the only divergent path. Removed the `if empty { vec![len] }` sentinel branch in BOTH wasm choose_targets and choose_permanents_to_sacrifice (same provable class: server has no len-sentinel option, native sends empty). Now 0-chosen round-trips byte-identically. File: mtg-engine/src/wasm/network/local_controller.rs.
