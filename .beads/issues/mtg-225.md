# Description

## Eliminate Hacky Substring Operations on Card Scripts

## Problem

We had 17+ instances of hacky `.contains()` calls on card script bodies in `card.rs` that bypassed proper tokenized parsing:

```rust
// BAD - what we had:
if body.contains("AB$ Draw") || body.contains("DB$ Draw") { ... }
if body.contains("DB$ PutCounter") { ... }
if body.contains("DB$ GainLife") { ... }
```

These were fragile because:
- `contains("Damage")` matches "DealDamage", "PreventDamage", "AllDamage"
- `contains("add")` would match "Madden", "adding"
- O(n) per call vs O(1) map lookup after tokenized parse

## Solution

We ALREADY had proper parsing infrastructure in `ability_parser.rs`:
- `AbilityParams::parse()` - tokenizes by `|` and `$`
- Returns structured data with `api_type: ApiType`, `params: HashMap`

Refactored all card script parsing in `card.rs` to use `AbilityParams::parse()` instead of raw substring matching.

## Implementation (2026-01-08_#1582)

### New Helper Functions in ability_parser.rs

Added three helper functions for safe API type checking:

```rust
/// Parse an SVar body (without prefix) and return the params if successful
pub fn parse_svar_body(body: &str) -> Option<Self>

/// Check if an SVar body has a specific API type
pub fn is_api_type(body: &str, expected: ApiType) -> bool

/// Check if an SVar body has any of the specified API types
pub fn is_any_api_type(body: &str, expected: &[ApiType]) -> bool
```

### Refactored Sections

All 17 `.contains("AB$|DB$ ...")` calls in card.rs were replaced with tokenized parsing:

1. **extract_token_scripts()** - Uses `AbilityParams::parse_svar_body()` + `ApiType::Token`
2. **ETB Triggers** - Token, ChangeZone, Draw+Discard (looting), Earthbend
3. **Dies Triggers** - Mana effects
4. **Phase Triggers** - DealDamage, GainLife, Earthbend
5. **Attack Triggers** - Draw, PutCounter, GainLife, DealDamage, Mana
6. **SpellCast Triggers** - PutCounter, Pump
