# Description

PRINCIPLE (user rrnewton): deterministic processing over a monotonic/replicated store (I-vars, Kahn Process Networks) must NEVER react to ABSENCE of information — branching on `cards.try_get(id) -> None` is a desync red flag because absence is not stable/agreed across the server 'golden' state and the client 'shadow' state. mtg-mb668 sig-2c was exactly this (shadow skipped opponent reserved-ID cards on try_get=None -> different Fisher-Yates RNG draw count -> desync). TASK: audit EVERY use of try_get/try_get_mut (+ any other absence checks) in mtg-engine; for each, does behavior BRANCH on None, and is that branch provably shadow-safe? Eliminate/minimize None-driven control flow (prefer total ops / identity-independent reserved-ID handling).

========================================================================
AUDIT COMPLETE 2026-06-03 (slot01-2, READ-ONLY; on integration tip 329133dc). 229 try_get/try_get_mut sites enumerated across mtg-engine, classified by 5 parallel read-only passes, every RISK site re-verified by reading the code.

CLASSIFICATION RUBRIC:
  SAFE-TOTAL      = None & Some converge (error-propagating `?`; logging/display-only read; None path == no-mutation identical no-op).
  SAFE-SYMMETRIC  = branches on None BUT the id is provably always a real instance on BOTH server+shadow here (battlefield permanent = public both sides; the active player's OWN hand/library/graveyard; a just-created card; a resolving stack object; a bound chosen target).
  RISK-SHADOW     = branches on None where the id COULD be a reserved/instance-less OPPONENT card on a shadow but real on the server (the sig-2c anti-pattern) AND the branch affects state/log/RNG/count.
  RISK-REWIND     = branches on None where absence could differ across a rewind/replay of the SAME side.
  VIEW-ONLY/INFRA = display/view-model builders (cannot affect engine state/choices/hash) / accessor definitions / test-only.

HEADLINE FINDINGS:
  1. The wasm SHADOW display code (fancy_tui.rs, gui_view_model.rs: 11 sites) and the network/server.rs golden-side I/O (4 sites) are ALL SAFE — every None branch only omits a card from rendered JSON / suppresses an outbound CardRevealed message; none feeds engine state, choices, hashed counts, or RNG. wasm/network/remote_controller.rs & wasm/replay_verifier.rs have ZERO cards.try_get sites. => class-A desync is NOT rooted in the shadow's own wasm code.
  2. EVERY genuine RISK-SHADOW site is in the SHARED ENGINE (actions/mod.rs, state.rs, game_loop/steps.rs) — branch-on-absence in code that runs identically on server & shadow but observes opponent reserved IDs differently. THESE shared-engine sites ARE the class-A root-cause candidates (slot03's workstream). Fixing them = the principled class-A close.
  3. The overwhelming majority (122 of ~155 real branch sites) are SAFE-SYMMETRIC because they iterate self.battlefield.cards (public both sides) or the active player's OWN zones, or operate on a resolving stack object / bound target.

RISK-SHADOW SITES (class-A root-cause candidates — OWNED BY slot03 class-A; DEFER here):

[R1] actions/mod.rs:1624  count_cards_in_zone_matching — HIGHEST CONFIDENCE.
  filter returns false on try_get=None. Zone selector supports OppOwn/OppCtrl; for Zone::Hand/Zone::Library the opponent's cards are reserved IDs on a shadow -> excluded from the count, but counted on the server. Any effect whose magnitude depends on opponent hand/library size (X = count, cost reduction by count, damage = count) diverges. Graveyard/Exile are public so only hidden zones bite. VERIFIED (read).

[R2] actions/mod.rs:4312  Effect::ChangeZoneAll private-zone origin — PARTIALLY MITIGATED.
  Already special-cases shadow reserved cards via move_reserved_in_shadow = is_shadow_game && restriction.is_unrestricted() (the original sig-2c fix), but ONLY for the UNRESTRICTED case. A RESTRICTED mass move listing Hand/Library as origin still hits `None => {}` and skips opponent hidden reserved cards on the shadow -> divergent move count -> if shuffle, divergent RNG. Reachability depends on whether any supported restricted card lists Hand/Library as origin.

[R3] actions/mod.rs:4785 + 6432  library search / dig_card_score — RISK only for NON-SELF (opponent) library.
  Self-targeted library search/dig (the dominant pattern) is SAFE-SYMMETRIC (own library, reconciled by the choose_from_library hook). Becomes RISK-SHADOW only if an effect searches/digs an OPPONENT's library: shadow sees reserved (None -> no match / score 0) vs server's real cards -> different pick / RNG. Confirm reachability of any non-self Dig/Search caller.

[R4] state.rs:3488  DelayedEffect::ReturnToBattlefield — RISK-SHADOW, NOT mitigated.
  `if let Some(card)=try_get(card_id){...} else { return Ok(()) }` — branches on absence to SKIP the entire return (move + RevealCard). A delayed trigger's tracked_card can be an opponent's reserved instance-less id on a shadow (it tracks a card across hidden zones), so server fires the return while shadow skips it -> divergent MoveCard/Reveal count + zone state. Returns BEFORE move_card, so the move_card shadow-tolerance fallback does NOT cover it. VERIFIED (read). Exact sig-2c shape.

[R5] state.rs:3539  DelayedEffect::ExileCard owner default — RISK-SHADOW, partially mitigated.
  owner = try_get(card_id).map_or(controller, |c| c.owner) with NO zone==Battlefield gate (unlike the Sacrifice arm at 3529); a reserved opponent card yields a wrong (controller-defaulted) owner vs the server's real owner into move_card. The shadow `!removed` early-return in move_card (≈1371) likely absorbs it incidentally; harden to the symmetric pattern (state.rs:1214) rather than rely on incidental convergence.

[R6] game_loop/steps.rs:236  exile-zone trigger scan — RISK-SHADOW, low likelihood.
  `if try_get(id).is_some_and(matches_trigger){ push }` over every player's exile, then EXECUTES those triggers (mutation + possible RNG). A face-down/hidden opponent exiled card (foretell/hideaway-style) with a TriggerZones$ Exile trigger is reserved (None) on the shadow -> trigger silently skipped while server fires it. Battlefield scan (steps.rs:223) is safe (public); exile is the one zone here that can hold a hidden opponent card. Fix pattern: push_reveals/sync exile-trigger candidates first (mtg-253 library pattern), or treat None-on-candidate as fatal.
