== MEASUREMENT 2026-06-09_#3045(bb0fdd18) (slot04 validate-streamline-2): TCP_NODELAY (f216c13a) impact on the validate network chain ==
Full `make validate` baseline @bb0fdd18 on the 16-core dev box: wall-clock 648s (~10.8min), serial-sum 1353s, parallel speedup 2.09x on -j16, CPU avg 38.7% (70.5% of time <50% util — long browser-serialized tail starves cores).

TCP_NODELAY VERDICT: dramatic on the NATIVE-network e2e steps (the historically soak-bound ~190s ones are now near-instant): network.equiv-random 4s, equiv-zero 2s, equiv-heuristic 3s, robots42 0s, fuzz 5s. Substantial on the browser GUI baseline: network.gui 76s (was ~190s per this issue). network.human-input 19s, click 20s, landing 43s, redo-reload 28s, redo-lobby 20s, smoke 3s, deploy-nav 2s.

NOT improved: network.multideck still 191s (mtg-726 recorded ~191s). Its wall-clock is N-decks x per-deck headless-browser SESSION setup (each deck = fresh page/server handshake), NOT per-choice Nagle latency — so TCP_NODELAY (which fixed per-choice 28ms->~0) leaves it unchanged. network.multideck is now the single dominant pole; the remaining tail is the cap-1 browser-resource serialization (wasm.browser 125s + 4 equiv + 10 net-browser steps, all one-at-a-time).

slowest steps @bb0fdd18: examples.run 252s, lint.clippy 221s, network.multideck 191s, wasm.browser 125s, unit.nextest 114s, wasm.bundle 81s. The CPU front (examples/clippy/nextest) overlaps fine; the browser tail does not. Cutting multideck (parallelize decks within the step, or split into sub-jobs) is the highest-leverage remaining win — deferred to the focused profiling session this issue already scopes.

== LOAD-FLAKE INSTANCE 2026-06-09_#3053(f3f5506c) (slot04 validate-streamline-2) ==
Captured a network.multideck flake under HEAVY box saturation (~4-5 concurrent cross-slot validates, load ~26 on 16 cores) as additional load-sensitivity evidence. Scenario 1/4 (decks/monored.dck seed=13, mirror) FAILED after 28.2s exit 1.

SIGNATURE = NOT a desync. The game ran cleanly to Turn 13 with 143 SRV_P1_RECV exchanges, and EVERY one had client_hash==server_hash AND client_ac==expected_ac (0 mismatches / 0 divergence — quantified by grep). The detail log truncated mid-line at Turn 13, i.e. the JS test-harness exited on its OWN internal deadline (NOT a harness SIGKILL — no 'STEP TIMED OUT' marker; the validate-runner did not externally kill it). One transient 'Connection closed before message' at connect (normal retry). So this is the load-induced test-harness-deadline flake, distinct from the FIXED mtg-586 monored desync (which had real hash mismatches).

This instance corroborates mtg-726's core thesis: network.multideck (runs 4 browser+server+AI scenarios in PARALLEL) is the most resource-intensive + timing-sensitive validate step, and flakes under contention WITHOUT any correctness problem. Detail saved to slot04 debug/multideck_seed13_failure_20260609.log. Sibling network.redo-reload reportedly hit its reveal-divergence flake under the same load and passed on isolated retry — same load-sensitivity class.

Actionable (already in mtg-726 scope): the --quick multideck soak should be made more load-robust (longer per-scenario deadline, or fewer parallel scenarios, or run it under a per-step memory/cpu reservation) so concurrent cross-slot validates don't false-fail it. The new --max-mem cgroup containment (mtg-887) helps memory but not CPU contention.
