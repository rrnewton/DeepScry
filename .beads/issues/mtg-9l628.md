# Description

Engine gap (bug class): triggered abilities with a target (e.g. The Abyss's
"destroy target nonartifact creature that player controls of their choice")
do NOT go on the stack and do NOT route the target choice through the
controlling player's PlayerController. Instead the engine auto-selects a
target deterministically (lowest CardId among legal targets) in
`GameState::choose_triggered_destroy_target` (mtg-engine/src/game/actions/mod.rs),
shared by `check_triggers` and `check_triggers_for_controller`.

This matches the EXISTING project convention for triggered abilities
(see the pre-existing DestroyPermanent / AttachEquipment auto-target arms in
`check_triggers`) and preserves network determinism / controller
information-independence (docs/NETWORK_ARCHITECTURE.md): the same visible
state always yields the same target on server and client.

However it deviates from MTG rules CR 603.3d / 115: the ABILITY'S CONTROLLER
should choose the target when it is put on the stack. For The Abyss the
"of their choice" clause means the ACTIVE player picks which of their own
nonartifact creatures dies. Today the engine picks for them (lowest CardId).
Gameplay is correct in that SOME legal active-player nonartifact creature is
destroyed and can't be regenerated; only the CHOICE authority differs.

Real fix (large, cross-cutting): put triggered abilities on the stack as
distinct objects and request the target via ChoiceRequest to the controlling
player, mirroring the spell-cast targeting path
(`choose_targets_with_hook` in game_loop). This affects every targeted
triggered ability, not just The Abyss.

Discovered while bringing The Abyss (mtg-sgkjv) to WORKING for the 1994
Old School playtest (mtg-pph0s). The Abyss is functionally correct; this
issue tracks upgrading auto-target to true controller choice.
