# Description

## Background

CPU profiling with both Callgrind (instruction count) and Linux perf (wall-clock time) identified the top hotspots.

**NEW: Linux Perf Profiling (2025-11-29_#966)**

Wall-clock sampling (5000 games, 8.09B cycles, 2,838 games/sec):

1. **priority_round** (17.75% CPU) - game_loop/priority.rs
   - 3.2% in push_activatable_abilities
   - 2.2% in HashMap probing
   - 1.3% in EntityId comparisons
   
2. **ManaEngine::update** (15.00% CPU) - mana_engine.rs
   - 2.9% general overhead
   - 2.4% Vec::push operations
   - 1.5% card type checks
   - 1.1% HashMap lookups

3. **String operations** (8.89% combined)
   - 3.25% str::is_contained_in
   - 2.15% String::write_str
   - 1.49% to_lowercase (mostly cached already)
   
4. **bounds_check_payment** (4.79% CPU) - mana_payment.rs

**Previous: Callgrind Profiling (2025-11-08_#831)**

Instruction count (250 games, 18.8B instructions):

1. **ManaEngine::update** (30.5% CPU, 5.7B instructions)
2. **cast_spell_8_step** (23.3% CPU, 4.4B instructions)
3. **GreedyManaResolver::check_payment** (14.3% CPU, 2.7B instructions)

## Key Findings

### Perf vs Callgrind Comparison

**Why different rankings?**
- **Perf (wall-clock):** Includes cache misses, memory stalls → priority_round #1
- **Callgrind (instructions):** Pure CPU work → ManaEngine #1
- **Both valid:** Complementary views of performance

**Agreement:**
- ManaEngine confirmed hotspot (15% perf, 30.5% Callgrind)
- String operations significant (8.89% perf)
- Mana payment overhead (4.79% perf, 14.3% Callgrind)

**Perf-specific insights:**
- priority_round dominated by HashMap lookups (not in Callgrind top 3)
- EntityStore lookups = 3% overhead across multiple call sites
- String operations memory-bound (not instruction-bound)

### Cross-Reference: CPU vs Allocations

| System | CPU Hotspot (Perf) | CPU (Callgrind) | Allocation Hotspot | Root Cause |
|--------|-------------------|-----------------|-------------------|------------|
| Action Gen | priority_round (17.75%) | - | get_available_spell_abilities (51 KB) | HashMap lookups + action list generation |
| Mana System | ManaEngine::update (15%) | 30.5% | mana_engine.rs:550 (ELIMINATED) | Vec growth + card type checks |
| Spell Casting | cast_spell_8_step (2.05%) | 23.3% | actions.rs (1.71 MB) | Error recovery paths |
| Mana Payment | bounds_check_payment (4.79%) | 14.3% | mana_payment.rs:580 (2.57 MB) | Greedy algorithm O(n²) |
| Strings | 8.89% combined | - | CardCache (MOSTLY CACHED) | Pattern matching + formatting |

**Pattern:** Recompute-every-time architecture that repeatedly:
- Scans battlefield for mana sources (ManaEngine)
- Generates action lists from scratch (priority_round)
- Allocates temporary Vecs
- Performs O(n) lookups in HashMaps

## Optimization Roadmap (Updated with Perf Results)

### Quick Wins (1-2 days, 10-15% CPU reduction)

**Priority order by risk/reward:**

1. **OPT-1: Cache land type flags in CardCache** (EXISTING from Callgrind analysis)
   - Pre-compute is_land, is_creature, is_artifact as bool flags
   - Impact: 4.5% CPU reduction (Callgrind), eliminates 1.5% card type checks (Perf)
   - Effort: 1-2 hours
   - Risk: Low (CardCache already exists)

2. **OPT-7: Reuse candidates Vec in mana payment** (EXISTING from Callgrind)
   - Pre-allocate and clear instead of reallocate
   - Impact: 4.3% CPU (Callgrind) + 2 MB allocation, reduces 4.79% payment overhead (Perf)
   - Effort: 30 min
   - Risk: Low (simple buffer reuse)

3. **OPT-STR-1: Complete mana color bitfield migration** (NEW from Perf)
   - get_simple_mana_color() still uses string comparisons (1.15% Perf)
   - ManaProductionKind already uses bitfield (b33ddea0)
   - Impact: 2% CPU reduction (eliminate string pattern matching)
   - Effort: 1 hour
   - Risk: Low (precedent exists)

4. **OPT-NEW-3: Pre-compute castability flags** (NEW from Perf)
   - Cache has_activated_abilities, is_castable_spell in CardCache
   - Impact: 2% CPU reduction (avoid repeated scans in priority_round)
   - Effort: 2 hours
   - Risk: Low (simple caching)

**Quick wins total: 10-13% CPU reduction**

### Medium Impact (2-4 days, additional 10-15%)

5. **OPT-NEW-1: Cache available actions** (NEW from Perf)
   - Invalidate on: card played, ability activated, phase change
   - Impact: 5-7% CPU reduction (priority_round: 17.75% → 10%)
   - Effort: 1 day
   - Risk: Medium (invalidation logic complexity)

6. **OPT-NEW-2: EntityStore Vec indices instead of HashMap** (NEW from Perf)
   - Replace HashMap<EntityId, T> with Vec<Option<T>>
   - Impact: 3% CPU reduction (eliminate hash probes across all call sites)
   - Effort: 1-2 days
   - Risk: Medium (memory overhead for sparse IDs, API changes)

7. **OPT-8: HashSet for tap_order lookup** (EXISTING from Callgrind)
   - Impact: 5.0% CPU (Callgrind)
   - Effort: 1 hour
   - Risk: Low

### High Impact, High Risk (1 week, additional 5-7%)

8. **OPT-INC-1: ManaTracker incremental caching** (RECOMMENDED in original analysis)
   - Recompute mana sources only on battlefield changes
   - Impact: 5-7% CPU reduction (ManaEngine: 15% → 8%)
   - Effort: 2-3 days
   - Risk: High (complex invalidation, undo/rewind integration)

**See ai_docs/perf_cpu_profiling_2025-11-29.md for detailed incremental design.**
