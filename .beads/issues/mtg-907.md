# Description

## Problem (DRY violation + correctness hazard)

Several DIVERGENT, hand-rolled string-based card-filter implementations each
re-implement a subset of MTG's Valid$ / ValidTgts type+subtype matching. They
disagree on edge cases (a latent correctness bug) and violate the "No Hacky
String Operations On Structured Data" rule by matching raw strings instead of
the structured TargetRestriction parser/matcher.

## Inventory (2026-06-10, integration base 939ac4c1)

CANONICAL structured matcher (consolidation target):
- core/effects.rs TargetRestriction::parse(&str) — full ValidTgts grammar
  (types, subtypes, controller, color, set-origin, powerGE/LE, cmc,
  nonToken/nonArtifact/nonCreature/Other/HasCounters).
- core/effects.rs TargetRestriction::matches(&Card) (+ matches_excluding,
  matches_with_controller).
- core/effects.rs TargetType::matches, DigFilter::matches (fold/share).

DIVERGENT duplicates to route through the canonical filter:
1. game/actions/mod.rs card_matches_type_filter_static — Land/Creature/Artifact/
   Enchantment, Permanent=>true, else subtype.
2. game/game_loop/actions.rs card_matches_type_filter — NEAR-IDENTICAL copy of #1.
3. game/actions/mod.rs card_matches_search_filter — comma-lists, dotted
   Type.Subtype, single-word disambig, Basic group. (callers in priority.rs)
4. game/actions/mod.rs card_matches_type — known types, else FALSE.
5. game/actions/mod.rs card_matches_subtype — subtype + Basic.

## Known divergences (rules-accurate behavior wins)

- Permanent: #1/#2 => true (matches instants/sorceries too); DigFilter::Permanent
  correctly excludes them. Masked today (battlefield-only calls), wrong in general.
- #4 returns FALSE for unknown tokens; #1/#2 fall through to subtype check, so
  "Goblin" matches via #1/#2 but not #4.
- Qualifiers (.Blue, .powerGE4, setARN, .nonArtifact, .Other) handled ONLY by
  TargetRestriction; the string copies silently ignore them.

## Plan

One canonical entry point (TargetRestriction::parse + matches_excluding, likely
a thin card_matches_valid wrapper); route all 5 sites through it. Rules-accurate
behavior wins on disagreement; flag+rules-justify each intentional change; add a
focused test per deliberate fix. Prove byte-identical vs the Phase-0 baseline
(debug/refactor_baseline.sh) for non-bugfix slices.

Related: mtg-376, mtg-877. Sibling to mtg-245; both on claude/refactor-execute-effect.

## Additional divergent filter found (2026-06-10)

6. game/actions/effects/pump.rs execute_animate_all (was inline in execute_effect) filters AnimateAll candidates with raw filter.contains("YouCtrl") / .contains("Creature") / .contains("Land") / .contains("Planeswalker") / .contains("OppCtrl") substring checks — a 6th divergent hand-rolled filter and a "No Hacky String Operations On Structured Data" violation. Its sibling PumpAllCreatures (effects/pump.rs execute_pump_all_creatures) correctly uses TargetRestriction::parse(filter) + matches_with_controller, so the two mass-effects disagree on filter parsing. Route execute_animate_all through TargetRestriction too.

## PROGRESS (2026-06-11, branch claude/mtg907-filter-consolidation off affe9bc4)

3 sites fixed, 0 fully deferred (1 partial-deferral documented):

- SITE 1 (86e2041a) — DRY-merge: GameLoop::card_matches_type_filter was a
  BYTE-IDENTICAL copy of GameState::card_matches_type_filter_static. Deleted the
  copy, routed its call site to the canonical (now pub(crate)) one. Pure DRY, no
  behavior change. validate 35/35. Also documented that this SIMPLE type/subtype
  filter is DISTINCT from card_matches_search_filter (Type.Subtype/comma-list
  fetch grammar) and TargetRestriction::parse (ValidTgts .-modifier grammar) —
  verified against cardsfolder: `ChangeType$ Land.Basic` uses .=subtype while
  `ValidTgts$ Creature.YouCtrl` uses .=modifier (SAME syntax, OPPOSITE meaning).
  So those are NOT duplicates of each other — see SITE-5 note below.

- SITE 2 (46c8638e) — execute_animate_all: replaced raw filter.contains(...)
  substring matching with TargetRestriction::parse + matches_with_controller
  (mirrors the sibling execute_pump_all_creatures). Behavior-identical for every
  shipping ValidCards$ string; ALSO fixes latent qualifier-dropping. validate 35/35.

- SITE 3 (bfc57f0a) — execute_force_sacrifice: replaced the hand-rolled
  `match sac_type { "Creature"=>..., _ => is_creature() }` with
  TargetRestriction::parse(sac_type).matches(card). FIXES real bugs:
  `SacValid$ Creature,Planeswalker` (planeswalkers were never sacrificable),
  bare subtypes (`Food`/`Mountain` fell to is_creature), `.nonArtifact` dropped.
  Added unit test test_force_sacrifice_comma_list_includes_planeswalker (proves
  the planeswalker is now a candidate). validate 35/35.
  PARTIAL DEFERRAL (documented, not guessed): DYNAMIC board-state predicates on
  SacValid$ — .attacking / .untapped / .withFlying / .sharesCardTypeWith / the
  `Self` selector — are resolved by NEITHER the old match NOR TargetRestriction
  (both degrade them). Extending TargetRestriction to model board predicates is
  the proper fix — OPEN QUESTION for the user (see below).

## REMAINING SITES

- card_matches_search_filter (+ its helpers card_matches_type / card_matches_
  subtype): this is the SearchLibrary / ChangeType$ fetch grammar (`Land.Basic`,
  `Plains,Island` comma-subtype-lists). It is NOT a duplicate of TargetRestriction
  — the `.` and `,` mean different things in the two grammars (SITE-1 note). So it
  is NOT a straight "route through TargetRestriction" consolidation. The cleaner
  end-state is a SINGLE structured ChangeType/search-filter parser (distinct from
  ValidTgts), but that is a larger, judgment-heavy refactor. Assessing next.
- execute_return_graveyard_card_to_hand: its type_filter is PRE-STRIPPED at
  convert time (effect_converter.rs:514 strips `.YouCtrl` then joins bare types
  with ','), so the effect only sees `"Instant,Sorcery"`. The str-match
  `match t { "Instant"=>..., _=>false }` is behavior-correct for the shipping
  use (Stormchaser's Talent — your own instant/sorcery). Smell, but no live bug.
  Converting to TargetRestriction would mean changing the effect's data model
  (type_filter: String -> restriction). Assessing whether worth it.

## OPEN QUESTION FOR THE USER (do not guess)

`SacValid$` (and `Valid$` generally) carries DYNAMIC board-state predicates that
the static TargetRestriction grammar does not model: `.attacking`, `.untapped`,
`.withFlying`, `.sharesCardTypeWith<...>`, plus the `Self` selector. Today both
the old hand-rolled matches AND TargetRestriction silently DROP these (degrading
e.g. `Creature.attacking` to bare `Creature`). Properly supporting them requires
extending TargetRestriction (or a sibling matcher) with board-state predicate
evaluation that takes `&GameState` (not just `&Card`) — a real feature, not a
cleanup. SHOULD this be done as part of mtg-907, or filed as its own feature
issue? Deferring the decision to you rather than guessing.
