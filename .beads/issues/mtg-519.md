# Description

Test all behavioral aspects of Mana Drain in MTG Forge-rs.

Card: cardsfolder/m/mana_drain.txt
Set: LEG (Legends)
Deck: thedeck Peterschnidrig (mtg-413, main deck)
Test puzzle: test_puzzles/mana_drain_deferred_mana.pzl

Card text:
  Mana Drain {U}{U} Instant
  Counter target spell. At the beginning of your next main phase, add
  an amount of {C} equal to that spell's mana value.

Findings (2026-05-30_#2538(58715f3f base), compat-wave19-manadrain):

1. [x] Parses as {U}{U} Instant; SP$ Counter works end-to-end.
2. [FIXED] DB$ DelayedTrigger | Mode$ Phase now implemented as a GENERAL
   phase-delayed-trigger construct (not a Mana-Drain hack):
   - core/delayed_trigger.rs: TriggerPhase += Main1/Main2 + from_script_name();
     DelayedTriggerCondition::Phase carries a SET of phases (Main1,Main2 = "your
     next main phase, either one"); matches_phase()/find_phase_triggers();
     DelayedTrigger += serialized remembered_amount; store += undo_add() that
     rolls next_id back (id stability across rewind/replay).
   - effect_converter.rs: Counter parses RememberCounteredCMC$ True ->
     CounterSpell.remember_mana_value; params_to_delayed_trigger_with_svars
     handles Mode$ Phase (Phase$ + ValidPlayer$).
   - loader/card.rs: follow_sub_ability_chain routes Charm/DelayedTrigger/
     ImmediateTrigger sub-abilities through the SVar-aware converters (the
     silent-drop root cause for `SubAbility$ DBDelTrig`).
   - game/state.rs: GameState += serialized remembered_amount; CounterSpell
     records countered spell's mana value (cmc + x_paid, CR 202.3); the chained
     CreateDelayedTrigger captures it onto the trigger; check_delayed_triggers_on_phase
     fires Phase triggers; fire_delayed_trigger handles ExecuteEffect{AddMana}.
   - game/game_loop/steps.rs: main_phase fires Main1/Main2 Phase triggers at
     phase begin, guarded by serialized per-(turn,which-main) flags.

DETERMINISM (the hard part — snapshot/resume + rewind/replay):
   The delayed-trigger lifecycle is now fully UNDO-LOGGED so rewind-to-turn-start
   (snapshot/resume + undo-search) reverses it exactly:
   - new GameAction::RegisterDelayedTrigger (undo = undo_add: remove + roll
     next_id back so a replay re-add reuses the same id),
   - GameAction::FireDelayedTrigger (undo = restore the removed trigger),
   - GameAction::SetRememberedAmount (undo = restore previous),
   - the firing's mana add is logged via the existing AddMana action.
   Without this, prop_snapshot_resume_fidelity diverged (the rewound turn-start
   state retained the trigger -> replay double-registered -> double {C}). Now the
   full and snapshot/resume runs produce byte-identical final state hashes.

Reproducer:

```sh
./target/release/mtg tui --start-state test_puzzles/mana_drain_deferred_mana.pzl \
  --p1=fixed --p2=fixed --p1-fixed-inputs='cast Hill Giant;*;*' \
  --p2-fixed-inputs='cast Mana Drain;*;*' --stop-on-choice=30 --seed 42 --verbosity 3
```

NEXT main phase (P2's Main1, not P1's Main2); fires once then one-shot removed.
