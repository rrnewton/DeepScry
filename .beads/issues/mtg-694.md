# Description

Parent: mtg-682. Live-playtest UX fixes on branch lobby-flow-fixes (base integration 60af178f). Web/JS + one small Rust export change (no engine/network/determinism impact).

FOUR items (verbatim from the user 2026-06-01 playtest):

(1) KILLED the residual waiting-room pane + bogus "sharable link" in index.html. Create now goes STRAIGHT lobby->launcher (launcher.html?game=&role=create&pass=&name=&ws=). Removed #pane-waiting DOM, all .wr-*/.invite-block/.waiting-details/.prefetch-bar CSS, and all waiting-room JS (showWaitingRoom/cancel/render/invite/handleGameCreated/handleGameStarted/handleWaitingRoomUpdate/handleJoinFailed/formatDeckSummary/formatJoinFailReason/setWaitingStatus/isWaiting + the game_created/waiting_room_update/game_started/join_failed handlers + PLACEHOLDER_DECK). "Create & Wait"->"Create Game".

(2) Created game STAYS LISTED. ROOT CAUSE: the lobby sent CreateGame on its BROWSE WS then dropped it on navigate-away -> server pruned the pending game. FIX: the lobby no longer creates on the browse WS at all; the GAME PAGE (native_game/tui_game) opens its own durable WS and sends CreateGame via consumeLobbyParams() -> that WS hosts and keeps the game LISTED for the life of the game tab. Game created exactly once, on the socket that hosts it. NO SERVER CHANGE needed: the host-pinned PendingGame model is correct + determinism-safe (the existing raw-WS pagination test proves a game stays listed while its host WS is open). A detached-host reconnect model was deliberately NOT introduced (risky for determinism, unnecessary once the double-create is gone).

(3) Join redirects ONLY the joiner: no shared waiting room -> each tab redirects only itself; the creator tab is never yanked.

(4) Split launcher "Deck Editor" -> "New Deck" (#btn-deck-editor, blank) + "Edit Deck" (#btn-edit-deck -> deck_editor.html?edit=<name>&source=<custom|premade>). Editing a PREMADE saves a COPY as a custom deck (deck_editor forces a unique "(copy)" name, never mutates the premade); custom decks edit in place. ALSO fixed a pre-existing data bug: deck_editor wrote .dck STRINGS under key mtg_custom_decks, but launcher/game pages/all tests read mtg-forge-custom-decks in {main_deck,sideboard} OBJECT form -> editor decks were invisible everywhere; deck_editor now writes the canonical object form under the shared key.
  Required a small Rust export change: mtg export-wasm only emitted built-in decks as a binary decks.<hash>.bin (WASM-only) + deck_names, with no WASM-free per-deck card lists. Added a deck_contents field to SetIndex (mtg-engine/src/main.rs): BTreeMap<deck_name, {main_deck:[[card,count]], sideboard:[[card,count]]}> so launcher Edit Deck + deck_editor load a premade WASM-free. Pure export/serialization; web/data/ is gitignored (regenerated by make wasm-export / deploy).
