# Description

PROBLEM: `load_deploy_secrets` in scripts/deploy-cloud.sh fails SOFT and SILENT when `.oauth.env` / `.r2.env` are not found. Its search covers only the checkout plus the immediate parent, so a deploy run from a WORKTREE misses `<parent>/.oauth.env`. This silently shipped a credential-less server: deepscry.net login vanished at deploy sha 21b504c90 (2026-06-14) and was recovered by an env-only restore.

FIX:
- Print a LOUD stderr WARNING that lists every directory searched when the secrets are absent, so a credential-less deploy cannot slip through unnoticed.
- Optionally make the secret search worktree-proof: walk up to the git superproject parent so `<parent>/.oauth.env` is found regardless of which worktree the deploy is invoked from.

POLICY: deploys MUST be run from the primary checkout (not a worktree).
