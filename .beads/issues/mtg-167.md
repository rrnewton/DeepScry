# Description

Improve ability parsing safety with tokenization and validation

## Problem

Card ability parsing used unsafe substring matching that could cause false positives:
- `ability.contains("DealDamage")` could match "PreventAllDamage"
- `ability.contains("add")` could match "Madden"
- No validation of parameter values or required parameters
- Silent failures when parsing fails
- O(nk) performance (re-scan string for each contains() call)

See ai_docs/ability_parsing_comparison.md for detailed analysis.

## Solution Implemented

**Option A + B (Hybrid)**: Tokenization + Validation

Created three new modules:

### 1. ability_parser.rs - Core Tokenization (496 lines)

- `AbilityParams::parse()`: Splits by `|` delimiter, extracts key-value pairs
- `ApiType` enum: 50+ recognized ability types from Java Forge
- `AbilityRecordType` enum: SP$, AB$, DB$, ST$ classification
- `AbilityParseError`: Clear error types for diagnostics
- Safety: Tokenizes ONCE, then queries (O(n + k) instead of O(nk))

### 2. effect_converter.rs - Parameter Validation (225 lines)

- `params_to_effect()`: Converts AbilityParams → Effect with validation
- Type-safe parameter extraction: `get_i32()`, `get_u8()`, `get()`
- Returns `Option<Effect>` (None for unsupported/invalid abilities)
- Handles 12 effect types: DealDamage, Draw, Destroy, GainLife, Pump, Tap, Untap, Mill, Counter, ChangeZone, Exile, SearchLibrary

### 3. card.rs - Refactored parse_effects()

- Replaced 165 lines of unsafe substring matching with 30 lines using tokenized parser
- Old logic preserved as `parse_effects_old()` for reference (will be removed later)
- Error handling: Logs parse errors, continues processing other abilities
- Silent skips for unsupported API types (intentional - avoid spam)

## Performance Impact

**Before:** O(nk) - scan entire string for each contains() call
**After:** O(n + k) - parse once, lookup many times

Expected improvement: 2-3x faster parsing (not measured yet, minimal impact on overall load time).

## Safety Improvements

1. **No false positives**: Tokenization prevents substring matches
2. **Validation**: Required parameters checked, errors reported
3. **Type safety**: ApiType enum instead of raw strings
4. **Error messages**: Clear diagnostics for malformed abilities
