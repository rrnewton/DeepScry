# Description

REDO of the debug-vs-release validate profiling experiment, done RIGOROUSLY on a quiet machine.

WHY REDO: the first run (mtg-763, experiments/validate_alldebug_20260605/) was CONFOUNDED by machine contention — the two back-to-back runs shared the 16-core box with other agents' validates, so binary-independent steps "sped up" ~2.5x between runs and the apparent total-wall-clock win was contention noise, not a real debug speedup. The KEEP-RELEASE decision still holds on principle (CI-parity + tests the shipped artifact), but the NUMBERS are unreliable and the user wants trustworthy ones.

RUN CONDITIONS (user 2026-06-05, STRICT):
- ONLY when the machine is TOTALLY QUIET: exactly ONE agent working, and while it runs its measured passes, NOTHING else major running. Basically AFTER all active development for the day is finished.
- Do NOT hold up any open development priority to run this. It is deferred / run-when-idle.

WHAT TO MEASURE (both modes: mtg-release-binary AND all-debug):
- Wall-clock AND CPU-time for ALL major steps, in BOTH modes.
- Measured BOTH fully-sequentially AND in a fully-parallel run. The delta between them = contention vs parallel scalability.
- For each mode, a CPU-utilization graph from T_start to T_end (the full validate wall-clock). Quantify: how much of the run is at significant parallelism vs a sequential bottleneck.
- Expected texture to confirm/refute: the release COMPILE phase is slow but well-parallelized (machine near-fully utilized during it), while the browser/network e2e tail is the low-utilization wall-clock drag.

DEPENDS ON the validate STATS/metrics redesign (separate wall-clock-sum vs cpu-time-global-sum + the CPU-utilization-over-time sampler + 2-level sequential flattening) — that infra is what makes these numbers measurable cleanly.
