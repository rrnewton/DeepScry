# Description

## Unify Mana Affordability Checking

## Problem

There are **3+ different implementations** of mana affordability checks with inconsistent semantics:

### Layer 1: ManaPool::can_pay() (core/mana.rs:190-206)
```rust
pub fn can_pay(&self, cost: &ManaCost) -> bool {
    // Checks if pool already has enough mana
}
```

### Layer 2: ManaCapacity::can_pay_simple() (game/mana_engine.rs:160-193)
```rust
pub fn can_pay_simple(&self, cost: &ManaCost) -> bool {
    // DUPLICATES ManaPool::can_pay() logic for ManaCapacity struct
}
```

### Layer 3: ManaEngine::can_pay_with_pool() (game/mana_engine.rs:889-946)
```rust
pub fn can_pay_with_pool(&self, cost: &ManaCost, pool: &ManaPool) -> bool {
    // Complex logic combining pool + untapped sources
}
```

### Inconsistent Call Sites

| Location | Method Used | Semantics |
|----------|-------------|-----------|
| game_loop/actions.rs:426 | `mana_pool.can_pay()` | Pool only |
| game_loop/actions.rs:433 | `mana_engine.can_pay_with_pool()` | Pool + sources |
| actions/mod.rs:4168 | `player.mana_pool.can_pay()` | Pool only |
| game_loop/legacy.rs:135 | `player.mana_pool.can_pay()` | Pool only |

## The Bug Risk

If code uses `mana_pool.can_pay()` when it should use `can_pay_with_pool()`, spells may appear uncastable even when the player has untapped lands that could produce the mana.

## Proposed Solution

1. **Single entry point**: `GameState::can_afford_cost(player_id, cost) -> bool`
   - Checks pool + untapped mana sources
   - All other methods become private implementation details

2. **Remove duplication**: `ManaCapacity::can_pay_simple()` should delegate to shared algorithm

3. **Audit all call sites**: Replace direct `mana_pool.can_pay()` calls with the unified method

4. **Clear documentation**: When to use pool-only vs pool+sources check

## Acceptance Criteria
- [ ] Single `can_afford_cost()` method for gameplay checks
- [ ] ManaCapacity::can_pay_simple() removed or delegates to shared code
- [ ] All call sites use consistent semantics
- [ ] Tests verify both "pool has mana" and "pool + sources have mana" scenarios

## Performance Requirements

**IMPORTANT**: Follow OPTIMIZATION.md guidelines when implementing this refactoring:
- No performance regressions allowed - run benchmarks before/after
- Mana checking is called many times per game loop - this is a hot path
- The unified method must not be slower than the fastest current implementation
- Avoid allocations (no temporary Vec for mana sources)
- Consider caching mana availability when game state hasn't changed
