# Description

**Phase 1.2 of mtg-151**: Decide strategy for SmallVec (189 occurrences), which lacks Allocator API support.

This is the **MAJOR BLOCKER** for full allocator coverage. SmallVec is heavily used
for inline optimization but doesn't support custom allocators.

**Parent Issue**: mtg-151 (Allocator API implementation)

## Problem

SmallVec does not implement `Allocator` trait:
- 189 occurrences across 19 files
- Critical for performance (inline storage avoids heap allocation)
- Used in hot paths (Card counters, abilities, zones)

## Three Options

### Option A: Keep SmallVec as-is (RECOMMENDED)

**Approach**: Accept that SmallVec overflow allocations use global allocator

**Pros**:
- Zero code changes required
- Maintains inline optimization performance
- Most SmallVecs stay inline (no heap allocation)
- Overflow is rare in practice

**Cons**:
- Overflow allocations use global allocator (some parallel contention)
- Not 100% allocator coverage (~10% of allocations remain global)

**Risk**: Low
**Effort**: None
**Expected Impact**: 80% allocator coverage, 60-70% parallel efficiency

**Decision Point**: Implement per-game allocators with SmallVec unchanged,
then measure parallel efficiency:
- If ≥60%: Accept SmallVec as-is ✅
- If <60%: Escalate to Option B or C

---

### Option B: Fork SmallVec with Allocator support

**Approach**: Create allocator-aware SmallVec fork

**Pros**:
- Complete allocator coverage (100%)
- Maintains inline optimization
- Could contribute upstream

**Cons**:
- Significant engineering effort (1-2 weeks)
- Maintenance burden (track upstream changes)
- May not be accepted upstream
- Complex implementation (inline storage + custom allocator)

**Risk**: Medium
**Effort**: High (1-2 weeks)
**Expected Impact**: 100% allocator coverage, 70-80% parallel efficiency

**Decision Point**: Only pursue if Option A proves insufficient (efficiency <60%)

---

### Option C: Replace with Vec<T, A>

**Approach**: Replace all SmallVec with Vec using allocator parameter

**Pros**:
- Full allocator support
- Simple refactoring
- Clean type signatures

**Cons**:
- Lose inline optimization (always heap-allocates)
- Likely performance regression in sequential case
- Defeats original purpose of SmallVec

**Risk**: High (performance regression)
**Effort**: Medium (189 replacements + testing)
**Expected Impact**: 100% coverage, but possible sequential slowdown

**Decision Point**: Avoid unless absolutely necessary. Benchmark required.

## SmallVec Usage Patterns

From allocation site analysis:

**Pattern 1: Card Counters** (Inline: 4)
```rust
pub counters: SmallMap<CounterType, u32, 4>,
```
- Most creatures: 0-2 counter types
- Heap overflow: Rare (<5% of cards)

**Pattern 2: Activated Abilities** (Inline: 2)
```rust
pub activated_abilities: SmallVec<[ActivatedAbility; 2]>,
```
- Most permanents: 0-1 abilities
- Heap overflow: Low (~10% of cards)

**Pattern 3: Subtypes** (Inline: 2)
```rust
pub subtypes: SmallVec<[Subtype; 2]>,
```
- Most creatures: 1-2 subtypes
- Heap overflow: Low (~15% of cards)

**Pattern 4: Temporary Buffers** (Inline: 8)
```rust
let mut targets = SmallVec::<[CardId; 8]>::new();
```
- Variable size
- Heap overflow: Medium (30-50%)

**Analysis**: Patterns 1-3 benefit strongly from SmallVec (inline dominant).
Pattern 4 could potentially use Vec<T, A> with per-turn allocator.
