# Description

CLUSTER: UI/infra — but MAY require engine/network investigation (escalate to gameplay cluster if it's a real desync).

Playing 01_rogue_rogerbrand vs 02_thedeck_peterschnidrig in the Web TUI (networked), the game STOPPED RESPONDING and the dev console showed:

[VERIFIER FIELD DIFF] 'cards' PRIOR: [...4/4 Black creature...] CURRENT: [...4/4 Black creature...]

The WASM client's state-verifier detected a divergence in the battlefield 'cards' field between its shadow state and the server's authoritative state, then halted (per "Desync is ALWAYS Fatal"). The diffed card is a 4/4 Black creature (likely Sengir Vampire or Sedge Troll).

Two possibilities:
1. REAL desync — the client's controller/state diverged from the server (information leakage or nondeterminism). This is a genuine engine/network bug and must be root-caused (gameplay cluster).
2. Verifier false-positive — the field serialization differs cosmetically (e.g. a field that's legitimately client-private or ordering-dependent) but the game state is actually equivalent. Then the verifier's field comparison needs tightening.

First step: capture the FULL prior-vs-current diff (the console truncates it). The divergence is somewhere after the shown prefix. Determine which sub-field differs. If it's a real desync, hand to the gameplay/engine cluster. Note: Sengir Vampire's DamagedBy trigger is already flagged as a parser gap (mtg-408 per the playtest skeleton) — could be related.
