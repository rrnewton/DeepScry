== B1 [FIXED 2026-06-11] Honden of Cleansing Fire ‚Äî '/Times.N' count multiplier dropped ==
Card: cardsfolder/h/honden_of_cleansing_fire.txt (4x in deck03 03_asahara_enduring_ideal).
Script: SVar:X:Count$Valid Shrine.YouCtrl/Times.2 ; TrigGainLife:DB$ GainLife | Defined$ You | LifeAmount$ X.
Symptom: the '/Times.2' multiplier suffix on a Count$Valid expression is an unrecognized filter token. Runtime: '[WARN count] Unknown filter type in count expression: Shrine.YouCtrl/Times.2' (3930x across tourney).
EMPIRICAL: puzzle debug/survey2005/honden_gainlife.pzl (P0 controls Honden of Cleansing Fire + Honden of Seeing Winds = 2 Shrines) -> upkeep logged 'Player 1 gains 6 life' (WRONG; correct = 2 life x 2 Shrines = 4). The multiplier handling is broken, not merely dropped ‚Äî output is non-zero but incorrect.
Root cause class: count-expression parser gap (the '/Times.N' post-filter multiplier). Search the Count$ filter parser (grep 'Times' / 'count expression' under mtg-engine/src). Generalizes to every Shrine cycle + any '/Times.N' card.
FIX: CountModifier::Times(N) enum variant added; parse splits on '/' to extract modifier; count_permanents_matching subtype fallback now handles enchantment-subtype strings like "Shrine". Warning eliminated. Life gain now correct: 3 Shrines x 2 = 6 life per trigger. Commit: 6ba5da1fa.

== B3 [FIXED] Umezawa's Jitte ‚Äî Charm modal reaches execute_effect fallback ==

Script: A:AB$ Charm | Cost$ SubCounter<1/CHARGE> | Choices$ JittePump,JitteCurse,JitteLife.
Symptom: '[WARN actions] ModalChoice effect reached execute_effect - should have been resolved during casting. 3 modes available.' (832x). The Charm modal is expected to be resolved at cast/activation time; reaching execute_effect means the mode selection took a fallback path.

FIX LANDED branch claude/compat-2005-wave4 (agent slot58):
Three-part fix:
1. effects.rs: added `needs_targeting: bool` field to `ModalMode` so per-mode ValidTgts$ targeting (JitteCurse) is distinguished from Defined$-based modes.

No longer falls through to execute_effect fallback.
3. abilities.rs (heuristic AI): classify_activated_ability now recognises ModalChoice effects containing PumpCreature modes and returns Pump{+2,+2}; should_activate_ability Pump arm has an equipment-source fast-path that fires in Main1/Main2/End when source is a non-creature.
Puzzle evidence (all three modes execute, no crash):

== B3 [FIXED 2026-06-12] Umezawa's Jitte ‚Äî Charm modal reaches execute_effect fallback ==
Card: cardsfolder/u/umezawas_jitte.txt (3x: decks 01, 04).
Script: A:AB$ Charm | Cost$ SubCounter<1/CHARGE> | Choices$ JittePump,JitteCurse,JitteLife.
Symptom: '[WARN actions] ModalChoice effect reached execute_effect - should have been resolved during casting. 3 modes available.' (832x). The Charm modal is expected to be resolved at cast/activation time; reaching execute_effect means the mode selection took a fallback path.
Root cause: Two-layer bug ‚Äî (1) loader/card.rs called params_to_effect_with_svars for ApiType::Charm, which fell through to params_to_effect and produced placeholder DrawCards{count:0} modes instead of resolving the named SVars (JittePump/JitteCurse/JitteLife); (2) game_loop/priority.rs had no ModalChoice arm in the activated-ability resolution loop, falling through to execute_effect (no-op).
FIX (commit 0d513fb77):
- loader/card.rs: added ApiType::Charm special case to use params_to_charm_effect_with_svars, which correctly resolves SVar mode names.
- game_loop/priority.rs: added ModalChoice arm calling choose_modes_with_hook, resolves each mode's placeholder variables, executes chosen sub-effect.
- heuristic_controller/{mod.rs,abilities.rs,creature_eval.rs}: added ActivatedAbilityType::Charm so AI classifies and triggers Jitte in Main1/Main2.
Evidence (2026-06-12_#3246): 'Random2 activates Umezawa's Jitte [remove charge counter: choose mode]' + 'Random2 gains 2 life' (JitteLife mode confirmed firing).

Symptom: '[WARN actions] Unimplemented effect SkipPhase resolved as no-op' (84x). CONFIRMED in debug/survey2005/v3_02.log: 'Trigger: Yosei ... skips their next untap step. Tap up to five ...' then 'SkipPhase resolved as no-op' then 'Yosei goes to graveyard'. The skip-untap LOCK half is silently dead.
WORKING: Flying keyword, 5/5 body, and the chained TrigTap (DB$ Tap) sub-ability are IMPL. So the tap-up-to-5 half works; only the untap-skip is missing.
Root cause class: missing SkipPhase/SkipStep effect (skip a named step for target player next turn). Affects Yosei + Kokusho-cycle siblings + any 'skips next untap/draw step' card. Sibling to 1994 Stasis skip-untap work (mtg-711) but that was a replacement on the controller; this is a targeted next-turn step-skip curse.
Repro: deck01/02 tourney OR the v3_02.log Yosei-dies snippet; a Wrath-of-God + Yosei puzzle reproduces deterministically.
STATUS: OPEN ‚Äî not yet fixed.

== B7 [FIXED 2026-06-12] Dredge keyword has no draw-replacement firing logic ==
Cards: cardsfolder/l/life_from_the_loam.txt (Dredge 3, deck 02), cardsfolder/n/nightmare_void.txt (Dredge 2, deck 02).
Symptom: Dredge IS parsed (mtg-engine/src/core/keyword_set.rs: KeywordArgs::Dredge { amount }; loader/card.rs:1125 'Dredge' arm) but there is NO logic anywhere in mtg-engine/src/game/ that REPLACES a draw with "mill N, return this card from graveyard to hand".
FIX (commit 0d513fb77):
- GameState::try_apply_dredge(player_id) in game/state.rs: finds first dredge card in graveyard with sufficient library cards, mills N, moves dredge card graveyard‚Üíhand, returns true.
- Hooked in draw_step() (game_loop/steps.rs): call before draw_card; if dredge fires, skip the normal draw.
- Hooked in execute_draw_cards() (game/actions/effects/cards.rs): each individual draw tries dredge replacement first.
Evidence (2026-06-12_#3246): 'Random1 dredges Nightmare Void (mills 2, returns to hand)' confirmed firing in seed 7.
Tests: test_dredge_replaces_draw, test_dredge_insufficient_library_no_replacement (keywords.rs).
