# Description

PRE-EXISTING network desync, deck-INDEPENDENT, reproduces in unmodified tui_game.html. Discovered while building the native network renderer (mtg-669).

REPRO (both fail with the IDENTICAL hash regardless of deck):
  make build-network && make wasm-network
  cd web && node test_network_gui_e2e.js --human                                  # grizzly_bears
  cd web && node test_network_gui_e2e.js --human --deck decks/old_school2/white_weenie_classic.dck

SYMPTOM (identical across decks):
  FATAL: P2 state hash mismatch! server=f7ec406da80a882e client=3b7dd10b9f66711d at choice_seq=1 action_count=45
  Server state: Turn 1 Upkeep, Hands [7,7]. Client (browser P2, Human controller) has already advanced to
  'WebHuman must discard 1 cards (hand size: 8, max: 7)' on its Turn 2 — i.e. the WASM Human-controller
  network client RUNS AHEAD of the server through P2's draw step into the cleanup discard, instead of
  blocking on the server's choice stream.

SCOPE:
  - Triggered ONLY with a Human controller on the WASM network client. The RANDOM controller path
    (test_network_gui_e2e.js with no --human) syncs fine — which is the ONLY mode wired into
    'make validate' (Makefile validate-network-e2e-step runs the random mode only). That is why this
    slipped through.
  - The identical server/client hash pair across two different decks proves it is the Human-controller
    run-ahead logic (likely tui_run_turn -> run_until_choice not yielding to the remote/server choice
    stream during P2's opening draw+cleanup), NOT deck data.
  - INDEPENDENT of the renderer: reproduces through the ratzilla TUI renderer (tui_game.html) and through
    the new native card renderer (native_game.html via launch_network_game_session) at the same point.
    The native renderer work (mtg-669) is correct; this is an orthogonal engine/controller bug.

LIKELY AREA: WASM Human controller in network mode + run_until_choice gating in mtg-engine/src/wasm
  (fancy_tui.rs tui_run_turn / WasmFancyTuiState::run_until_choice) and the network coordinator's choice
  sequencing for the non-active player's cleanup discard. The client should NOT advance P2's turn-2 draw
  locally before the server has sequenced it.

NEXT: add a Human-mode network sync leg to validate once fixed (currently only random mode is in validate).
See docs/NETWORK_ARCHITECTURE.md (desync is fatal; controllers must be information-independent).

--- 2026-06-02 netarch-dev4: ROOT CONFIRMED (structural) + unification = the fix ---
Reproduced on grizzly seed42 (test_network_human_input.js): FATAL P2 hash mismatch server!=client at choice_seq=1 action_count=46 (client raced to its turn-2 cleanup discard while server still at turn-1 upkeep).

NOT a controller-gating bug. WasmNetworkLocalController::choose_spell_ability_to_play (wasm/network/local_controller.rs:288-379) is the SAME wrapper for AI and human and gates every priority/auto-pass on check_choice_request_ready() — i.e. the inner human auto-pass (human_controller.rs:325 'Ok(None) when sorted.is_empty()') is GATED by the wrapper exactly like the AI's pass. So the wrapper is correct + identical for both paths.

THE STRUCTURAL ROOT: run_network_mode_human_v2 (fancy_tui.rs:2437) is the OLD pre-rewind structure, NOT the principled ai_v2 forward/replay machinery:
- Its 'normal run' branch (2586-2662) builds a FRESH inner each tick via human.clone() and calls run_until_input from the CURRENT game state.
- It has NO re-entry detection (net_forward_run_turn) and NO turn-1 baseline.
- It only rewinds when needs_replay==true (set when the USER clicks a choice).
=> On a JS re-entry that is NOT a user choice (the loop yielded NeedInput waiting for the server, then JS re-calls on the next server message), needs_replay is false, so it takes the 'normal run' branch AGAIN and re-runs forward from wherever the game already advanced — racing further ahead each re-entry instead of rewind+replay. By the FIRST submitted choice_seq the client is at action 46 (turn-2 discard) vs server turn-1 → fatal hash mismatch.

Contrast: run_network_mode_ai_v2 (2727) detects re-entry via net_forward_run_turn==current_turn → run_network_ai_replay (rewind to turn start, ReplayController-replay both players' recorded choices, delegate ONLY the new frontier choice to the persistent inner). That keeps it in lockstep — which is why AI mirrors (grizzly/monored/rogerbrand) sync at turn 1 and the human path does not.

THE FIX = deliverable (4) unification: route the Human controller through run_network_mode_ai_v2's forward/replay machinery, with the only difference being the frontier-choice source (human pending_choice injection vs AI live compute). Then DELETE run_network_mode_human_v2 + the dead net_turn1_baseline-only human clone.

DESIGN (avoids dyn downcast): add a PlayerController trait method e.g. fn set_pending_human_choice(&mut self, PendingChoice) with a default no-op; WasmNetworkLocalController forwards to inner_mut(); WasmHumanController sets its pending_choice. Then ensure_net_our_controller gets a Human arm building Box::new(WasmNetworkLocalController::new(WasmHumanController::new(id), client)); the dispatch (run_network_mode match arm, fancy_tui.rs:2383) injects p1_human_controller.pending_choice.take() into net_our_controller via the trait method, then calls run_network_mode_ai_v2 for ALL controller types. Re-entry detection (net_forward_run_turn) already works for both (a user click is a same-turn re-entry → replay path; the frontier choice is the injected pending choice). The forward run blocks correctly because the human inner returns NeedInput when it has options + no pending choice.

PROOF: test_network_human_input.js must go GREEN, then WIRE it into validate-network-e2e-step (Makefile ~348). This ALSO addresses mtg-4z4r9 (turn-2 own-drawn PlayLand) since that is the same human-path re-entry-without-rewind class manifesting on turn 2.

STATUS: analysis complete + test reproduces deterministically. Implementation is the next contiguous unit (sized ~1 trait method + impls, ensure_net_our_controller Human arm, dispatch change, delete ~225 lines, WASM rebuild + human-test iterations).

--- 2026-06-03 netarch-dev5: STALL DIAGNOSED = TEST BUG (engine is CORRECT) + oracle HARDENED ---

Root of the apparent stall = TWO test defects:
1. 1-BASED KEY MISMATCH: the WASM choice UI maps digit '''1'''->idx0, '''0'''->idx9 (10th).
