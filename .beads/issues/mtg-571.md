# Description

CLUSTER: UI/infra. Adopt content-addressed (hashed, immutable) web asset filenames so caching can be `immutable, max-age=1y` and stale-bundle bugs (mtg-475 / mtg-2indh: cached JS glue paired with fresh .wasm) become structurally impossible. index.html stays the sole mutable pointer. User decision: use trunk.

## STATUS: CORE landed on branch `trunk-cas-assets` (web-tooling; validation deferred, see below).

### Multi-page wrinkle — RESOLVED (and why trunk is staged, not wholesale)
Empirically verified with trunk 0.21.14: trunk's `rel="rust"` model OWNS the cargo+wasm-bindgen build and replaces a `<link data-trunk rel="rust">` with its OWN injected bootstrap, exposing the module only as `window.wasmBindings`. Our FOUR game pages (index is just the lobby; tui_game.html, native_game.html, demo.html, wasm_ai_harness.html) instead do `import init, { ...20+ named exports... } from './pkg/mtg_forge_rs.js'` plus dynamic `import('./pkg/...')`. Trunk's injected bootstrap CANNOT serve hand-authored named-export static imports of a hashed filename. Migrating all pages onto `window.wasmBindings.*` is a large, risky rewrite — deliberately deferred while a network-desync fix is in flight.

So trunk is adopted as the declared tool (Trunk.toml) with a documented migration path, and the content-addressing that actually kills the stale-bundle bug is delivered NOW via a structured post-build step that fits the existing wasm-pack + Rust-exporter + deploy flow.

### What IS content-addressed today (CORE, done)
1. **Per-set data bins** (largest asset, ~32MB / 315 bins): the exporter (`mtg export-wasm`, mtg-engine/src/main.rs) now writes `<YYYY>-<CODE>.<hash>.bin` and puts the hashed name into sets/index.json's `file` field (and the per-card `cards` map) + records a `hash` field. ZERO client change — the pages already fetch `./data/sets/${s.file}` straight from the manifest. Verified: 315 hashed bins emitted; curl of a manifest-listed hashed bin returns 200 with the manifest's byte count.
2. **wasm-bindgen pkg pair**: `scripts/hash_web_assets.sh` content-hashes mtg_forge_rs.js + mtg_forge_rs_bg.wasm to `<name>.<hash>.<ext>` on the DEPLOY-STAGING copy and does a STRUCTURED rewrite of the two controlled injection points in the HTML: the `./pkg/mtg_forge_rs.js` import specifier (static + dynamic) and the no-arg `init()` -> `init({ module_or_path: './pkg/<hashed>.wasm' })` (wasm-bindgen's documented override, so the generated glue is NOT edited). Verified on a staging copy: all 5 pages rewritten correctly; hashed JS+WASM serve 200, old fixed names 404.
3. **Cache tiers** (mtg-engine/src/web_server/mod.rs): `/data/**/*.bin` (hashed) -> `public, max-age=31536000, immutable`; `/images/**` immutable (already); mutable pointers index.json + decks.bin + tokens.bin + HTML + server-config.js -> no-cache / short max-age. NEW carve-out routes for decks.bin / tokens.bin (fixed-name, NOT hashed -> must stay no-cache per the stated INVARIANT: a route may be `immutable` ONLY if its URL is content-addressed). `/pkg/*` stays no-cache because the committed source HTML still imports the FIXED pkg name (hashing is applied only on the deploy-staging copy until the multi-page migration lands).
4. **deploy-cloud.sh**: replaces the old `?v=<sha>` query-string cache-bust with `hash_web_assets.sh` on the staging copy + a manifest-driven GC mark-sweep that drops orphaned hashed bins not referenced by the current index.json, then `rsync --delete` to prune the VM. Post-deploy probe updated to derive the hashed pkg names from the deployed tui_game.html and to fetch the first manifest-listed hashed bin. Edits kept localized + commented for clean rebase against the queued release-protocol task.
5. **Trunk.toml** + web/.gitignore dist/ entry: trunk declared as the build tool; settings documented; migration path noted.
