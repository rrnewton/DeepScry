# Description

CLUSTER: UI/infra. Adopt content-addressed (hashed, immutable) web asset filenames so caching can be immutable, max-age=1y and stale-bundle bugs (mtg-475 / mtg-2indh) become structurally impossible. index.html stays the sole mutable pointer.

## STATUS: matured on branch `trunk-cas-assets` (PR #10), rebased onto mtg-engine integration. Exporter non-determinism BLOCKER (recorded in prior rebase note) is now FIXED — see DETERMINISM FIX below. make validate GREEN.

### What IS content-addressed today
1. Per-set data bins: exporter writes <YYYY>-<CODE>.<hash>.bin; hashed name in index.json file/cards + a hash field.
2. wasm-bindgen pkg pair: hashed + HTML-rewritten on the DEPLOY-STAGING copy by `mtg hash-web-assets`.
3. Cache tiers (web_server/mod.rs): hashed /data/**/*.bin -> immutable 1y; content-aware /pkg; decks.bin/tokens.bin no-cache.
4. deploy-cloud.sh: content-address staging rewrite + GC mark-sweep + rsync --delete; PRE-rsync local smoke gate.

### DETERMINISM FIX (2026-05-28_#2422) — unblocks merge
ROOT CAUSE: `mtg export-wasm` was non-deterministic. Two consecutive runs on the identical cardsfolder produced DIFFERENT blake3 content hashes for the same set (1994-LEG -> d4a5a8394f52dc0e then 7c8883262f1db3c2). Two compounding sources of HashMap-iteration-order nondeterminism in the serialized bytes:
  (a) run_export_wasm (main.rs) collected each per-set bucket into a `HashMap<&str,&CardDefinition>` and bincode-serialized THAT (random pair order). Same for tokens.bin and decks.bin.
  (b) DEEPER: `CardDefinition.svars: HashMap<String,String>` is itself serialized, so even with sorted top-level bins each card's bytes varied run-to-run.

FIX (byte-deterministic, real fix):
  - main.rs run_export_wasm: per-set bins now collect into `BTreeMap<&str,&CardDefinition>` before serialize; tokens.bin + decks.bin likewise via sorted BTreeMap. bincode's map wire format is order-independent on READ, so the WASM `HashMap` deserializer is unchanged.
  - loader/card.rs: `CardDefinition.svars` gets `#[serde(serialize_with = "serialize_svars_sorted")]` — emits the map in sorted-key order without changing the runtime HashMap type (O(1) lookups + ~20 call sites unchanged). New helper serialize_svars_sorted borrows entries into a BTreeMap, no value clone.
  - Verified: CardCache / ManaProduction / ManaCost / KeywordSet / DeckList have NO serialized HashMap/HashSet; svars was the only one inside CardDefinition.
PROOF: two `mtg export-wasm` runs now produce byte-identical index.json, identical content-addressed filenames, and identical .bin sha256 across all sets (1994-LEG stable at d05dda69d20b77e8). New unit test loader::card::tests::test_svars_serialize_deterministically (inserts svars in two different orders, asserts identical bincode bytes + HashMap round-trip).

DEFENSE-IN-DEPTH: Makefile validate-network-e2e-step now order-depends on validate-wasm-e2e-step so the two browser-driven wasm steps never run their export-wasm + `rm -rf web/pkg` + `cp pkg web/pkg` clobbers of the shared web/data + web/pkg trees concurrently under -j4. Other parallel steps still run concurrently.
