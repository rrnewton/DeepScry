# Description

Structural refactor (no behavior change) to bring heuristic_controller.rs under the ~2000-line file guideline in CLAUDE.md.

## What
Split the 10110-line mtg-engine/src/game/heuristic_controller.rs into a module
directory mtg-engine/src/game/heuristic_controller/ organized by MTG-decision shape:
- mod.rs: HeuristicController struct, constructors, helper types (CombatFactors,
  CombatOutcome, ActivatedAbilityType), and the PlayerController trait impl
  (the decision entrypoints the engine calls, which dispatch into submodules).
- creature_eval.rs: creature/card valuation (evaluate_creature family).
- mana_lands.rs: mana counting, land selection, mana-source scoring.
- spell_selection.rs: master choose_best_spell dispatcher.
- combat.rs: attack evaluation, block assignment, combat math.
- spell_eval.rs: should_cast_* family, pump tricks, removal targeting.
- abilities.rs: activated-ability classification/scoring/activation timing.
- tests.rs: the unit-test module.

## Behavior preservation
Pure relocation: method bodies are byte-identical; only visibility was widened
to pub(crate) so sibling submodules can call across module boundaries, and the
three helper types were made pub(crate) for the same reason. Verified by diffing
seeded heuristic-vs-heuristic game logs (multiple seeds x decks) before and after.

PROBLEM (empirically verified on dev box, systemd 255 / cgroup v2 / user-delegated cpu+memory+pids):
The outer `systemd-run --user --scope` re-exec (mtg-743) CONTAINS descendants but did NOT reliably TEAR THEM DOWN:
1. Per-step teardown used killpg only; a setsid/double-fork grandchild (orphan mtg server / http.server / chromium) ESCAPES killpg (changes pgid, stays in the cgroup). This is the recurring zombie/port-collision/false-desync class.
2. On NORMAL exit, a setsid orphan left in the scope cgroup keeps the scope active running; `--collect` does NOT GC a scope with live procs, so the orphan survives the whole run. (Probe: payload exit 0 + setsid sleep -> orphan alive, scope active; only systemctl stop / cgroup.kill flushed it.)
3. Nothing stopped the scope on Ctrl-C / kill of the runner — only an after-the-fact kill_zombie_processes.py did.
Also verified: transient --user scopes are SIBLINGS under app.slice, so a naive per-step `systemd-run --scope` would NOT cascade from an outer-scope stop (stopping a parent scope does not kill a sibling child scope).

FIX (scripts/validate_cgroup.py + validate.py + kill_zombie_processes.py):
- Outer scope created with `-p Delegate=yes`. In-scope, the runner moves itself into a supervisor/ child cgroup (cgroup-v2 no-internal-processes rule), enables +cpu +memory +pids subtree_control, and gives EACH step its OWN child cgroup step-<group>.<job>/. The step's bash leader self-moves into it FIRST (fork-inheritance), so cgroup.kill reaps the whole subtree incl. setsid escapees. Verified: leader + setsid orphan both land in the step cgroup; cgroup.kill kills the orphan killpg would miss.
- SIGINT/SIGTERM handler stops the whole scope (systemctl stop + a cgroup.kill on the scope cgroup FIRST for instant SIGKILL — chromium ignores the SIGTERM systemctl sends, which otherwise stalls in stop-sigterm). Measured stop_scope latency 0.09s even with a SIGTERM-ignoring child.
- NORMAL exit: kill_all_remaining() cgroup.kill+rmdir any leftover step cgroup (does NOT stop the scope, so a green run's exit code is preserved).
- killpg retained as fallback when cgroups unavailable (locked-down host); --no-step-cgroups opts out.
- kill_zombie_processes.py SIGKILL-recovery path also cgroup.kills first (fast).

Also: memory-aware parallelism — --max-mem (abs or auto from MemAvailable) sets the scope MemoryMax/MemoryHigh AND caps -j to floor(budget/--mem-per-job) so concurrent cross-slot validates don't OOM-thrash.

KILL-TEST PROOF: SIGINT/SIGTERM on a real validate mid-browser-step leave zero slot04 orphans (no stray mtg/chromium/node, scope gone, no lock), scoped by /proc/<pid>/cwd so a concurrent slot03 validate is never touched. SIGKILL (uncatchable) recovered via kill_zombie_processes.py.
