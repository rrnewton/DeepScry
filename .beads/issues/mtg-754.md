# Description

SYSTEMIC TEST-EXECUTION GAP found while fixing mtg-747.

PROBLEM: the gui_view_model unit tests (mtg-engine/src/wasm/gui_view_model.rs — e.g. build_view_model_on_minimal_game, view_model_surfaces_session_error_message, the new aura_on_land_is_surfaced_as_attachment, the choice-dedup tests, etc.) live under `#[cfg(feature = "wasm")] pub mod wasm;`. They are COMPILED (lint-checked) by the clippy-wasm step (`cargo clippy -p mtg-engine --all-targets --features wasm,network -- -D warnings`) but NEVER EXECUTED: the only unit-test runner in both `make validate` (Makefile validate-test-step → `cargo nextest run --features network`) and CI (.github/workflows/ci.yml: `cargo nextest run -p mtg-engine --features network ...`) uses `--features network`, which does NOT enable `wasm` (wasm is only pulled by wasm-tui/wasm-tui-egui/wasm-network). So these runtime assertions never actually run — a test that should fail on a regression silently passes by not executing.

IMPACT: any view-model regression (wrong section grouping, missing attachment surfacing, dedup logic, error-message threading) is caught only at COMPILE time, not by assertions. The mtg-747 aura-on-land Rust test is currently in this state (covered at runtime instead by the JS e2e web/test_aura_render.js, which DOES run in validate-wasm-e2e-step).

PROPOSED FIX (tightly scoped, fast): add a validate + CI step that executes just these tests:
  cargo nextest run -p mtg-engine --features wasm,network -E 'test(/wasm::gui_view_model/)'
(or the broader `kind(lib) and not test(/network/)` wasm set if cheap). Compiles a native binary with the wasm feature (the module is target-arch-agnostic by design) and runs the ~10 tests fast. Wire into validate-parallel-steps + the CI matrix.

ROUTING: validate-structure domain — belongs with slot02's validate-infra work (mtg-717 / mtg-726). Deferred from mtg-747 to avoid colliding with the in-flight CI/Makefile rewrite. Flagged by slot04; team-lead to route.
