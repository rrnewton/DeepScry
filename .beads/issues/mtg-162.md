# Description

## Problem: Catastrophic Parallel Slowdown

The new `bench_game_par_rewind_play_again` benchmark (mtg-161) reveals **severe allocator contention** that will cripple future parallel MCTS implementation:

**Sequential baseline (1 thread):**
- 179,679 turns/sec
- 44,920 games/sec

**Parallel aggregate (16 threads):**
- 41,763 turns/sec (0.23x speedup - SLOWER than sequential!)
- 10,441 games/sec

**Root cause:** System allocator (glibc malloc) uses a global lock.

See mtg-166 for completed optimization work.

---

## Experimental Results: Subtype String→Arc<str> (2026-03-10_#1889)

**Experiment:** Convert Subtype from String to Arc<str> to reduce clone overhead.

**Results:** Allocation reduced by 4.7% but **single-threaded regression of 6-7%**.
Arc<str> atomic ops overhead exceeds benefits. **Rejected**.

---

## Experimental Results: Trigger Boolean Flags (2026-03-10_#1895)

**Experiment:** Replace `trigger.description.contains("[other]")` and
`trigger.description.contains("[landfall]")` string scanning with pre-computed
boolean fields `requires_other` and `requires_landfall`.

**Changes:**
- Set `requires_other = true` during trigger parsing instead of adding "[other]" to description
- Set `requires_landfall = true` during landfall trigger parsing
- Use O(1) boolean checks in `check_triggers()` instead of O(n) string scanning

**Results:**

| Metric | Before | After | p-value |
|--------|--------|-------|---------|
| simple_bolt time/game | 21.12µs | 21.75µs ± 0.3µs | p > 0.05 |
| robots_mirror time/game | 273µs | 274µs | p = 0.05 |

**Analysis:**

- No statistically significant improvement
- `check_triggers` closure was 6.19% of CPU but string scanning is not the bottleneck
- Most triggers don't have [other] or [landfall] markers
- The iterator/closure overhead dominates, not the string operation

**Conclusion:**

Boolean flags provide cleaner code but no measurable performance gain.
The change was **reverted** per project guidelines (only commit measurable improvements).

Future work: The trigger checking could benefit from:
1. Caching which cards have triggers of each type
2. Only iterating cards with relevant trigger types
3. Using a trigger registry instead of per-card iteration
