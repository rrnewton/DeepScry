# Description

make validate's `validate-network-e2e-step` (web/test_network_gui_e2e.js, run in CI per .github/workflows/ci.yml:225) is currently RED on the `integration` base commit, independent of any feature work. The SAME bug class also bites the bounded fuzz EQUIVALENCE sweep (see "fuzz-consolidation reproducer" below).

EVIDENCE (2026-05-28): Reproduced on clean integration commit 897881c9 (no local changes) in worktree fix-allhallows-eve after `make build-network wasm-network`:

  make validate-network-e2e-step
  --- Scenario 1/2: monored.dck seed=13 ---   PASS (24.2s)
  --- Scenario 2/2: old_school/01_rogue_rogerbrand.dck seed=3 ---  FAIL
    FATAL: P2 state hash mismatch! server=7bc17eb6... client=dc9e5bc4... at choice_seq=250 action_count=1352
  === MULTI-DECK TEST FAILED ===

The mismatch choice_seq/action_count VARIES run-to-run (observed 41, 149, 250 across runs) and which scenario fails varies (monored passed on one run, failed on others) — i.e. a NONDETERMINISTIC / timing-sensitive desync, not a fixed-seed deterministic one. This is the same bug class as mtg-1jtoy (reveal-ordering desync in random network games, mid-game P2 state hash mismatch) and mtg-b2tqp (shadow-state divergence). A concurrent agent's validate run happened to PASS both scenarios (got lucky on timing), confirming flakiness.

ADDITIONAL REPRODUCER under load (2026-05-30, fuzz-consolidation branch): the bounded native local-vs-network EQUIVALENCE sweep (`bug_finding/fuzz_determinism_netequiv.sh --invariant equivalence --controllers random`, 1 old-school pair x 2 seeds) — when it was briefly wired into validate via tests/fuzz_determinism_netequiv_e2e.sh Part 2 — DIVERGED on:

    random, 01_rogue_rogerbrand vs 02_thedeck_peterschnidrig, seed=1  (gamelog diff = 641 lines)

This combo PASSES in isolation on plain integration (`✓ ALL 1 COMBOS PASSED`) but FAILS under full concurrent `make validate` load. Same load-sensitive native local-vs-network desync, rogerbrand family. This is the direct reason the equivalence SWEEP is kept as a bug_finding expedition only and NOT in the validate gate — see DECISION below.

DECISION (fuzz-consolidation): the random x old-school-pair equivalence SWEEP is expedition-only (`bug_finding/fuzz_determinism_netequiv.sh --invariant equivalence ...`). It was REMOVED from the validate-wired bounded leg (tests/fuzz_determinism_netequiv_e2e.sh now runs DETERMINISM only — local-only, sub-second, reliably green). Validate retains deterministic equivalence coverage via tests/network_vs_local_equivalence_e2e.sh 3 random + 3 zero (single pinned seed, stable). Rationale: validate's randomized legs must be deterministically green; a sweep that is green only when the machine is quiet violates that policy. Documented in docs/FUZZ_AND_STRESS_TESTING_STRATEGY.md ("Why the network EQUIVALENCE sweep is expedition-only"). Related: mtg-589 (WASM-shadow rogerbrand family).

IMPACT: `make validate` cannot complete green on integration via the WASM network-e2e step until this is fixed (separate from the fuzz leg, which is now safe). Until fixed, validation proof for unrelated feature branches that hit validate-network-e2e-step must call out the flake explicitly.

NEXT STEPS: root-cause the reveal-ordering / shadow-state divergence per mtg-1jtoy; add a deterministic seed-pinned regression once the race is understood. The desync must be ELIMINATED (desync is always fatal), not papered over. Consider whether WASM browser timing under CPU contention contributes (failures cluster when CPU-contended).
