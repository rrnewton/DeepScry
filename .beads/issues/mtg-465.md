# Description

FIXED in branch fix-lobby-create-join (2026-05-27) + Phase 1 deck-picker and waiting room (branch lobby-usable-phase1, 2026-05-31).

**Phase 0 fix (fix-lobby-create-join, closed 2026-05-27):**
Root cause was: index.html createGame()/joinGame() built a query string and
window.location.href = 'native_game.html?...' instead of sending
ClientMessage::CreateGame / JoinGame over the open lobby WebSocket.

Fix (Option-B: redirect to tui_game.html): web/index.html now
 - Closes the browse-only lobby WS and redirects to tui_game.html with
   ?lobby_create=NAME or ?lobby_join=NAME params.
 - tui_game.html's `maybeAutoLaunchFromLobby()` detects these params and
   auto-fires CreateGame / JoinGame via a fresh WS + the WASM lobby exports.
 - On game_started, tui_game.html surfaces waiting UI.

**Phase 1 (lobby-usable-phase1, 2026-05-31):**
Closes the deck-picker gap (hardcoded placeholder deck) and improves the waiting room pane.

1. Real deck-picker (closes mtg-465 deck-picker follow-up):
   - `mtg export-wasm` now writes `deck_names: Vec<String>` into `data/sets/index.json`
     (field added to `SetIndex` struct in main.rs). Sorted list of available built-in
     deck names. Single-fetch, manifest-resolved, no binary parse required in the lobby.
   - web/index.html fetches `./data/sets/index.json` and populates a `<select id="create-deck">`
     dropdown from `idx.deck_names`. Graceful fallback: if the field is missing (old index.json)
     or the fetch 404s, the select shows "(No built-in decks available)" and the Create button
     is still enabled (tui_game picks a deck).
   - Selected deck name passed as `?deck=<name>` in redirectToGamePage → tui_game.html's
     `maybeAutoLaunchFromLobby` pre-selects it in the p1-deck dropdown.
   - Joiner also gets their own deck choice from the same select.

2. Real waiting room:
   - After clicking "Create & Wait", instead of immediately redirecting, the lobby
     shows `#pane-waiting` with a detail block (game name, passcode, chosen deck, status)
     and a shareable invite snippet (copy button).
   - "Launch Game" button does the actual redirect to tui_game.html with all params.
   - "Cancel" returns to the lobby form.
   - This gives the creator a moment to copy the game name+passcode for their opponent
     before the game page starts waiting.

3. Test updated (web/test_landing_page_ux.js):
   - scenarioFullFlow: waits for btn-create to be enabled (async deck load), clicks Create,
     waits for #pane-waiting, verifies wr-game-name/wr-passcode fields, clicks btn-launch-game,
     then verifies redirect to tui_game.html with lobby_create + lobby_pass params.
   - Remaining test checks (bob, charlie uniqueness, refresh timer, mobile, offline) unchanged.
