## Summary

Network desync from reveal-ordering on draw/dig: the displayed card identity on a network shadow depends on async reveal-arrival timing, so the SAME log line renders the opaque `card#<id>` fallback on the shadow's first forward pass (reveal not yet arrived) and the revealed NAME on a rewind+replay (reveal landed). The rewind/replay verifier then flags a fatal LogMismatch.

## LIVE deepscry.net FATAL (2026-06-09, random-vs-random native WASM network game)

```
REWIND/REPLAY FATAL: log entry at buffer index 41 (prefix offset 8 of 19, replay tail length 25) diverged.
  expected: "playrrr2 puts card#106 into Hand"
  actual:   "playrrr2 puts Monument to Endurance into Hand"
```

## Root cause (CONFIRMED, fixed)

The `Effect::Dig` family logs `"<player> [reveals and ]puts <card> into <Zone>"` via plain `gamelog` (no stable verifier key). `Effect::Dig { destination: Hand }` moves Library->Hand, which `move_card` reveals to OWNER ONLY (state.rs:1311 `maybe_reveal_to_player`), so the opponent's shadow never gets a synchronous reveal; its forward pass renders `card#106` while a later async reveal materialises the instance that a rewind+replay then names. This is the SAME async-reveal-timing asymmetry already solved for discards via `verifier_stable` (actions/mod.rs:7025, state.rs:1633).

Four unstable dig log sites (all `card#<id>` fallback + plain `gamelog`):
- mtg-engine/src/game/actions/mod.rs:5975 (selected -> destination)  <- LIVE BUG
- mtg-engine/src/game/actions/mod.rs:6040 (rest -> hand/graveyard/exile)
- mtg-engine/src/game/game_loop/priority.rs:3462 (selected -> destination, priority-routed)  <- LIVE BUG
- mtg-engine/src/game/game_loop/priority.rs:3520 (rest -> hand/graveyard/exile, priority-routed)

The other 3 `card#<id>` log sites already carry `verifier_stable` (the 3 discard paths) and were not the bug.

## Fix (branch claude/netarch-desync-sweep)

All four dig sites now call `logger.gamelog_reveal_stable(display, stable)` where `stable = "<player> [reveals and ]puts card#<id> into <Zone>"`. The DISPLAY text is unchanged (UI still shows the real name when known); only the verifier comparison key is steered to the reveal-timing-INDEPENDENT id form. No string-matching hacks — uses the existing structured `verifier_stable` mechanism. Desync remains fatal for a GENUINELY different card (different id -> stable keys diverge -> LogMismatch).

## FOLLOW-UP (separate concern, NOT a desync)

`Effect::Dig { destination: Hand }` logs the dug card NAME publicly via `gamelog` even though hand is private — a latent information-leak (an opponent reading the full log learns what was dug into hand). Drawing uses `gamelog_private` for exactly this reason. Consider masking dig-into-hand to a private form. Filed as a sibling concern; does NOT affect the desync fix (the stable verifier key is correct regardless). Tracked here pending decision.

## SECOND fix under this issue (e0c33784): opening-hand ac-0 reveal drop

WASM-network fuzz surfaced a DISTINCT reveal-binding desync in the SAME family: the shadow's reveal-apply cursor dropped the reveal stamped at game action_count == 0 (the first opening-hand draw / top of library). `last_applied_reveal_ac` starts at 0 and the apply filter is `*ac > last_applied_reveal_ac`, so `0 > 0` is false → the ac-0 reveal was NEVER applied → the first-drawn card stayed `RevealCard(N = ??? ...)` on the shadow. If that card is a land, the shadow omits its PlayLand from the legal-action set → "Local abilities (4) != server abilities (5)" FATAL at the first choice. Present in BOTH the wasm and native clients (identical filter); both fixed via an `any_reveal_applied` flag that makes ac 0 eligible on the first apply only (and re-armed on cursor reset so rewind+replay re-applies it). Regression tests `reveal_at_ac_zero_is_pending_until_applied` and `reset_state_sync_cursor_re_arms_ac_zero_reveal` added to the wasm client. After this fix games run past the opening hand; the next (separate) desync is mtg-gfr2a (Turn1 EndCombat state-hash mismatch at ac 66).

Both fixes (dig verifier_stable d1d95349 + ac-0 reveal apply e0c33784) are pushed on claude/netarch-desync-sweep.
