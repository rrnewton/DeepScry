# Description

## Symptom

Enabling the WASM rewind/replay verifier (`tui_set_verify_rewind_replay(true)`) and playing a Human-vs-Heuristic game with the rogue_rogerbrand deck eventually trips:

```
REWIND/REPLAY FATAL: turn-start state hash for turn N changed across rewinds
(expected 0x..., got 0x...). The undo log is no longer a faithful inverse of forward play.
```

After the controller_choice filter fix in commit `<this commit>`, the field-diff diagnostic in `run_replay_verification` reveals the underlying cause: **hand card ORDER differs across rewinds**.

Example field diff (timestamp: 2026-05-13_#2211):

```
PRIOR  hand: [27, 32, 61, 18, 28, 33, 26]   (61 at position 2)
CURRENT hand: [27, 32, 18, 28, 33, 26, 61]  (61 at position 6)
```

Same multiset, different order. `player_zones.hand.cards` is a Vec; its order is part of the serialized state and therefore part of `compute_state_hash`.

## Likely cause

Some action that moves a card OUT of hand and back IN (e.g. a draw-and-replace effect, a 'reveal then put back' action, or undo of a discard) restores the card to the END of the hand instead of its original position. Suspects:

- `MoveCard` undo path
- `Bazaar of Baghdad` activated ability (draw 2, discard 3) — sequence is rewind-heavy and exercises hand mutations
- Any DSL effect using `Sub$ ChangeZone` with `Origin$ Hand Destination$ Hand`

The undo log entry for a hand mutation needs to record the SOURCE INDEX so undo can re-insert the card at the correct position rather than appending.

## Repro

1. `make wasm-dev`
2. Open `web/tui_game.html`, check "Debug Mode (TRACE logs)", launch a Human-vs-Heuristic game with a rogue_rogerbrand deck
3. Press a mix of '0' and '1' / '2' to advance through a few turns
4. After ~20–50 input events, the verifier fires the FATAL message above

The verifier's per-field diagnostic dump (added in the same commit) prints the differing hand order, narrowing the hunt.

## Scope

This is a pre-existing bug — exists on integration before commit 8502a55f (per-card draw logging). The user originally reported it together with a separate `<Choice>` log-mismatch bug; only the latter was caused by 8502a55f exposing controller_choice / ReplayController asymmetry. The hash-drift is independent.

## Next steps

- Track down which undo action(s) restore cards to the wrong hand position
- Add a hand-order undo invariant test similar to `rewind_to_turn_start_produces_stable_replay_hash` in `undo.rs`
- Once fixed, remove the field-diff diagnostic comment in `fancy_tui.rs::run_replay_verification` (or keep it — it's cheap and only fires on failure)
