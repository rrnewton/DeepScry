# Description

DONE 2026-05-28_#2366. Resolves mtg-572.

Files: mtg-engine/src/wasm/gui_view_model.rs, web/native_game.html

Goal was to wire the shared layout engine into native_game.html via the view
model. Investigation found the wiring ALREADY EXISTS:
- native_game.html consumes the Rust GUI view model (gui_view_model.rs,
  exported as tui_get_gui_view_model_json) for battlefield section grouping
  + order (build_battlefield_sections) — the same categorize_card /
  FancyTuiRenderer helpers the TUI uses.
- It also already calls the shared Rust battlefield_layout engine
  (battlefield_layout::pick_card_size_for_battlefield) for per-card sizing.

So there was NO duplicated JS layout logic to remove; gui_view_model.rs needed
no change. The remaining defect (mtg-572) was a pure CSS flow bug in
native_game.html: the section container .card-grid used flex-direction:column,
stacking each section on its own full-width row.

CHANGE (web/native_game.html): .card-grid -> wrapping flex row
(flex-wrap:wrap); .bf-section -> content-sized flex item. Sections now pack
horizontally and wrap, matching tui_game.html's density, with section
order/grouping + card sizing still sourced from the single Rust source of
truth.

Regression test web/test_battlefield_layout.js added to validate-wasm-e2e-step.
This unblocks mtg-574 (launcher/ready-up page) and the move to make the native
GUI the default. Cross-ref mtg-568 (same-name stacking) is a SEPARATE
enhancement (stack identical permanents into piles) — NOT addressed here,
though the denser packing reduces the symptom.
