# Description

Follow-up to mtg-751. The wasm view-model serialize (tui_get_gui_view_model_json + JSON.parse) is ~2.27ms/tick for an 11-card board — half the per-tick cost, paid on EVERY tick (the v1 render-skip doesn't touch it; the O(1) serialize-skip follow-up only skips it on UNCHANGED ticks). Slim the per-tick model so it's cheaper on ALL ticks (incl. the ~23% that render): drop oracle_text + full CardDetailView from the per-card battlefield/hand payload (only serialize the FULL detail for the single selected card, which is all the details pane needs); send logs incrementally (only new lines since last tick) instead of the full tail every time. gui_view_model.rs (build_card_view / build_view_model). slot01-safe (no reveal-apply coupling). Measure before/after with debug/measure_render_cost.js. NOTE: gui_view_model tests are #[cfg(feature=wasm)] → see mtg-754 (they're compiled but not executed in validate).

PRIORITY (team-lead 2026-06-04): do this BEFORE the v2 revision-skip (mtg-755). It's bulletproof (always serialize the visible essentials; heavy bits on demand) and helps CHANGED ticks too (v2 only helps skipped ticks). If slim gets per-tick serialize from ~2.27ms to ~0.5ms, v1+slim captures most of v2's win without the skip-correctness risk.
