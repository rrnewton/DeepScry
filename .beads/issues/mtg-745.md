# Description

mtg-725 R2, characterized natively during mtg-728 class-A work. Effect::ChangeZoneAll (actions/mod.rs ~4341 None=>{}) with a RESTRICTED restriction (types/controller/color/etc., i.e. NOT TargetRestriction::is_unrestricted) and a HIDDEN-zone origin (Hand/Library) cannot be replicated on a viewer's shadow: the opponent's cards there are reserved (instance-less) ids, so restriction.matches() is unevaluable — the shadow cannot know which reserved cards match the filter, so it cannot reproduce the server's move COUNT/identity.

CHARACTERIZATION (throwaway native test, not committed): shadow with 5 reserved opponent hand ids + a restricted (Creatures-only) Hand->Graveyard ChangeZoneAll -> shadow moves 0 (all 5 skipped at None=>{}).

This is the HIDDEN-INFO DESIGN CLASS, NOT the sig-2c count-parity class:
- sig-2c (move_reserved_in_shadow = is_shadow_game && restriction.is_unrestricted()) correctly handles UNRESTRICTED mass moves (Timetwister/Wheel) by moving reserved ids symmetrically.
- A RESTRICTED move from a hidden zone is fundamentally different: it branches on hidden info the shadow can't see. DO NOT paper-over with a count-parity hack (would force a wrong count / assert a non-lockstep state).

REACHABILITY: likely NONE — no known supported card does a restricted/typed mass zone-change from an opponent's HIDDEN hand/library (you can't deterministically mass-move opponent hidden cards by type). If a real card EVER hits this, the correct fix is a reveal-buffer: the server reveals which cards the restricted hidden-zone move affected, and the shadow applies that reveal (the mtg-610 reveal-history-buffer vehicle), NOT a count hack. Until then this is a latent code property, not a live desync. Related: mtg-728, mtg-725.
