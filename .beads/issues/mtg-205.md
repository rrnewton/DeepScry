# Description

## Description

## Bug Summary

During the Declare Attackers phase, the game correctly shows 'Available creatures' that can attack, but the 'Player1 available actions' list does not include any 'attack' options. This makes it impossible to declare attackers in agentplay mode.

## Root Cause

The issue was that 'attack X' commands were being consumed during the priority round (choose_spell_ability_to_play) that happens BEFORE the declare attackers step. The command would error because it didn't match any spell/ability.

The turn sequence is:
1. BeginCombat step → priority_round() ← 'attack zhao' consumed here and errors
2. DeclareAttackers step → choose_attackers() ← never reached

## Fix

Updated RichInputController::choose_spell_ability_to_play() to recognize 'attack' and 'block' commands as combat commands that should be deferred. When these commands are detected during priority, the controller now passes priority (returns None) WITHOUT consuming the command, allowing it to be processed later by choose_attackers() or choose_blockers().

## Changes

```rust
// In choose_spell_ability_to_play():
let is_combat_command = cmd_lower.starts_with("attack ") || cmd_lower.starts_with("block ");
// ...
} else if is_combat_command {
    // Combat command during priority - don't consume, pass priority
    ChoiceResult::Ok(None)
}
```
