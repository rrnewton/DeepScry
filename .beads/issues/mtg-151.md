# Description

**MAJOR PROJECT**: Implement custom allocator support using Rust's Allocator API to enable per-game and per-turn arena allocators for parallel MCTS optimization.

This epic tracks the complete migration from global allocator to custom allocator architecture, enabling:
1. **Per-game allocators**: Each parallel simulation gets its own bump arena (zero contention)
2. **Per-turn allocators**: Temporary data allocated per-turn, bulk-freed at turn boundaries
3. **Runtime selection**: Flag to choose between system allocator and bumpalo arenas

**Parent Issue**: mtg-2 (Optimization and performance tracking)

## Architecture Design

### Two-Level Allocator Strategy

1. **Per-Game Allocator**:
   - Each `GameState` owns a bump allocator
   - All game-lifetime allocations use this arena
   - Bulk-freed when game completes
   - Zero contention in parallel simulations

2. **Per-Turn Allocator**:
   - Nested arena for turn-scoped temporaries
   - Reset at turn boundaries
   - Eliminates repeated alloc/dealloc overhead
   - Examples: temporary ability lists, combat calculations

### Runtime Selection

```rust
// Command-line or environment variable:
// --allocator=system|bumpalo
// --per-turn-allocator=system|bumpalo

pub enum AllocatorMode {
    System,     // Global allocator (glibc malloc)
    Bumpalo,    // Bump arena (lock-free)
}

impl GameState {
    pub fn new_with_allocator(
        game_alloc: AllocatorMode,
        turn_alloc: AllocatorMode,
    ) -> Self { ... }
}
```

**Use cases:**
- Debugging: Use system allocator for better valgrind/asan support
- Benchmarking: Compare performance between allocators
- Development: Verify correctness with both allocators
- Production: Use bumpalo for maximum performance

## Known Challenges & Mitigations

### Challenge 1: SmallVec Lacks Allocator API

**Problem**: SmallVec is heavily used (189 occurrences) but doesn't support Allocator trait

**Options:**
1. **Keep SmallVec as-is** (pragmatic)
   - Accept that overflow allocations use global allocator
   - Most SmallVecs stay inline (no allocation)
   - Trade-off: Some contention remains, but minimal

2. **Fork SmallVec** (ambitious)
   - Create allocator-aware version
   - Significant engineering effort
   - Benefit: Complete allocator coverage

3. **Replace with Vec<T, A>** (simple)
   - Lose inline optimization
   - May regress performance
   - Would need benchmarks to validate

**Recommendation**: Start with Option 1, measure impact, revisit if needed

### Challenge 2: Lifetime Complexity

**Problem**: Allocators have lifetimes, making type signatures complex

**Example:**
```rust
// Before:
fn process_game(game: &mut GameState) { }

// After:
fn process_game<'a, A: Allocator>(game: &mut GameState<'a, A>) { }
```

**Mitigation:**
- Use default `A = Global` to maintain simple signatures
- Document lifetime rules clearly
- Provide helper functions to hide complexity

### Challenge 3: Clone vs clone_in()

**Problem**: Standard `Clone` trait can't change allocator

**Solution:**
- Keep `Clone` for same-allocator cloning
- Add `clone_in<B: Allocator>(&self, alloc: B) -> GameState<B>` for cross-allocator
- Document when to use each
