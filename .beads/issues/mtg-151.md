# Description

**MAJOR PROJECT**: Implement custom allocator support using Rust's Allocator API to enable per-game and per-turn arena allocators for parallel MCTS optimization.

This epic tracks the complete migration from global allocator to custom allocator architecture, enabling:
1. **Per-game allocators**: Each parallel simulation gets its own bump arena (zero contention)
2. **Per-turn allocators**: Temporary data allocated per-turn, bulk-freed at turn boundaries
3. **Runtime selection**: Flag to choose between system allocator and bumpalo arenas

**Parent Issue**: mtg-2 (Optimization and performance tracking)

🔍 **CRITICAL FINDING: Allocation Pattern Analysis (2025-11-06)**

DHAT profiling of 20-turn game revealed unexpected results:
- **Total allocations**: 208.49 KB in 681 blocks
- **UndoLog: 103.59 KB (49.7%)** ← Dominant hotspot!
- **CardZone::add: 992 bytes (0.5%)** ← Minimal during gameplay

**Reality check**: Our assumption that CardZone/PlayerZones are the dominant allocation sites during gameplay is WRONG.

**However**: Profiling was NOT representative of MCTS clone workload because:
- Test game had no mana (all Lightning Bolts)
- No spell casting occurred
- Minimal zone movement (only draw/discard)
- UndoLog is per-game transient (not cloned in MCTS)

**Critical unknown**: Does GameState.clone() for MCTS trigger CardZone allocations with Global allocator even after parameterization?

**Current recommendation**: Option A - finish what we started, then validate empirically.

## Performance Goals

**Current state (from mtg-a6ca26):**
- Parallel efficiency: 5.6% on 32 cores (should be 70-80%)
- Per-thread: 17.8x slower than sequential
- Root cause: 40% allocator contention + 60% memory system contention

**After per-game allocators (predicted):**
- Parallel efficiency: 50-60%
- Aggregate speedup: 16-19.2x on 32 cores
- Eliminates allocator lock contention entirely

**Combined with other optimizations (target):**
- Parallel efficiency: 70-80%
- Aggregate speedup: 22.4-25.6x
