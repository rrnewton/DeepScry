# Description

INCIDENT 2026-05-28 ~12:03-12:51 UTC: deepscry.net:8080 went down for ~48min. Root cause: the deepscry.service runs as a systemd --user service but the service user (newton) did NOT have lingering enabled (loginctl show-user -> Linger=no). systemd user managers (and their services) are torn down when the user's last login session ends, so the server died whenever the admin/deploy SSH session closed, and only restarted when a new SSH session re-created the user manager. Journal confirms: shutdown signal received at 12:03:22 UTC (session end), no restart until 12:51:16 UTC (next SSH = status check). MITIGATED live by running loginctl enable-linger newton (now Linger=yes).

FIXED (branch infra-deploy-hardening): scripts/deploy-cloud.sh config phase, systemd-user branch, now:
- hard-fails if loginctl is missing (durable lingering is mandatory in user mode),
- runs loginctl enable-linger "$USER" UNCONDITIONALLY (idempotent) right after moving the unit into place, instead of the old conditional grep-guarded best-effort attempt,
- escalates a missing loginctl/enable-linger failure from a soft WARNING to a hard error (exit 1) so a non-durable bootstrap can never silently succeed,
- ASSERTS loginctl show-user "$USER" reports Linger=yes afterward, exiting 1 with an actionable message otherwise.
No hardcoded user/host: uses the existing $USER inside the remote bootstrap (which is $REMOTE_USER on the VM). Orchestrator will re-run deploy-cloud.sh config post-merge to apply linger durably + redeploy. Bash-only change; not exercised by make validate (which stays hermetic per CLAUDE.md), but full make validate run green to prove the tree.
