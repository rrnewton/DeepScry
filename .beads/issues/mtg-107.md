# Description

## Task: Modify all scripts under tests/ to use cargo run

## Investigation Results (2025-11-03)

### Bash Test Scripts (tests/*.sh)
All bash test scripts are **ALREADY using cargo run**:
- ✅ controlled_draw_e2e.sh: Uses `cargo run --bin mtg` (line 59)
- ✅ heuristic_grizzly_bears_attack_e2e.sh: Uses `cargo run --bin mtg` (line 69)
- ✅ interactive_tui_e2e.sh: Uses `cargo run --bin mtg` (line 73)
- ✅ puzzle_load_e2e.sh: Uses `cargo run --bin mtg` (lines 52, 114)
- ✅ heuristic_royal_assassin_e2e.sh: Placeholder script, doesn't invoke binary

### Python Stress Test Scripts (scripts/*.py)
These scripts use pre-built binaries from `target/release/mtg` or `target/debug/mtg`:
- scripts/snapshot_stress_test_single.py
- scripts/test_snapshot_determinism.py

**Why this is acceptable:**
- The stress test system is designed for performance: `run_stress_tests.sh` calls `cargo build --release` once (line 98), then runs the Python scripts multiple times in parallel
- Using the pre-built binary is MUCH faster than `cargo run` for repeated invocations
- The wrapper script (`run_stress_tests.sh`) ensures the binary is built before running tests

**Python script behavior:**
- `find_mtg_binary()` looks for existing binary in target/debug or target/release
- Does NOT call `cargo build` itself
- Assumes binary is up to date (which it is when called from run_stress_tests.sh)
- Errors with helpful message if binary not found

## Conclusion

✅ **All test scripts under tests/ already use cargo run as intended.**

The Python helper scripts use pre-built binaries for performance, which is the correct design for stress tests that run the binary hundreds of times.
