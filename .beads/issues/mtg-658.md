# Description

Follow-up from mtg-650 (Fireball variable target count, Phase A). The engine, AI (Random/Heuristic), Network, and FixedScript controllers fully support variable target count (min..max) for DivideEvenly X-spells like Fireball. The HUMAN-facing multi-select UX is best-effort/incomplete:

- InteractiveController (stdin TUI): has an 'add target / done' loop honoring min/max (implemented).
- FancyTuiController (native fancy TUI): still single-target prompt — does NOT offer multi-select for variable-count spells.
- WASM human_controller / wasm fancy_tui / wasm rich_input: the pending-choice path already maps a VECTOR of indices (so a chosen multi-target set round-trips), but the JS UI for letting a human PICK how many targets + which ones for a divide spell is not built — the front-end currently surfaces single-select.

Scope: add a 'choose 0..N targets' multi-select affordance to the fancy native TUI and the web/WASM human UIs, driven by the (min_targets, max_targets) now passed to Controller::choose_targets. NOT a determinism/correctness blocker — AI-vs-AI, network, and equivalence harnesses do not exercise the human UX. See controller.rs choose_targets signature and the best-effort notes in fancy_tui_controller.rs / wasm/human_controller.rs / wasm/rich_input_controller.rs.
