# Description

GOAL: make `make validate` FAST (target 1-2 min, lighting up ALL cores) and its output log clean + low-noise. TWO deliverables: (1) an improved validate; (2) a SKILL FILE enshrining the principles for keeping validate clean+efficient AND how to repeat these audits when validate rots again (which audits to run, which principles to steer toward). CI validate must stay IDENTICAL to local but SHARDED across top-level "jobGroup" identifiers (decompose the top level of the validate work-tree into separate GitHub CI steps for parallelism).

Case-study log: `<parent>/scratch/validate_8f7d3e38a5b9b0ba6293cdc17f1639c48b95b39d.log` (16,325 lines — far too noisy). Reported by user (rrnewton). This is the slot02 workstream.

== A: EFFICIENCY / PARALLELISM ==
Symptom (htop/pstree mid-validate): only ONE chrome-headless, ~22% of ONE CPU; machine massively underutilized. Want all cores lit, minimum wall-clock.
- FIRST inventory existing profiling support for `make validate` (timing/instrumentation already present?).
- Agent reads the validate code (Makefile validate-* targets, scripts/validate.sh, web e2e runners) and BUILDS A DIAGRAM of the current parallelism structure (sequential vs parallel; fork/join points).
- Identify the LONG CRITICAL PATH (playwright browser testing suspected).
- Design FOR parallelism: break up longest sequential phases, start them as EARLY as possible, run other work in background, shrink them. Minimize the unavoidable start/end fork/join under-utilization.

== B: BUILD DE-DUPLICATION (zero duplicate build work) ==
- "Export complete!" appears 4x in the log -> build steps mis-wired (duplicate export-wasm / db export).
- "Compiling mtg-engine v0.1.0 ..." appears MID-log during the wasm testing portion -> the mtg binary must compile ONCE up front (critical path); everything reuses it; test binaries compile AFTER.
- Review debug-vs-release: which steps need release vs debug binary? Build can overlap test in some ways. Compile mtg ONCE first.

== C: LOG / OUTPUT HYGIENE ==
Principle: a step's DETAILED output -> a /tmp file; the validate log shows only high-level start/end/PASS + duration per step. On FAILURE, dump the failing step's stack-trace/detailed output INTO the log (self-contained failure description).
Leaks to fix:
- agentplay tests dump raw GAME LOGS to stdout (wrong) -> only "test running / ran / PASSED + duration".
- playwright tests vomit per-request server logs + Browser [log] lines + raw game logs (e.g. Grizzly Bears) into the log.
- native-vs-WASM equivalence sweep dumps per-deck [PASS] detail (15 decks) -> high-level only.
- network-vs-local equivalence E2E dumps "Analyzing Results / Turns / Winners / GAMELOG comparison" -> high-level only.
Tagging: prefix EVERY log line with `[jobGroup.jobId]` so you can grep one part of the work-tree and ignore interleaving, while still seeing real-time completion order. Interleaving is FINE (e.g. a nextest PASS landing mid-other-output); tags make it navigable. Parallelism-smell: tag [X] is the last-running long tail but starts later than it could.
3-LEVEL naming: (jobGroup, jobId, testName) = globally unique. jobGroup = top-level CI shard; jobId = a distinct test workflow (seq or parallel) within a group; testName = e.g. nextest "mtg-engine game::graveyard_display::tests::empty_graveyard". 3 levels is plenty.
Explore a FRAMEWORK for managing a tree of processes dumping parallel output to one stdout with per-line [jobGroup.jobId] tagging.
VERBOSE mode: `make validate` verbose variant SAVES every step's log to a directory, files named to match the [jobGroup.jobId] tags. Default stays terse.

== D: TEST CONSOLIDATION / ORTHOGONALITY ==
- Multiple equivalence-test clusters appear (native-vs-WASM equivalence sweep; network-vs-local equivalence E2E "deterministic gamelog identity" mtg-380; whole_game_rewind_replay; etc.). Check orthogonality + consolidation (any redundant?).
- Audit slow network/wasm/browser tests: any sequential that needn't be? Trim overall footprint in the validate timeline.

== CI SHARDING ==
CI validate IDENTICAL to local but sharded across top-level jobGroup IDs -> separate GitHub CI steps per jobGroup for parallelism.

== DELIVERABLES ==
1. Faster, cleaner `make validate` (terse log + verbose-dir mode + [jobGroup.jobId] tagging + zero duplicate builds + maximal parallelism).
2. CI workflow sharded by jobGroup (identical work, parallel steps).
3. A SKILL FILE (e.g. `.claude/skills/validate-hygiene/SKILL.md`) enshrining the principles (terse-log / detail-to-/tmp / dump-on-fail; [jobGroup.jobId] 3-level naming; build-once; design-for-parallelism; start-the-long-tail-early) AND the RE-AUDIT procedure (diagram parallelism, find critical path, detect log leaks, detect duplicate builds) so validate stays clean over time.
