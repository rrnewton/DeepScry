# Description

Redesign validate.py STATS so the parallelism numbers are honest and disambiguate the long poles.

PROBLEM (user 2026-06-05): the current "serial-sum" = sum of per-STEP wall-clock durations, but each step is INTERNALLY parallel (lint.clippy uses all cores; run_examples.sh runs ~18 examples via GNU parallel; the network group runs concurrently). So "serial-sum" is a weird hybrid — a sum of internally-parallel wall-clocks — and "parallel speedup" (serial-sum / wall) conflates inter-step overlap with intra-step parallelism. It also makes lint.clippy and examples.run LOOK like the long poles by serial-sum, which clashes with the intuition that the browser/network e2e is the bottleneck. Both are true in different metrics; the single number can't show it.

REQUESTED REDESIGN:
1. Report SEPARATELY:
   (a) wall-clock sum-of-steps (the current hybrid — keep it, but label it honestly), AND
   (b) cpu-time global-sum = the true total CPU work (sum of user+sys CPU across ALL processes in the run; the honest "if fully sequentialized to one core" number).
   Show wall-clock vs cpu-sum so the REAL parallelism (cpu-sum / wall) is visible, distinct from the step-overlap number.
2. CPU-utilization graph from T_start to T_end of the full run. The background sampler already exists (validate.py:_start_utilization ~:935) — extend it to emit a time-series + a simple sparkline/graph and quantify: how much wall-clock is spent at significant parallelism vs at a sequential (single-core) bottleneck.
3. Two-level sequential flattening, CONFIGURABLE:
   (a) between-steps sequential (run steps one at a time, each still internally parallel), and
   (b) within-steps sequential (also force each step to a single core).
   So one can measure pure-serial CPU-time vs the parallel wall-clock at both granularities.

WHY: resolves the "is clippy/examples or the browser the long pole?" confusion — clippy+examples dominate CPU-TIME (compute-heavy, well-parallelized, high utilization), while the browser/network e2e dominates the LOW-UTILIZATION wall-clock (I/O-bound, serialized on the capacity-1 browser resource). The redesign makes each visible. This is also the metrics prerequisite for the rigorous debug-vs-release profiling experiment [linked].
