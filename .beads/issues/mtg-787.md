# Description

From audit Q3/§B1. The choice payload (choice_indices + spell_ability + library_search_result + target_card_ids) is re-declared in 5 places: BufferedFact::Choice (protocol.rs:439-447), ChoiceEntry (choice_entry.rs:43-83), ServerMessage::OpponentChoice (protocol.rs:816-862, superset), CachedOpponentChoice (remote_controller.rs:24-30, a field-renamed clone: 'indices' vs 'choice_indices'), and the client->server subset SubmitChoice/ChoiceResponse (protocol.rs:319-356, controller.rs:106-121). Define ONE strong-typed struct ChoicePayload { choice_indices: Vec<usize>, spell_ability: Option<SpellAbility>, library_search_result: Option<CardId>, target_card_ids: Option<Vec<CardId>> }. Reuse inside BufferedFact::Choice (envelope fields choice_seq/choice_type/description stay on the variant), ChoiceEntry, OpponentChoice (until removed), and DELETE CachedOpponentChoice entirely. Add shared decode helpers (apply_indices_to_slice + CardId-vs-index resolver). SubmitChoice/ChoiceResponse: reuse the subset OR define a deliberately tighter SubmittedChoice (justified tighter-fit — document it). RISK: touches ChoiceEntry + controllers (slot03 area). SEQUENCE AFTER deep-AC.
