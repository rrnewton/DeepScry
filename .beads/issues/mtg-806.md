# Description

User-flagged 2026-06-05. The mtg server ALREADY supports --port 0 (binds OS-ephemeral + reports the real port via 'MTG Server listening on {bound}', server.rs:615-626). But the e2e harnesses do NOT use it: web/test_network_utils.js allocates RANDOM available ports via isPortAvailable() (bind+release CHECK) then passes the number to the server (--port N) — a check-then-use TOCTOU race; another process or an outgoing ephemeral connection can grab the port between the check and the server's bind. THIS is the source of the concurrent-validate port collisions we currently serialize validates around (no netns). web_server/mod.rs:285 ('bind ephemeral and immediately dropping') has the same bind-drop-return TOCTOU. FIX: switch ALL network test harnesses (test_network_gui_e2e.js, test_network_*.js, play-web.sh, the reproducers/canary) to pass --port 0 and PARSE the server's reported real port from stdout/log, keeping the OS-atomic ephemeral bind (no check-then-use gap); fix the web_server free-port helper to bind :0 and KEEP the listener. BENEFIT: eliminates the port-collision class entirely -> enables PARALLEL network e2e (no serialization, no netns) -> directly attacks the validate long-pole. Effort S-M, high leverage. Related: mtg-726 (network-test perf), mtg-743 (cgroup isolation), mtg-773 (orphan port contention), mtg-767 (shared-resource race).
