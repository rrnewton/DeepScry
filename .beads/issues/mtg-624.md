# Description

Bug: Drain Life — StoreSVar effect unimplemented, deals damage to None — FIXED

## Status: FIXED (2026-06-09, claude/compat-1994-damage)

Drain Life (X1B Sorcery: "deals X damage to any target. You gain life equal to
the damage dealt, but not more than the target's life/loyalty/toughness before
the damage was dealt") now works end-to-end.

## What was broken (re-confirmed at HEAD before the fix)
- 3x "Unimplemented effect 'StoreSVar' resolved as no-op" warnings spamming the
  gamelog (the head ability + 2 chained StoreSVar SVars).
- The chained `DBGainLife` (LifeAmount$ DrainedLifeCard) was DROPPED entirely
  from the effects vector — DynamicAmount::parse didn't recognize the
  `SVar$Y/LimitMax.Limit` body, so gain_life_dynamic_from_params returned None
  and no life-gain effect was produced (no life gained at all).
- (The "damage to None" symptom from the original report was already partly
  improved by SubAbility target threading; with a real target chosen the X
  damage now lands on the target.)

## The fix (model the OUTCOME, not the generic SVar machinery)
Rather than implement a generic runtime SVar store + the per-target
ConditionDefined/Present/Compare selector (CardToughness vs Loyalty vs
LifeTotal), the engine reads the life-gain cap DIRECTLY from the target:
1. core/effects.rs: new DynamicAmount::DamageDealtCappedByTarget { cap } =
   "min(damage dealt this resolution, target's pre-damage life/loyalty/
   toughness)". DynamicAmount::parse recognizes the `SVar$ <var>/LimitMax.…`
   shape (Drain Life's DrainedLifeCard) tokenized (a `/LimitMax.` segment).
2. core/effects.rs: new Effect::NoOp { api_type } — a recognized, SILENT no-op
   (distinct from Unimplemented which warns). effect_converter routes
   `DB$ StoreSVar` to NoOp, since its only job (computing the Limit cap) is
   modeled by the snapshot. No more StoreSVar warning spam.
3. game/actions/mod.rs: snapshot_target_amounts now also captures the drain cap
   (creature toughness / planeswalker loyalty / player life — handling player
   sentinels) into a typed TargetSnapshot, BEFORE any effect runs (CR 608.2g/2h
   last-known information — a player's life has already dropped by the time the
   chained GainLife runs). resolve_dynamic_gainlife_snapshot locks the cap;
   resolve_dynamic_amount computes min(damage_dealt_by_source, cap). The plain
   DynamicAmount::DamageDealt now also reads damage_dealt_by_source (was a 0
   stub).

## Relationship to Java Forge
Forge-Java runs the literal StoreSVar pseudo-API into a per-card SVar map, then
DrainedLifeCard = Y/LimitMax.Limit reads it. This Rust version does NOT maintain
a runtime SVar store; it reads the cap directly from the target snapshot
(DamageDealtCappedByTarget) — same observable result (gain = min(damage, target
characteristic before damage)), less machinery. StoreSVar is a documented NoOp.
