--- DEPLOY-ONLY NAV 404 FIX (branch fix-deckeditor-launcher-nav, 2026-06-01) ---
USER-REPORTED LIVE BUG: on the hashed deploy, launcher -> Edit Deck -> deck editor -> 'Back to Launcher' 404'd on https://deepscry.net/launcher.html?... because launcher.html is content-hashed (served only as launcher.<hash>.html).

ROOT CAUSE (confirmed empirically via mtg hash-web-assets on a staged tree): deck_editor<->launcher form a dependency CYCLE (launcher->deck_editor forward + deck_editor->launcher back), so the CAS renamer (asset_hash.rs) leaves launcher.html as a LOGICAL name in the runtime manifest (asset-manifest.json maps launcher.html -> launcher.<hash>.html) for RUNTIME resolution. The launcher-parity pass built deck_editor's back-link with a LITERAL 'launcher.html' (JS launcherReturnUrl builder + a static <a href='launcher.html'> placeholder), bypassing resolveAsset()/installManifestHrefRewrite() -> dangling literal -> 404.

FIX (web/deck_editor.html, web-only, NATIVE==WASM untouched):
 - Static #back-to-launcher anchor: added data-asset-href='launcher.html'; a deferred <script type=module> imports installManifestHrefRewrite() from asset_manifest.js and runs it on load (matches the game pages' tui<->native cycle-edge pattern).
 - JS-built back-link: launcherReturnUrl() now returns resolveAssetName('launcher.html')+query; deck_editor's main script is a CLASSIC <script> (cannot import) so the module bridges resolveAsset to window.__mtgResolveAsset (identity fallback on dev/empty manifest) and re-runs setupBackToLauncher() once the resolver is live.
 - index.html links left as literals (stable unhashed entry -- exempt).

AUDIT (all web pages, literal hashed-page links): the ONLY remaining defect was deck_editor->launcher. launcher->deck_editor (forward DAG edge) is statically rewritten by the renamer (verified hashed in staged tree); native_game<->tui_game already use data-asset-href+installManifestHrefRewrite; lobby_launcher.js GAME_PAGE uses resolveAsset; demo->game-pages statically rewritten; index.html fully statically rewritten; game_boot_params.js is test-only (not deploy nav).

SYSTEMIC GATE (web/test_deploy_tree_nav.js): extended to (c2) fetch the served deck_editor and walk its REVERSE/back-edges (deck_editor->launcher round trip, deck_editor->lobby/index) through the manifest; plus a new assertPageNavResolvable() invariant -- any served page emitting a LOGICAL (manifest-key) cycle-edge nav target MUST carry the runtime resolver (asset_manifest.js import + installManifestHrefRewrite()/resolveAsset(), logical anchors must be data-asset-href tagged). Applied to deck_editor AND the game pages. PROVEN: FAILS on the buggy literal deck_editor (2 failures: missing resolver + bare anchor lacking data-asset-href), PASSES after. An HTTP-only fetch-through-manifest walk alone MASKED the bug (the served literal href is itself a manifest key); the source-invariant check catches the browser-runtime dangling literal.

## The architecture (LOCKED — 4 pages, clear separation)
1. **Lobby** (`index.html`, served at `/`): register a unique name → browse live
   games → **Create** or **Join**. NO renderer choice here. NO deck picker here.
   Creating/joining a game says NOTHING about how you personally render or which
   deck — those are launcher concerns. After Create/Join → go to the LAUNCHER.
2. **Launcher** (`launcher.html`, NEW): the per-player pre-game screen. Here you
   pick: **your deck** (with deck COLLECTIONS, not a flat dropdown) + a **"Deck
   Editor" launch** button + **your renderer: Native GUI (DEFAULT) or Web TUI**
   (alternate). Renderer is a per-player experience detail chosen HERE, never in
   the lobby. Then "Play" → go to the matching game page. This is the ONE launcher.
   EXTRACTION SOURCE: the BETTER built-in launcher to lift from is **tui_game.html**
   (lines ~750-893): it has Deck COLLECTIONS (3-pane layout, `#p1-collection`/
   `#p2-collection`, deck-buttons incl. the deck-editor link) — the richer UX the
   user remembers. native_game.html's built-in launcher is WEAKER (no deck-editor
   link, fewer collection refs). So `launcher.html` should be extracted from
   tui_game.html's launcher, generalized to be renderer-agnostic (renderer =
   native default / TUI alternate), and BOTH game pages then lose their built-in
   launcher entirely.
3. **Native game** (`native_game.html`): PURE renderer (card DOM). NO built-in
   launcher (delete the `#launcher` block). Receives everything via params.
4. **Web TUI game** (`tui_game.html`): PURE renderer (ratzilla). NO built-in launcher.
   - **Native is the default.** A creator and joiner each pick their own renderer
     in their own launcher; the two can differ (one native, one TUI) — same game.
5. **Deck editor** (`deck_editor.html`): reachable from the launcher's Deck Editor
   button (already exists; wire it in).

Flow: `index(lobby)` → create/join → `launcher.html?game=…&role=create|join&…`
(register/create/join already happened on the lobby WS; launcher picks deck+renderer
+ ready) → `native_game.html` OR `tui_game.html` with the full param contract.

### Space-advance in network mode?
**Not needed.** In network mode with a non-human controller, the WASM sets `auto_run=true` (fancy_tui.rs:1563) so the game advances automatically without any key presses. ### Architecture finding
The harness mirrors test_network_gui_e2e.js but opens TWO browser clients instead of one browser + one native AI. Both use WASM random controllers. Both advance without freeze/desync.

### Verdict: AI-over-network IS a viable e2e driver
- No freeze, no desync, advances ≥3 turns reliably.
- The Space-advance mechanism works but is unnecessary for AI games.
- mtg-679 (human controller desync) is NOT a blocker for the AI-driven test path.
- The redo UI can be built incrementally, with each stage gated by this harness.
