# Description

CLUSTER: UI/infra. Related: mtg-474 (play-from-lobby), mtg-569 (black screen), mtg-572 (native GUI layout bug), mtg-465/mtg-466 (lobby create/join WS wiring).

STRATEGIC DIRECTION (per user): the native web GUI (native_game.html) is the PRIMARY surface; the Web TUI (tui_game.html) is secondary/power-user. BUT the UI surface is a PER-PLAYER choice, not a lobby-wide or creator-only setting.

AMENDED DESIGN (user, 2026-05-28) — pre-game launcher / ready-up page:

Today the lobby (index.html) launches multiplayer straight into the Web TUI only. Replace that with a dedicated pre-game LAUNCHER PAGE (like character/kart-select in many games), used by BOTH entry paths:
- The lobby 'Create & Wait' button → launcher page (as host, waiting for an opponent).
- The lobby 'Join' button → launcher page (as joiner).

The launcher page requirements:
1. Shows BOTH players' status: name + Ready / Not-Ready indicator, updating live as each connects/toggles.
2. EACH player independently chooses THEIR OWN UI surface (native web GUI vs Web TUI) — the joiner picks their own; the host picks their own. This supersedes the earlier 'creator toggles / lobby-wide default' framing. Default selection = native web GUI; either player may switch to Web TUI before readying.
3. Each player marks themselves READY. The game starts only when BOTH are Ready.
4. The launcher is the home for any future PER-PLAYER preference settings (e.g. auto-pass / auto-yield behavior, animation speed, numeric-choice mode, etc.) — design it so adding such toggles later is easy.
5. When the game starts, each player is redirected into THEIR OWN chosen surface (native_game.html or tui_game.html) with the appropriate lobby/game/pass/name/ws params. The two players can be on DIFFERENT surfaces simultaneously (one native, one TUI) — the server is surface-agnostic; both are just clients of the same game.

WIRING: per-player UI choice + ready state must round-trip through the lobby WS protocol (or a small pre-game coordination channel) so each client sees the other's status and the server only launches when both ready. Keep controllers information-independent / network-deterministic regardless of which client surface each player uses (per docs/NETWORK_ARCHITECTURE.md).

SEQUENCING: native GUI has open quality bugs (esp. mtg-572 broken battlefield layout) that should land before/with making native the default selection on the launcher — otherwise we default players into a buggy surface. Sequence: fix native GUI layout (mtg-572) -> build launcher/ready-up page (this issue).

OUT OF SCOPE (future): matchmaking, >2 players, spectators.

ACCEPTANCE: from the lobby, both Create and Join land on the launcher; each player independently picks native-GUI-or-TUI (default native) and toggles Ready; both-ready starts the game; each player loads their chosen surface; mismatched surfaces (one native, one TUI) play correctly against each other.
