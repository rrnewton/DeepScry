# Description

## Context

Phase 1 of the puzzle assertion DSL (mtg-935) implemented final-state assertions ([assertions] section in .pzl files). This issue catalogs the Rust test assertions in puzzle_e2e.rs that CANNOT be migrated to the inline DSL yet because they depend on structured game log events (trigger fired, activation logged, etc.).

## What was migrated (Phase 1 migration, claude/puzzle-migration branch)

8 puzzle files received inline [assertions] sections covering final-state checks:
- serra_angel_should_attack.pzl — game won, opponent life lt 20
- flying_vs_ground.pzl — game won, opponent life lt 8
- vigilance_blocks_back.pzl — game won, NOT game lost
- lethal_through_blockers.pzl — game won, turn le 5
- must_attack_creature.pzl — game won, opponent life lt 20
- large_creature_attack.pzl — game won, opponent life lt 12
- crusade_buff_e2e.pzl — game won, opponent life lt 3
- forestwalk_blocks_forest_owner.pzl — game won, opponent life eq 0
- life_race_decision.pzl — game ended
- lifelink_race_evaluation.pzl — game ended

The corresponding Rust test assertions are kept as belt-and-suspenders (not removed).

## Deferred: EVENT/LOG-derived assertions (require structured GameEvent stream)

These Rust tests in puzzle_e2e.rs have assertions that check game log text (e.message.contains(...)) — these VIOLATE the NO HACKY STRING OPERATIONS rule if moved to the DSL via substring matching. The correct fix requires a structured GameEvent enum (Phase 2 of the DSL, tracked in PUZZLE_ASSERTION_DSL.md).

Count: 17 deferred assertions across 14 test functions.

## 2026-06-13 — Event log WIRED INTO BULK RUNNER (claude/puzzle-event-wiring)

Phase-2 event assertions now work end-to-end in the BULK runner (make puzzle-bulk-check), not just the dedicated puzzle_assert_e2e.rs path. Three root causes fixed:

A. mtg-engine/tests/puzzle_bulk_runner.rs: enable_event_log() is now called on the game's logger BEFORE play (mirrors puzzle_assert_e2e.rs:49). ALSO fixed a latent bug: the runner read final_game.logger.events() from a CLONE of the game, but GameLogger::clone() deliberately starts the clone with an EMPTY event_log so every event was discarded. Now borrows the live game_loop.game instead of cloning.

B. Combat lifelink (Keyword::Lifelink) in mtg-engine/src/game/actions/combat.rs (~line 1139) now pushes a LifeChanged LogEvent for the gained life, so 'life gained' assertions fire for combat lifelink. (Life-gain EFFECTS already emitted LifeChanged via actions/effects/life.rs.)

C. Death-WATCHER triggers in mtg-engine/src/game/actions/mod.rs check_death_triggers() now push a TriggerFired LogEvent naming the SOURCE (watcher) for CreatureDies (Fecundity/Blood Artist), EquippedCreatureDies (Skullclamp) and DamagedCreatureDies (Sengir Vampire) loops. Previously only the dying creature's OWN dies-trigger emitted TriggerFired.

Showcase test_puzzles/fecundity_nighthawk_event_showcase.pzl now PASSES all three event assertions (creature died Wind Drake / trigger fired from Fecundity / life gained ge 2) in the bulk path. spirit_link_aura.pzl also passes.

DETERMINISM: observability-only. state_hash.rs strips 'logger' from ALL hash modes (state/rewind/undo/network); golden-log oracle mismatch=0 confirms the STRING log is byte-identical. push_event is a no-op when event_log_enabled=false (MCTS/fuzz/network paths).

REMAINING (gameplay gaps, NOT observability; tracked here under mtg-935): spirit_link_blocked_creature_damage.pzl and spirit_link_noncombat_pinger.pzl still fail 'life gained' because Spirit Link's 'deals damage -> you gain that much life' trigger does NOT fire for combat-damage-to-a-creature or non-combat damage (only damage to a player works, per spirit_link_aura). Real engine behavior gap, deliberately NOT touched by this observability task.

Bulk baseline MAX_ASSERT_FAIL tightened 10 -> 5 (4 puzzles still assert-fail: 2 alt-cost + 2 spirit-link gameplay gaps).
