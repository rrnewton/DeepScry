# Description

## Summary

The fuzz tester (`bug_finding/network_test_lib.py`) currently only supports one WASM client target: `web/wasm_ai_harness.html`. To meet the QA goal of testing all three UI modes (native TUI, tui_game.html TUI, native_game.html native GUI) end-to-end, the harness needs extension.

## What works today

- `web/wasm_ai_harness.html` is a minimal AI-only page that:
  - Reads `server`, `controller`, `name`, `seed` from URL parameters
  - Calls `wasm.run_network_ai_step()` in a loop
  - Sets `window.gameResult = { winner, choices, error }` on completion
  - Logs `[WASM_DONE]` to console.

The Python harness opens this page via Playwright headless Chromium, polls `window.gameResult`, and treats console messages as the client log.

## What does NOT work today

`web/tui_game.html` and `web/native_game.html` are interactive UI shells:

- They require the user to pick a controller from a `<select>`, then click 'Start' or similar
- They do not parse URL parameters to bypass setup (or do so only partially)
- They do not expose the `window.gameResult` / `window.getGameResult()` API
- They depend on the rendering subsystem (RatZilla / DOM) which the harness page deliberately avoids

A test trying to load `native_game.html` directly will hang waiting for user interaction and time out.

## What would need to change

1. **Add URL-parameter auto-start** to tui_game.html and native_game.html so that visiting `?server=X&controller=Y&seed=Z&autostart=1` skips the setup screen and starts the network game immediately.
2. **Expose the test completion API** (`window.gameResult`, `window.getGameResult()`) on those pages so Playwright can detect when the game ends.
3. **Add a 'page' field to TestConfig** in `network_test_lib.py` (e.g. `harness | fancy | game`) and use it to construct the URL in `run_wasm_client`.
4. **Render-loop visibility**: when tui_game.html or native_game.html is the harness, console output is not the same as the AI loop's gamelog. We need either DOM-text scraping or a dedicated logging hook the page can populate.

## Why it matters

Bugs that affect the rendering layer (mtg-188: idle render frequency; mtg-191: pass-only loop in browser TUI; mtg-fancyfix work) cannot be reproduced by the fuzz tester today. native_game.html-specific desyncs (mtg-198 tracking) similarly cannot be exercised.
