# Description

## Store SVars as Structured Rust Types

## Problem

Currently SVars are stored as raw strings:

```rust
/// Script variables (SVars) for SubAbility chaining
/// Key: SVar name (e.g., "BalanceHands")
/// Value: SVar body (e.g., "DB$ Balance | Zone$ Hand | SubAbility$ BalanceCreatures")
pub svars: std::collections::HashMap<String, String>,
```

This is:
- Inefficient (re-parsing on every access)
- Error-prone (deferred parsing can hide errors until runtime)
- Inconsistent (we parse abilities eagerly, but not SVars)

## Solution

Parse SVars into structured Rust types at card loading time:

```rust
/// Parsed SVar - structured representation
pub enum ParsedSVar {
    /// DB$ SubAbility - parsed to AbilityParams
    SubAbility(AbilityParams),
    /// Numeric value (e.g., "NonCombatPriority:1")
    NumericValue(i32),
    /// String value (e.g., "AILogic:BlackLotus")
    StringValue(String),
    /// Unparsed (fallback for unknown formats)
    Raw(String),
}

pub svars: std::collections::HashMap<String, ParsedSVar>,
```

## Benefits

1. **Eager error detection** - Parse errors caught at load time
2. **Performance** - No re-parsing on access
3. **Type safety** - Strong types for different SVar uses
4. **Consistency** - Same approach as ability parsing
