# Description

Two issues, both visible to user from the landing-page lobby (web/index.html):

## Bug 1: 'Already authenticated / in a game' error after game starts

ROOT CAUSE: The lobby's 5-second refreshTimer keeps calling requestGameList() (sending ClientMessage::ListGames) over the lobby's WebSocket even AFTER pairing succeeds. Once the lobby's WS gets paired via the server's run_create_flow / run_join_flow, it is taken over by handle_player_websocket() (server.rs ~line 2300+). In that handler, the match arm for {Authenticate, CreateGame, JoinGame, ListGames} returns ServerMessage::Error { message: 'Already authenticated / in a game', fatal: false }. The lobby's handleServer treats Error as a failPendingFlow and turns the waiting-pane status red.

User-visible symptom: 'turns green for a second and THEN turns red' -- the green is the GameStarted ack, the red is the next 5-second refresh tick hitting the post-pairing connection.

FIX: stop the refreshTimer when entering the waiting/paired state (pendingFlow != null); restart on cancelPendingFlow.

## Bug 2: 'Playing the game from this lobby page is not yet wired up'

The lobby pairs successfully but the post-pairing pane just shows the stub message (extends mtg-465). User wants to actually play the game from the browser after pairing.

DESIGN: Option B (redirect to tui_game.html) -- chosen over Option A (inline play).

Rationale:
- Option A (inline play) would require loading the multi-MB WASM bundle into the lobby page + a new 'WS adoption' protocol so the WASM client picks up mid-stream after the lobby already consumed Authenticate/GameCreated/AuthResult/GameStarted on its socket. Significant new code surface; risk of subtle desync.
- Option B uses the existing tui_game.html network-mode launcher. The lobby's WS for browsing closes BEFORE redirect. The game page opens its OWN fresh WS and uses NEW WASM exports (network_create_game / network_join_game) to attach to the right named slot. No second-pairing race because the lobby never CreateGame'd / JoinGame'd in the first place.

IMPLEMENTATION:
1. web/index.html: createGame() / joinGame() close the lobby WS, then redirect to tui_game.html?lobby_create=NAME&lobby_pass=PASS&name=USER&ws=URL (or lobby_join). Lobby UI no longer needs the waiting/paired pane.
2. mtg-engine/src/wasm/network/{client,exports}.rs: add create_game() / join_game() methods on WasmNetworkClient mirroring the existing authenticate() method, plus network_create_game / network_join_game wasm-bindgen exports.
3. web/network.js: connect() learns a 'mode' (auth | create | join) plus optional game_name / game_password, dispatched on WS open.
4. web/tui_game.html: on load, if URL has lobby_create / lobby_join params, skip the launcher UI and auto-launch network mode targeting the named slot.
5. web/test_landing_page_ux.js: extend with two-browser inline-pair test and refresh-doesn't-error test.

VALIDATION: full make validate run with validate_logs/<sha>.log captured.

Tracks: mtg-465 (extends, in-page play follow-up)
