=== SHARPENED ROOT (refines the 2026-06-05 diagnosis) ===
The unifying root is now pinned more precisely than "process_card_reveal mutates
outside the undo log." The real divergence is an ORDER mismatch in WHEN the shadow
applies a state-sync reveal relative to the engine action that consumes it:

  • FIRST forward pass: the WASM/native shadow applies reveals from the state-sync
    reveal_log EAGERLY / in websocket-ARRIVAL order, bounded by a completeness
    watermark (apply_state_sync_at: reveals bounded by max_received_choice_ac,
    ahead of the shadow's own action_count — client.rs ~942-992 native;
    wasm/network/client.rs apply path). An async reveal can therefore land AFTER
    the engine action whose log text / state depends on it.
  • REWIND + REPLAY pass: unwind_state_sync_to(R) clears future-only instances and
    re-materialises retained reveals (ac<=R) BEFORE replay, then reset_state_sync_cursor
    re-consumes the whole reveal_log in strict ac-order as the shadow re-advances
    (wasm/network/client.rs ~1740-1845). So replay materialises each reveal at/just
    before its ac, every time.

=> Forward pass and replay disagree on reveal-application order, so any action whose
   OUTPUT depends on a reveal being materialised at that moment diverges:
   - seed 5: discard_card (actions/mod.rs:6712-6740) does move_card(Hand->Graveyard)
     then reads card.name with an `unwrap_or_else(card#{id})` fallback. On P2's
     (WASM) shadow, opponent card 52 has NO instance at move time, so
     maybe_reveal_to_all takes the else-branch (state.rs:1193) — logs
     RevealCard{name:None}, does NOT materialise — and the name reads "card#52".
     On replay the reveal_log entry for card 52 (= the SERVER's discard-ac
     RevealCard{name:Some("Disenchant"),All}) has already been applied, instance
     exists -> "Disenchant". (Two formats appear because a present instance + its
     mask also flips which discard log path/branch prints; the divergence is the
     reveal-timing dependence, byte-confirmed.)
   - seed 2: a reserved P1 library card is materialised by process_card_reveal at a

=== THE PRINCIPLED INVARIANT TO ENFORCE ===
"Reveal before first use, identically on forward and replay." (docs/NETWORK_ARCHITECTURE.md
§RevealCard Actions, lines 110-141.) Concretely: on the shadow, a state-sync reveal
keyed at ac=K must be APPLIED (instance materialised + mask set) at the point the
shadow's action_count reaches K — BEFORE the engine executes the action(s) at K —
and this must hold on the FIRST forward pass exactly as it holds on rewind+replay.
The eager-ahead reveal application (added for the B2 apply-frontier stall: opponent
plays need their reveals materialised ahead so the shadow can replay forward to the
ChoiceRequest ac) must NOT change the action_count at which a reveal becomes visible
to an engine action that runs at a LOWER ac than the reveal-watermark.
