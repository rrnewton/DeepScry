# Description

Card Compatibility: Chaos Orb (LEA / mtg-383) — WORKING (owner's single-target digital interpretation).

CARD STATUS: WORKING.

== Owner interpretation (2026-06-01_#2639(f4402e9b), compat-oldschool-finish) ==
Chaos Orb is a paper DEXTERITY card. Per the project owner we approximate it
digitally as a single targeted destroy:
  "{1}, {T}: Destroy target nontoken permanent, then destroy Chaos Orb."
This INTENTIONALLY DIVERGES from Forge-Java (physical flip + DestroyAll over the
"touched"/Remembered set). The divergence is expressed in the Rust ENGINE
(effect_converter), NOT by editing the card script.

Card script (cardsfolder/c/chaos_orb.txt) is UNEDITED upstream Forge-Java:
  A:AB$ FlipOntoBattlefield | Cost$ 1 T | SubAbility$ DBDestroyTouched | ActivationZone$ Battlefield | ...
  SVar:DBDestroyTouched:DB$ DestroyAll | ValidCards$ Card.IsRemembered+!token | SubAbility$ DBDestroyChaosOrb | ...
  SVar:DBDestroyChaosOrb:DB$ Destroy | Defined$ Self | SubAbility$ DBCleanup | ...
  SVar:DBCleanup:DB$ Cleanup | ClearRemembered$ True

== Engine divergence (where the work lives) ==
effect_converter.rs ApiType::Unknown("FlipOntoBattlefield") arm is RETAINED and
branches on the tokenized SubAbility$ reference:
  - Chaos Orb (SubAbility$ DBDestroyTouched, the default branch) -> a single
    targeted DestroyPermanent restricted to nontoken permanents
    (requires_nontoken=true, controller=Any: owner may aim at their own board).
  - The chained DBDestroyChaosOrb (Destroy | Defined$ Self) still self-destroys
    the Orb; the intervening DestroyAll over the always-empty Remembered set is a
    harmless no-op (we never populate Remembered for the dexterity flip).

== Relationship to Java Forge ==
Java models the physical flip (FlipOntoBattlefield + DestroyAll over the
remembered "touched" permanents). We DIVERGE in the converter to a single
nontoken target because the dexterity flip has no faithful digital analog; per
owner, a skilled flipper destroys exactly the permanent aimed at. The upstream
card script in forge-java is left untouched; the divergence is expressed in the
Rust engine like every other Rust-vs-Java card divergence.

Supersedes the old mtg-392 OppCtrl workaround: the converter no longer forces an
opponent-controller restriction (owner may target their own permanents).
