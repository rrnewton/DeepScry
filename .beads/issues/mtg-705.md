# Description

User 2026-06-02 (BACKLOG / future, AFTER mtg-704). Two related HTML-DAG refactors:

1. PURE-FORWARDER index.html. Make index.html a pure forwarder — NO login UI, no lobby, ZERO release-divergent logic. Its entire job: (1) determine release (release= param else baked latest-token), (2) resolve the entry child via the manifest, (3) location.replace() to it. Move BOTH login (enter name) AND lobby (list/create games) into HASHED child pages: login.<hash>.html -> lobby.<hash>.html -> launcher.<hash> -> game pages (all immutable, release-versioned).
   WHY: login form/validation/lobby-protocol/styling are release-divergent; keeping them in the mutable no-cache index.html means changing login = changing the mutable root. Hashed children make every release's index.html structurally IDENTICAL (same tiny forwarder, differing only in baked latest-token + manifest hash) -> deferred multi-release dispatch (mtg-704) becomes trivial; minimal cache-attack surface; login becomes a release=-pinnable immutable artifact (provenance).
   TRADEOFF (accepted): one redirect hop on fresh visit deepscry.net/ -> login.<hash>.html. Mitigate with location.replace() (clean back-button) + minimal inline loading state (logo, never changes) so there is NO jarring flash. Sub-100ms same-origin -> imperceptible.

2. PROPER SIGN IN / SIGN OUT (user follow-up). Replace the ad-hoc 'change name' button with a first-class identity model. login.<hash>.html IS the Sign In page. Provide explicit Sign Out (clear stored identity -> return to login) and a clear logged-in/out state. Identity (chosen name + any server-assigned token) becomes first-class; lobby/launcher read it; Sign Out drops it.

3. REAFFIRMED + REFINED 2026-06-03 (team-lead, from user rrnewton; surfaced while fixing mtg-727). The user's design restated: index.html should contain NOTHING but the CAS dispatcher (the release=hash logic at index.html lines ~388-462). Today it ALSO inlines the splash/lobby/login PROSE (.hero section etc., lines ~220+). Finish the split: move the splash/lobby/login page into a CONTENT-HASHED asset the dispatcher loads by default (e.g. index.html?goto=lobby), leaving index.html minimal. This makes index.html the GENUINE sole-mutable entry.

4. DESIGN QUESTION to settle as part of this work (captured from mtg-727): should data/sets/index.json (the card-SET data resolver, set->hashed .bin) be FOLDED INTO the release-hash graph (content-addressed + immutable, fetched by hashed name like the .bin/.wasm/.js) rather than remaining a SECOND mutable/no-cache file? Today asset_hash.rs claims index.html is the 'SOLE mutable/no-cache file' but that is FALSE in practice: (a) the full-graph hasher DOES rename index.json -> index.<hash>.json (immutable) and rewrite the game-page fetch, yet (b) the lobby still fetches the FIXED ./data/sets/index.json (web/index.html:572, cache:'no-store') and source/partial-deploy trees serve a fixed index.json. mtg-727 NARROWLY hardened the server to serve any fixed-name /data/sets/index.json as no-cache (defense-in-depth) + asserts it in the probe + a hermetic test. The DURABLE fix (this issue): decide one model — either (i) index.json is fully hashed/immutable everywhere (drop all fixed-name fetches incl. the lobby prefetch; index.html truly sole-mutable), or (ii) index.json stays a deliberate fixed no-cache resolver and asset_hash.rs is corrected to stop renaming it + stop claiming sole-mutable. Pick (i) or (ii) and make the code + asset_hash.rs docstring + deploy-release SKILL consistent.
