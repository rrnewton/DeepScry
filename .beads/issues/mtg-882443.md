# Description

## Problem

The benchmark infrastructure has conflicting global allocator declarations that prevent proper DHAT heap profiling of gameplay workloads. Currently:

1. **benches/allocator.rs** sets global allocator via features (bench-stats-alloc, bench-mimalloc, bench-jemalloc)
2. **bin/rewind_bench.rs** tries to add DHAT support but conflicts with benchlib's allocator
3. **benches/dhat_profile.rs** works by being standalone (doesn't include benchlib)

**Result**: Cannot run DHAT profiling on rewind_bench's full 1000-game workload. Only captures initialization allocations, not gameplay.

**Critical Gap**: We need to profile the actual gameplay allocations to understand:
- Source of 78-92 GB allocations in parallel benchmarks (vs 3.2 GB sequential)
- GameState::clone() allocation patterns for MCTS
- Per-turn allocation hotspots during gameplay

## Current State (2025-11-07_#817)

## Files Involved

1. **mtg-benchmarks/benches/allocator.rs** (212 lines)
   - Sets #[global_allocator] via features
   - Provides AllocTracker for stats_alloc
   - Feature flags: bench-stats-alloc, bench-mimalloc, bench-jemalloc
   - Used by: game_benchmark.rs, all criterion benchmarks

2. **mtg-benchmarks/bin/rewind_bench.rs** (316 lines)
   - Standalone benchmark binary
   - Added dhat-heap feature (commit ee588682)
   - Tries to set #[global_allocator] when dhat-heap enabled
   - **CONFLICTS** with benchlib/allocator.rs

3. **mtg-benchmarks/benches/dhat_profile.rs** (89 lines)
   - Standalone DHAT benchmark
   - Sets #[global_allocator] = dhat::Alloc
   - **Does NOT include benchlib** to avoid conflict
   - Only profiles initialization, not full workload

4. **mtg-benchmarks/lib/mod.rs**
   - Re-exports allocator module
   - All benchmark infrastructure depends on this

## Conflict Analysis

\`\`\`rust
// benches/allocator.rs (included by lib/mod.rs)
#[cfg(feature = "bench-stats-alloc")]
#[global_allocator]
pub static GLOBAL: &StatsAlloc<System> = &INSTRUMENTED_SYSTEM;

// bin/rewind_bench.rs (commit ee588682)
#[cfg(feature = "dhat-heap")]
#[global_allocator]
static ALLOC: dhat::Alloc = dhat::Alloc;  // CONFLICT!
\`\`\`

**Issue**: Rust allows only ONE #[global_allocator]. When rewind_bench includes benchlib (for BatchBenchmark infrastructure), it gets allocator.rs's global allocator declaration, creating a conflict.

## Requirements

1. **Single Global Allocator**: Only one #[global_allocator] declaration per binary
2. **Feature Orthogonality**: Profiling features (dhat-heap) should not conflict with allocator features (bench-stats-alloc)
3. **Code Reuse**: rewind_bench needs BatchBenchmark infrastructure from benchlib
4. **Zero Overhead**: Profiling features should not impact non-profiling builds
5. **Runtime Selection**: Ideally support "cargo run --bin rewind_bench -- --dhat" without recompilation

## Proposed Solution

## Option A: Conditional Global Allocator (Recommended)

Make allocator.rs's global allocator conditional on NOT having dhat-heap:

\`\`\`rust
// benches/allocator.rs
#[cfg(all(feature = "bench-stats-alloc", not(feature = "dhat-heap")))]
#[global_allocator]
pub static GLOBAL: &StatsAlloc<System> = &INSTRUMENTED_SYSTEM;

#[cfg(all(feature = "bench-mimalloc", not(feature = "dhat-heap")))]
#[global_allocator]
pub static GLOBAL: mimalloc::MiMalloc = mimalloc::MiMalloc;

// etc for other allocators
\`\`\`

**Pros**:
- Minimal code changes
- Clear precedence: dhat-heap overrides bench-* features
- Binaries control their allocator

**Cons**:
- Feature interaction complexity
- Need compile-time checks for mutual exclusion

## Option B: Extract Allocator Module (Alternative)

Move allocator module OUT of benchlib. Each binary sets its own global allocator.

**Pros**:
- Complete separation of concerns
- Each binary has full control
- No feature conflicts

**Cons**:
- More code duplication
- Each binary needs allocator setup
- Harder to share AllocTracker utilities

## Option C: Runtime Allocator Selection (Ideal, Complex)

Use dlopen/dlsym to swap allocators at runtime. Not recommended due to complexity.

## Recommended Approach

**Phase 1: Quick Fix (Option A)**
1. Add compile-time checks for feature conflicts
2. Make bench-* allocators conditional on \`not(feature = "dhat-heap")\`
3. Update rewind_bench to properly set DHAT allocator
4. Test: \`cargo run --release --features dhat-heap --bin rewind_bench -- --dhat --batch-size 1000\`

**Phase 2: Verification**
1. Verify DHAT captures full 1000-game workload allocations (not just init)
2. Analyze gameplay allocation patterns
3. Profile GameState::clone() for MCTS
4. Document findings in experiment_results/

**Phase 3: Long-term (Option B, if needed)**
1. Extract allocator module if conflicts persist
2. Create allocator prelude for common utilities
3. Document allocator setup patterns
